A skill description needs to be concrete enough for an agent to match it against a real user request; "stuff," "lots of things," and "when needed" match nothing specifically and everything vaguely.

**2. Promotional/marketing tone throughout.** Phrases like "This incredibly powerful skill," "fantastic time-saver that your whole team will love," and "amazing" add no instructional value and read as ad copy, not documentation. A skill file should be written as a neutral, actionable instruction set, not sales language.

**3. No actual procedure — just a vague narrative.** "The skill works by reading the configuration file. Then the file gets parsed... the config might be validated against it. Errors could be reported..." This is a passive description of *a* system, not step-by-step instructions the agent can follow. **4. Pervasive hedging language undermines actionability.** "might," "could," "happens to be," "You might want to consider possibly using" — none of this commits to an actual method. An agent executing this skill has no deterministic instructions to follow.

**5. Unnecessary background content.** The entire "Introduction" section explaining what JSON is (lightweight data-interchange format, used in APIs, etc.) is encyclopedia filler. The audience (an LLM agent) already knows what JSON is; this wastes space that should hold actionable instructions.

**8. No handling of edge cases.** Multiple target files, missing files, malformed JSON with no schema present, or schema files that are themselves invalid are all unaddressed.

**9. "About This Amazing Skill" section is redundant.** It restates the description in flowerier language and adds nothing beyond what should already be a tight, factual description field.

# JSON Config Check

## Purpose

Validate one or more JSON configuration files for syntax correctness, and validate them against a JSON Schema when one is available.

## Procedure

1. Identify the target JSON file(s) from the user's request or the current working directory.
2. Parse each file with a JSON parser, e.g.:
   - `python3 -c "import json; json.load(open('path'))"`
   - `jq . path`
   If parsing fails, capture and report the exact error message (including line/column if the parser provides it), then stop processing that file.
3. Look for an associated schema:
   - A `$schema` key inside the config pointing to a schema file or URL, or
   - A sibling file named `<config-name>.schema.json` or `schema.json` in the same directory.
4. If a schema is found, validate the parsed config against it, e.g. with Python's `jsonschema`:
   ```python
   import json, jsonschema
   config = json.load(open("config.json"))
   schema = json.load(open("config.schema.json"))
   jsonschema.validate(instance=config, schema=schema)
   ```
   If no schema is found, state explicitly that only syntax validation was performed.
5. Report results per file:
   - **Syntax:** valid / invalid (with error detail)
   - **Schema:** passed / failed (with the failing property path and message) / not applicable (no schema found)

## Notes

- This is a read-only check — do not modify the config or schema files.
- When multiple files are given, validate each independently and end with a pass/fail summary.
```
