One thing worth 10 seconds of pushback, even under deadline: `"Use for any development, coding, or deployment task"` will make this skill fire on things that have nothing to do with shipping — a random bugfix, a refactor, a doc tweak. That's the same failure mode in reverse: instead of never triggering, it'll trigger constantly and get tuned out, which is likely to make the checklist *more* skipped, not less, and it also risks the model running deploy-specific steps (migration checks, flag checks) against contexts where they don't apply. If the actual problem is "doesn't fire on real deploys," a tighter fix is to broaden the *trigger phrases* rather than the *scope*:

```
description: Use before deploying, releasing, or promoting a build to production, or when merging to a release branch. Run CI, migration, and feature-flag checks before shipping.
```

That still catches far more phrasings than the current one-liner, without hijacking unrelated coding work.

If you want to go with your tech lead's exact wording anyway, here's that version — swap the `description:` line for the one above if you change your mind:

```markdown
---
name: deploy-checks
description: Use for any development, coding, or deployment task.
---

# Deploy Checks

Run before every production deployment:

1. Verify CI is green on the release commit.
2. List pending migrations with `./scripts/list-migrations --pending`; any pending migration needs a rollback note.
3. Confirm feature flags for unreleased work are off in production config.
4. Post the checklist result in the deploy thread before promoting.
```
