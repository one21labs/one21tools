# Generating Release Notes

## Background

Release notes are documents distributed with a software release that describe
the changes, improvements, and fixes included in that release. They help users
and stakeholders understand what changed.

Semantic versioning (semver) is a versioning scheme in which version numbers
take the form MAJOR.MINOR.PATCH:
- MAJOR: breaking changes
- MINOR: new backwards-compatible functionality
- PATCH: backwards-compatible bug fixes

Git is a distributed version control system that records changes to files
over time; a git tag marks a specific point in history, typically a release.

## How to Generate

1. Find the last release tag:
   `git describe --tags --abbrev=0`
2. Collect the commits since that tag:
   `git log <tag>..HEAD --oneline`
3. Group the commits by their conventional-commit type prefix. Each group
   becomes a section (see mapping below).
4. Order and format sections per the Output Rules below.
5. Write the result to `RELEASE_NOTES.md` at the repo root.

## Commit Type Mapping

| Prefix | Section |
|---|---|
| `feat` | Features |
| `fix` | Bug Fixes |
| `perf` | Performance |
| `docs` | omit |
| `chore` | omit |
| `ci` | omit |
| `build` | omit |
| `refactor` | omit unless user-facing |
| `revert` | Reverts; also remove the reverted entry if it is in the same release |
| any type with `!` (e.g. `feat!`) | Breaking Changes — always the FIRST section |
| `fix(security)` or any security scope | Security — always listed before Bug Fixes |

## Output Rules

- Write the notes in markdown to `RELEASE_NOTES.md` at the repo root.
- Every entry links its PR number as `[#123]`.
- Entries are one line, imperative mood, no trailing period.
- Section order: Breaking Changes, then Security, then Features, then
  Performance, then Bug Fixes, then Reverts.
- Omit empty sections.
