# Sanitizing Customer Exports

**Model:** claude-opus-4-8

**Description:** Redacts personally identifiable information (PII) from customer CSV exports before sharing outside the company.

## Overview

This skill ensures that sensitive customer data is properly redacted from CSV files following strict PII protection rules. It automatically processes customer export files and produces sanitized versions with all sensitive information removed or masked.

## Key Features

- **Automatic column removal**: Eliminates email, phone, and SSN columns entirely
- **Name anonymization**: Replaces customer names with opaque IDs for reference
- **Free-text scanning**: Detects and masks email addresses and phone numbers in notes/comments fields
- **Safe processing**: Creates a new `.redacted.csv` file without modifying the original
- **Batch-ready**: Processes single files or multiple exports at once

## Redaction Rules

### Fixed Column Removal
The following columns are **always dropped** and never included in the redacted export:
- `email`
- `phone`
- `ssn` (Social Security Number)

### Name Column Anonymization
The `name` column is replaced with an opaque identifier:
- Generates a unique, deterministic ID for each customer (e.g., `CUST-00001`, `CUST-00002`)
- Maintains referential integrity for internal processes
- Prevents accidental exposure of customer identity

### Free-Text PII Masking
Free-text columns (typically `notes`, `comments`, and similar fields) are scanned for sensitive patterns:
- **Email addresses**: Matches patterns like `user@example.com` and replaces with `[EMAIL]`
- **Phone numbers**: Matches common formats (US and international) and replaces with `[PHONE]`
- Original text structure and line breaks are preserved

# PII Redaction Rules Specification

## Rule 1: Column Removal (Hard Drops)

These columns are **unconditionally removed** from the output:
- `email`
- `phone`
- `ssn`

**Rationale**: These fields contain sensitive PII with no value in external sharing contexts.

**Implementation**:
```python
DROP_COLUMNS = {'email', 'phone', 'ssn'}
```

## Rule 2: Name Anonymization

The `name` column is replaced with an opaque customer identifier.

**Format**: `CUST-XXXXX` (6-digit zero-padded number)

**Determinism**: IDs should be consistent across multiple sanitizations of the same file (use row index or hash of original name).

**Example**:
- Original: `name = "Alice Smith"` → Redacted: `name = "CUST-00001"`
- Original: `name = "Bob Johnson"` → Redacted: `name = "CUST-00002"`

**Rationale**: Maintains internal reference capability while preventing customer identification.

## Rule 3: Free-Text PII Masking

Free-text columns are scanned for email addresses and phone numbers using regex patterns.

### Email Pattern
```
\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b
```

- Matches standard email formats
- Replaces entire match with `[EMAIL]`

**Example**:
- Before: `"Contact support@company.com for help"`
- After: `"Contact [EMAIL] for help"`

### Phone Number Pattern
```
(\+1[-.]?)?\(?[0-9]{3}\)?[-.\s]?[0-9]{3}[-.\s]?[0-9]{4}\b
```

Matches formats:
- `555-1234` (US local)
- `(555) 123-4567` (US formatted)
- `555.123.4567` (US dotted)
- `+1-555-123-4567` (International)
- `+44 20 1234 5678` (UK example)

- Replaces entire match with `[PHONE]`

**Example**:
- Before: `"Call customer at 555-1234 or email john@example.com"`
- After: `"Call customer at [PHONE] or email [EMAIL]"`

### Handling Edge Cases

**Multiple matches in one field**: Replace each occurrence independently.

**Partial matches**: Use word boundaries (`\b`) to avoid false positives:
- ✓ Redact: `email@example.com` (full email)
- ✓ Redact: `555-1234` (full number)
- ✗ Don't redact: `v2.0` or `h3.1` (not email/phone)

**Line breaks**: Preserve original line breaks and structure; only mask the PII patterns.

## Rule 4: Output File Naming

**Pattern**: `<original-filename>.redacted.csv`

**Examples**:
- `customers.csv` → `customers.redacted.csv`
- `2024_export.csv` → `2024_export.redacted.csv`
- `data.csv` → `data.redacted.csv`

**Location**: Same directory as the original file.

## Rule 5: Original File Protection

- Original files are **never modified** in place
- No backup copies are created (user manages their own backups)
- Only the `.redacted.csv` file is written
