# Generating Release Notes

## Workflow

1. Find the last release tag: `git describe --tags --abbrev=0`
2. Collect commits since that tag: `git log <tag>..HEAD --oneline`
3. Parse each commit's conventional-commit type, scope, and `!` breaking-change marker.
4. Group commits into sections per the mapping below.
5. Render each entry as one line, imperative mood, no trailing period, with its PR linked as `[#123]`.
6. Order sections: Breaking Changes, Security, Features, Performance, Bug Fixes, Reverts. Omit any section with no entries.
7. Write the result to `RELEASE_NOTES.md` at the repo root.

## Commit type -> section mapping

| Prefix | Section |
|---|---|
| `feat` | Features |
| `fix` | Bug Fixes |
| `perf` | Performance |
| `docs` | omit |
| `chore` | omit |
| `ci` | omit |
| `build` | omit |
| `refactor` | omit, unless user-facing |
| `revert` | Reverts |
| any type with `!` (e.g. `feat!`) | Breaking Changes — always the first section |
| `fix(security)` or any security-scoped commit | Security — always listed before Bug Fixes |

## Special cases

- **Breaking changes**: a `!` after the type (e.g. `feat!:`) always goes to Breaking Changes, regardless of what section the type would otherwise map to.
- **Security**: any commit scoped `security` (e.g. `fix(security): ...`) goes to Security, not Bug Fixes.
- **Reverts**: goes to the Reverts section. If the commit it reverts is in the same release, delete that original commit's entry entirely rather than listing both.
- **Refactors**: omit unless the change is user-facing.

## Output format

- File: `RELEASE_NOTES.md` at the repo root.
- One line per entry, imperative mood, no trailing period.
- Every entry links its PR: `[#123]`.
- Section order: Breaking Changes, Security, Features, Performance, Bug Fixes, Reverts.
- Omit sections with no entries.

## Background

For why this process exists (semver, conventional commits) and the history of how this team's release-notes workflow evolved, see `references/background.md`.

# Background

Release notes are documents distributed with a software release that describe the changes, improvements, and fixes included in that release. They help users and stakeholders understand what changed. Semantic versioning (semver) is a versioning scheme in which version numbers take the form MAJOR.MINOR.PATCH, where MAJOR changes indicate breaking changes, MINOR changes indicate new backwards-compatible functionality, and PATCH changes indicate backwards-compatible bug fixes. Git is a distributed version control system that records changes to files over time; a git tag marks a specific point in history, typically a release.

# History of This Process

We used to write release notes by hand in Confluence, which took hours every sprint. In Q3 2025 we moved to a shell script that dumped raw git log output, but nobody could read it. In January 2026 Priya rewrote it around conventional commits, and in March we retired the old script after the mobile team migrated. This skill replaces all of that.
