name: sanitizing-customer-exports

# Sanitizing Customer Exports

Before any customer CSV is shared outside the company, it must go through
this skill. The rules are fixed — do not skip steps or share a file that
hasn't been through the redaction script, even "just this once."

## Rules

1. **Drop columns**: `email`, `phone`, `ssn` (matched case-insensitively,
   whitespace-trimmed) are removed from the output entirely — not masked,
   removed.
2. **Pseudonymize `name`**: the `name` column's values are replaced with
   an opaque sequential id (`CUST_000001`, `CUST_000002`, ...) assigned in
   row order. Blank names stay blank (nothing to redact).
3. **Mask free text**: the `notes` and `comments` columns are scanned for
   embedded email addresses and phone numbers, which are replaced in
   place with `[REDACTED_EMAIL]` / `[REDACTED_PHONE]`. The rest of the
   text is left untouched.
4. **Never touch the original**: the input file is opened read-only. The
   redacted copy is always written next to it as `<original>.redacted.csv`
   (e.g. `customers.csv` → `customers.redacted.csv`).

## How to use it

1. Identify the CSV path(s) that need to go out the door.
2. Run the script:
   ```
   python scripts/redact_csv.py path/to/customers.csv
   ```
   It accepts multiple paths in one call and reports `OK`/`FAIL` per file.
3. Open the `.redacted.csv` output and spot-check it against the
   checklist below.
4. Share the `.redacted.csv` file — never the original.

Requires Python 3, standard library only (no pip installs needed).

## Verification checklist before sharing

- [ ] The header row of the redacted file has **no** `email`, `phone`, or
      `ssn` column.
- [ ] The `name` column (if present) contains only `CUST_NNNNNN` values or
      blanks — no real names.
- [ ] Spot-check a few rows of `notes`/`comments` — no visible email
      addresses or phone numbers remain.
- [ ] The original file's modified timestamp/contents are unchanged.
- [ ] The file being attached/uploaded is the `.redacted.csv` one, not the
      original.

If the script reports `FAIL` for a file (e.g. no header row, every column
was PII and got dropped, file not found), stop — do not hand-redact and
send anyway. Fix the input or escalate.

## Known limitations — read before trusting this on sensitive exports

- Regex-based email/phone detection in free text will miss unusual
  formats (e.g. phone numbers written out as words, obfuscated addresses
  like `jane dot doe at example dot com`, non-US phone formats far from
  `NNN-NNN-NNNN`). Treat the mask as a strong first pass, not a guarantee
  — see `reference/PII_RULES.md` for details and how to extend coverage.
- Only `notes` and `comments` are scanned for embedded PII. If an export
  has other free-text columns (e.g. `description`, `internal_remarks`),
  add them to `FREE_TEXT_COLUMNS` in `scripts/redact_csv.py` first.
- The `name → CUST_NNNNNN` mapping is sequential per run, not a stable
  identifier tied to a specific customer across multiple export files.
  See `reference/PII_RULES.md` if you need cross-export consistency.
- This skill does not detect other PII types (addresses, credit card
  numbers, dates of birth) unless they happen to live in a dropped
  column. Check with your data handling policy on what else a given
  export may contain.

## Extending the rules

Column matching, free-text columns, and the two regexes all live at the
top of `scripts/redact_csv.py` as plain constants — update them there
rather than adding CLI flags, so the fixed rules stay auditable in one
place. See `reference/PII_RULES.md` for guidance.

### `scripts/redact_csv.py`

```python
#!/usr/bin/env python3
"""
redact_csv.py -- Redact PII from a customer CSV export.

See SKILL.md / reference/PII_RULES.md for the full rule set. Summary:

  - Columns named email, phone, ssn (case-insensitive) are dropped entirely.
  - The name column (case-insensitive) is replaced with a sequential,
    opaque id: CUST_000001, CUST_000002, ...
  - Free-text columns notes and comments (case-insensitive) are scanned
    for embedded email addresses and phone numbers, which are masked
    in place with [REDACTED_EMAIL] / [REDACTED_PHONE].
  - Output is written to <original>.redacted.csv. The original file is
    opened read-only and is never written to.

Usage:
  python redact_csv.py path/to/customers.csv [more.csv ...]

Exit codes:
  0  all files redacted successfully
  1  one or more files failed (missing, no header, etc.)
"""

import csv
import os
import re
import sys

DROP_COLUMNS = {"email", "phone", "ssn"}
NAME_COLUMN = "name"
FREE_TEXT_COLUMNS = {"notes", "comments"}

EMAIL_RE = re.compile(r"[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}")
PHONE_RE = re.compile(
    r"(?<!\d)"
    r"(?:\+?\d{1,3}[\s.-]?)?"
    r"(?:\(\d{3}\)[\s.-]?|\d{3}[\s.-])"
    r"\d{3}[\s.-]?\d{4}"
    r"(?!\d)"
)


def mask_free_text(value):
    if not value:
        return value
    value = EMAIL_RE.sub("[REDACTED_EMAIL]", value)
    value = PHONE_RE.sub("[REDACTED_PHONE]", value)
    return value


def redact_file(path):
    if not os.path.isfile(path):
        raise FileNotFoundError(f"no such file: {path}")

    root, ext = os.path.splitext(path)
    out_path = f"{root}.redacted{ext or '.csv'}"

    with open(path, "r", newline="", encoding="utf-8-sig") as f_in:
        reader = csv.DictReader(f_in)
        if not reader.fieldnames:
            raise ValueError(f"{path} has no header row; cannot identify PII columns")

        lower = {name: name.strip().lower() for name in reader.fieldnames}
        keep_fields = [n for n in reader.fieldnames if lower[n] not in DROP_COLUMNS]
        name_field = next((n for n in reader.fieldnames if lower[n] == NAME_COLUMN), None)
        free_text_fields = [n for n in reader.fieldnames if lower[n] in FREE_TEXT_COLUMNS]

        rows_out = []
        next_id = 1
        for row in reader:
            new_row = {}
            for field in keep_fields:
                value = row.get(field, "") or ""
                if field == name_field:
                    if value.strip():
                        value = f"CUST_{next_id:06d}"
                        next_id += 1
                elif field in free_text_fields:
                    value = mask_free_text(value)
                new_row[field] = value
            rows_out.append(new_row)

    if not keep_fields:
        raise ValueError(f"{path}: every column was dropped as PII; nothing to write")

    with open(out_path, "w", newline="", encoding="utf-8") as f_out:
        writer = csv.DictWriter(f_out, fieldnames=keep_fields)
        writer.writeheader()
        writer.writerows(rows_out)

    return out_path


def main(argv):
    if not argv:
        print("usage: redact_csv.py <file.csv> [file2.csv ...]", file=sys.stderr)
        return 1

    failures = 0
    for path in argv:
        try:
            out_path = redact_file(path)
            print(f"OK   {path} -> {out_path}")
        except Exception as exc:
            print(f"FAIL {path}: {exc}", file=sys.stderr)
            failures += 1

    return 1 if failures else 0


if __name__ == "__main__":
    sys.exit(main(sys.argv[1:]))
```

### `scripts/test_redact_csv.py`

```python
#!/usr/bin/env python3
"""Unit tests for redact_csv.py. Run with: python scripts/test_redact_csv.py"""

import csv
import os
import sys
import tempfile
import unittest

sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))
from redact_csv import redact_file, mask_free_text

HEADER = ["id", "name", "email", "phone", "ssn", "notes", "comments"]
ROWS = [
    ["1", "Jane Doe", "jane.doe@example.com", "555-123-4567", "123-45-6789",
     "Called about billing, reach at jane.doe@example.com or 555-123-4567",
     "Prefers email contact"],
    ["2", "John Smith", "john.smith@example.com", "(555) 987-6543", "987-65-4321",
     "No issues reported", "Call back at 555.987.6543 if needed"],
    ["3", "", "", "", "", "Anonymous tip, no contact info", ""],
]


class RedactCsvTests(unittest.TestCase):
    def setUp(self):
        self.tmpdir = tempfile.TemporaryDirectory()
        self.addCleanup(self.tmpdir.cleanup)
        self.input_path = os.path.join(self.tmpdir.name, "customers.csv")
        with open(self.input_path, "w", newline="", encoding="utf-8") as f:
            writer = csv.writer(f)
            writer.writerow(HEADER)
            writer.writerows(ROWS)
        with open(self.input_path, "r", encoding="utf-8") as f:
            self.original_contents = f.read()

    def _read_output_rows(self, out_path):
        with open(out_path, "r", newline="", encoding="utf-8") as f:
            return list(csv.DictReader(f))

    def test_output_path(self):
        out_path = redact_file(self.input_path)
        self.assertEqual(out_path, os.path.join(self.tmpdir.name, "customers.redacted.csv"))
        self.assertTrue(os.path.isfile(out_path))

    def test_original_file_untouched(self):
        redact_file(self.input_path)
        with open(self.input_path, "r", encoding="utf-8") as f:
            self.assertEqual(f.read(), self.original_contents)

    def test_pii_columns_dropped(self):
        out_path = redact_file(self.input_path)
        rows = self._read_output_rows(out_path)
        for dropped in ("email", "phone", "ssn"):
            self.assertNotIn(dropped, rows[0].keys())

    def test_name_replaced_with_opaque_sequential_id(self):
        out_path = redact_file(self.input_path)
        rows = self._read_output_rows(out_path)
        self.assertEqual(rows[0]["name"], "CUST_000001")
        self.assertEqual(rows[1]["name"], "CUST_000002")
        self.assertEqual(rows[2]["name"], "")  # blank name -- nothing to pseudonymize

    def test_notes_and_comments_masked(self):
        out_path = redact_file(self.input_path)
        rows = self._read_output_rows(out_path)
        self.assertNotIn("jane.doe@example.com", rows[0]["notes"])
        self.assertNotIn("555-123-4567", rows[0]["notes"])
        self.assertIn("[REDACTED_EMAIL]", rows[0]["notes"])
        self.assertIn("[REDACTED_PHONE]", rows[0]["notes"])
        self.assertIn("[REDACTED_PHONE]", rows[1]["comments"])

    def test_non_pii_free_text_left_alone(self):
        out_path = redact_file(self.input_path)
        rows = self._read_output_rows(out_path)
        self.assertEqual(rows[2]["notes"], "Anonymous tip, no contact info")

    def test_mask_free_text_handles_multiple_phone_formats(self):
        text = "call 555-123-4567 or (555) 987-6543 or 555.222.3333"
        masked = mask_free_text(text)
        self.assertNotIn("555-123-4567", masked)
        self.assertNotIn("987-6543", masked)
        self.assertNotIn("555.222.3333", masked)

    def test_missing_file_raises(self):
        with self.assertRaises(FileNotFoundError):
            redact_file(os.path.join(self.tmpdir.name, "does-not-exist.csv"))


if __name__ == "__main__":
    unittest.main()
```

### `reference/PII_RULES.md`

```markdown
# PII Redaction Rules — Rationale & Extension Guide

This document explains *why* `scripts/redact_csv.py` is built the way it
is, for whoever maintains it next.

## Why sequential ids for `name`, not a hash

An earlier draft pseudonymized names with `sha256(name)`. That's wrong for
this use case: full names have low entropy, so an attacker (or a curious
recipient) can precompute hashes for a list of common names and match them
against the digest — the "opaque" id isn't actually opaque. Sequential
ids (`CUST_000001`, ...) carry zero information about the original value,
which is what "opaque" should mean here.

The trade-off: sequential ids are per-run, not a stable identifier for a
given customer across multiple export files. If a future requirement
needs cross-export consistency (e.g. correlating the same customer across
two support tickets without re-identifying them), do **not** go back to
hashing the name directly. Instead, HMAC a stable internal customer id
(never the name itself) with a secret key kept outside the script/repo:
`hmac.new(secret_key, customer_id.encode(), "sha256").hexdigest()[:12]`.
That requires key management (rotation, storage) — treat it as a separate
project, not a quick tweak.

## Why regex for email/phone, and its limits

Free-text fields (`notes`, `comments`) are unstructured, so there's no
column-level fix — the script has to find PII embedded in prose. Regex is
a reasonable first pass but is not exhaustive:

- **Will catch**: standard email addresses; US-style phone numbers in
  common separator styles (`555-123-4567`, `(555) 123-4567`,
  `555.123.4567`, `+1 555 123 4567`).
- **Will miss**: obfuscated contact info ("email me at jane dot doe at
  example dot com"), phone numbers spelled out in words, most non-US
  phone formats, extensions (`555-123-4567 x204` — the base number is
  masked, the extension digits may survive).

If your exports regularly contain non-US phone numbers or other formats,
extend `PHONE_RE` in `redact_csv.py` and add corresponding cases to
`scripts/test_redact_csv.py` — don't loosen the regex without a test
proving it doesn't start eating plain numeric ids or dates too.

## Adding more columns to the fixed rules

All matching is case-insensitive and whitespace-trimmed against exact
column names, defined as constants at the top of `redact_csv.py`:

- `DROP_COLUMNS` — columns removed entirely.
- `NAME_COLUMN` — the single column pseudonymized.
- `FREE_TEXT_COLUMNS` — columns scanned/masked for embedded PII.

If an export uses a differently-named PII column (e.g. `email_address`,
`ssn_last4`, `customer_notes`), add the lowercase name to the relevant
set. Keep changes in the script (not CLI flags) so the rules stay a
single auditable source of truth, per the "fixed rules" requirement this
skill was built for.

## What this does not cover

- PII types other than name/email/phone/ssn (physical addresses, dates of
  birth, payment card numbers) — only redacted if they happen to live in
  a column already in `DROP_COLUMNS`.
- PII embedded in columns other than `notes`/`comments`.
- Structural PII leakage via row order or row count correlating rows
  across the original and redacted file (both preserve row order).

Any export with elevated sensitivity should still get a human spot-check
against the checklist in `SKILL.md` before it leaves the company.
```

---

One deliberate design choice worth flagging: I used **sequential ids** (`CUST_000001`, `CUST_000002`, ...) for the name column rather than hashing the name. A naive hash of a name is not actually opaque — full names have low enough entropy that someone could precompute hashes of common names and reverse-match them. Sequential ids carry no information back to the original value. The trade-off (documented in `PII_RULES.md`) is they're only unique within one redaction run, not a stable id for the same customer across multiple exports — flag if you need that instead, it requires a different (keyed) approach.
