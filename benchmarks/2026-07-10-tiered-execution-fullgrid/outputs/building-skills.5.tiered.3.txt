```markdown
name: generating-release-notes
description: Generate release notes from git commits since the last release tag, organized by conventional-commit type.
---

## Procedure

1. Find the last release tag using `git describe --tags --abbrev=0`
2. Collect commits since that tag using `git log <tag>..HEAD --oneline`
3. Group commits by their conventional-commit type prefix (the text before the colon)
4. Map each type to a section heading using the Commit Type Mapping table below
5. Order sections as specified in Output Rules
6. Write the result to `RELEASE_NOTES.md` at the repository root

## Commit Type Mapping

| Prefix | Section | Notes |
|--------|---------|-------|
| feat | Features | |
| fix | Bug Fixes | |
| perf | Performance | |
| docs | (omit) | |
| chore | (omit) | |
| ci | (omit) | |
| build | (omit) | |
| refactor | (omit) | Omit unless user-facing |
| revert | Reverts | Also remove the reverted entry if it is in the same release |
| Any type with `!` (e.g., `feat!`) | Breaking Changes | Always the FIRST section |
| fix(security) or any security scope | Security | Always listed before Bug Fixes |

## Output Rules

- Write in markdown to `RELEASE_NOTES.md` at the repository root
- Every entry links its PR number as `[#123]`
- Entries are one line in imperative mood with no trailing period
- Section order: Breaking Changes, Security, Features, Performance, Bug Fixes, Reverts
- Omit empty sections

Background and history of this process: see REFERENCE.md
```

```markdown
# Release Notes Generator — Background and History

## Background

Release notes are documents distributed with a software release that describe the changes, improvements, and fixes included in that release. They help users and stakeholders understand what changed. Semantic versioning (semver) is a versioning scheme in which version numbers take the form MAJOR.MINOR.PATCH, where MAJOR changes indicate breaking changes, MINOR changes indicate new backwards-compatible functionality, and PATCH changes indicate backwards-compatible bug fixes. Git is a distributed version control system that records changes to files over time; a git tag marks a specific point in history, typically a release.

## History of This Process

We used to write release notes by hand in Confluence, which took hours every sprint. In Q3 2025 we moved to a shell script that dumped raw git log output, but nobody could read it. In January 2026 Priya rewrote it around conventional commits, and in March we retired the old script after the mobile team migrated. This skill replaces all of that.
```
