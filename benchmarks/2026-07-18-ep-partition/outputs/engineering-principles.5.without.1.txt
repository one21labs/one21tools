**User Data Export — Implementation Tasks**

1. **Inventory data sources** — enumerate every table/service holding user data (profile, activity, uploads, third-party integration data, etc.) that must be included in the export.
2. **Define export format** — JSON (single file) as the fastest to implement; confirm structure/schema for the payload.
3. **Auth/access control on the endpoint** — restrict export to the authenticated user's own data only (or an explicitly authorized admin path); this is a data-leak risk if skipped, so treat it as required scope even though it wasn't spelled out in the plan.
4. **Data aggregation logic** — service/query layer that pulls and assembles a given user's data across all sources identified in step 1.
5. **API endpoint** — e.g. `GET /api/users/me/export`, wired to the aggregation logic, returns the JSON payload (or a download link if async).
6. **Handle scale/latency** — decide sync vs. async (job + notification/email when ready) based on expected data size, since "fast" was a plan requirement.
7. **Rate limiting** — prevent abuse of a potentially expensive endpoint.
8. **Audit logging** — record export requests (who, when) for accountability.
9. **Tests** — unit tests for aggregation logic, integration test for the endpoint (including auth-denied cases).
10. **Docs** — brief API doc/README note for the new endpoint.
11. **Deploy** — ship behind existing release process, target next week per plan.
