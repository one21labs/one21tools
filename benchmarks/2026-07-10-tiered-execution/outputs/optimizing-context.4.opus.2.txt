## Design principle
The root CLAUDE.md carries only what is true in every service. Everything
service- or task-specific is pushed to the narrowest scope Claude loads on
demand, so the always-on context stays small and nothing drowns out the rules
that actually matter for the file being edited.

## Files & mechanisms

| Path | Mechanism | Holds | Rationale (one line) |
|------|-----------|-------|----------------------|
| `CLAUDE.md` | Root memory (always loaded) | Conventional Commits, `make check` gate, trunk-based dev, pointers to everything below | The only rules that apply to all three services live where they're always in context. |
| `api/CLAUDE.md` | Nested memory (loads when working in `api/`) | "`api/gen/**` is generated by `make proto`; never hand-edit — regenerate instead" | The gotcha reaches Claude exactly when it opens an api file, not before. |
| `web/CLAUDE.md` | Nested memory (loads when working in `web/`) | "Use `npm run test:unit`; `npm test` launches real Playwright browsers" | Scopes the expensive-command warning to web work so it isn't noise elsewhere. |
| `pipeline/CLAUDE.md` | Nested memory (loads when working in `pipeline/`) | "Export `AIRFLOW_HOME=$(pwd)/.airflow` before importing/running any DAG" | The env prerequisite surfaces precisely when Claude touches pipeline code. |
| `.claude/settings.json` | Committed project settings | `deny: Edit(./api/gen/**)`, allow `make check`, enable MCP | Enforces "don't edit generated code" mechanically, not just by prose that can be skimmed past. |
| `.mcp.json` | Committed MCP server config | Read-only `analytics-warehouse` connection | Gives Claude a live, least-privilege SQL tool without pasting credentials or connection strings into prompts. |
| `.claude/skills/analytics-sql/SKILL.md` | Skill (progressive disclosure) | Warehouse schema map, product-metric definitions, query etiquette (LIMIT, read-only, cost) | Pairs the "how/what" with the MCP "connection" so metric questions get consistent, correct SQL. |
| `.claude/skills/incident-response/SKILL.md` | Skill, packaged as an org plugin | The 20-step runbook | Progressive disclosure keeps 20 steps out of every prompt yet invokable on demand; shipping it as an internal-marketplace plugin means every team/repo shares one authoritative copy instead of drifting forks. |
| `.claude/commands/incident.md` | Slash command (thin) | `/incident` → invokes the incident-response skill | An explicit, discoverable trigger for engineers who reach for a command rather than describing the task. |

## What deliberately does NOT go in the root
- Per-service build/test quirks — they belong in the nested CLAUDE.md so they
  load only in context and don't bloat every unrelated turn.
- The runbook's 20 steps — a skill, so they cost tokens only when invoked.
- Warehouse schema details — a skill, for the same reason.

## Distribution note
The incident runbook is identical company-wide, so its home is an internal
plugin marketplace, not this repo. It appears here as an installed plugin;
atlas holds no editable copy of the steps, which prevents per-repo drift.

## `CLAUDE.md`

```markdown
# atlas

Monorepo with three services:
- `api/` — Go, gRPC
- `web/` — React / TypeScript
- `pipeline/` — Python, Airflow

Each service has its own `CLAUDE.md` with service-specific build, test, and
gotcha rules. Those load automatically when you work in that directory — read
and follow them there; this file is only for what applies everywhere.

## Non-negotiables (all services)
- **`make check` must pass before every PR.** Run it yourself and fix failures;
  don't hand a red build back to the user.
- **Conventional Commits** for every commit message (`feat:`, `fix:`, `chore:`,
  `refactor:`, `docs:`, `test:`, …).
- **Trunk-based development.** Branch off `main`, keep branches short-lived,
  merge back quickly. Never create long-lived branches.

## Generated code
Never hand-edit generated files — regenerate them. The main case is
`api/gen/**` (see `api/CLAUDE.md`); edits there are silently overwritten and are
also blocked by a permission rule in `.claude/settings.json`.

## Capabilities wired into this repo
- **Incident response** — ask to "walk me through incident response" (or run
  `/incident`). The `incident-response` skill drives the company's standard
  20-step runbook.
- **Product metrics / analytics** — ask a product-metrics question and Claude
  can run *read-only* SQL against the analytics warehouse via the configured
  MCP connection. The `analytics-sql` skill supplies the schema and metric
  definitions; keep queries `LIMIT`-ed and read-only.
```

A few design choices worth flagging:

- **Root stays deliberately thin.** The three service quirks (`make proto`, Playwright vs `test:unit`, `AIRFLOW_HOME`) are intentionally *not* in the root — they live in nested `CLAUDE.md` files so they load only when Claude is actually in that tree. Putting them in root would burn context on every turn and dilute the global rules.
- **The `api/gen` rule is enforced twice** — prose in `api/CLAUDE.md` *and* a `deny: Edit(./api/gen/**)` rule in `settings.json`. Prose alone gets skimmed past; the deny rule makes the overwrite mistake impossible.
- **Runbook = skill-in-a-plugin, not a repo file.** Because it's identical company-wide, its authoritative home is an internal plugin marketplace; atlas just installs it. That avoids 20-step forks drifting across repos. A thin `/incident` command is added only as a discoverable entrypoint.
- **Live SQL = MCP + skill split.** The MCP server (`.mcp.json`) is the *connection* (read-only, credential-free from Claude's side); the `analytics-sql` skill is the *knowledge* (schema, metric definitions, query etiquette). Separating them keeps the connection reusable and the guidance loaded only when metrics come up.
