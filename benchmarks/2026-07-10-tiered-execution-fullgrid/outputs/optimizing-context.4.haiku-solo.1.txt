I'll design a two-tier context architecture: root CLAUDE.md for org-wide standards, service-specific CLAUDEs for tool quirks, and shared resources for runbook and analytics.

## **layout.md**

```markdown
# Claude Code Context Layout for 'atlas'

## Design Principle
Split context into **org-wide (root) + service-specific** to avoid duplication. Each service engineer opens their directory and gets everything they need; root conventions apply everywhere.

## Files

### Root: `/CLAUDE.md`
**Contents:** Monorepo structure, Conventional Commits, `make check` requirement, trunk-based dev, api/gen/ read-only constraint, links to service CLAUDEs, pointers to runbook and analytics.  
**Rationale:** Every engineer on every team must know the org conventions and the api/gen/ sharp edge. Centralizing prevents drift.

### Service: `/api/CLAUDE.md`
**Contents:** Go/gRPC idioms, proto generation workflow (edit `.proto`, run `make proto`, api/gen/ is output-only).  
**Rationale:** Proto regeneration is a silent data-loss risk unique to api/. Documenting it here prevents hand-edit mistakes.

### Service: `/web/CLAUDE.md`
**Contents:** React/TypeScript patterns, test command gotcha (`npm test` opens real browsers via Playwright; `npm run test:unit` for isolated tests).  
**Rationale:** The browser behavior surprises people and can block CI; document the distinction to prevent accidental full-suite runs.

### Service: `/pipeline/CLAUDE.md`
**Contents:** Python/Airflow patterns, **`AIRFLOW_HOME=pipeline/.airflow` must be set before running DAGs** (they fail to import silently without it).  
**Rationale:** AIRFLOW_HOME is a hidden dependency with poor error messages; must be front-and-center.

### Shared: `/ops/runbooks/incident-response.md`
**Contents:** 20-step incident response procedure (identical across all company teams).  
**Rationale:** Versioning runbooks with code keeps them aligned; Claude can execute them step-by-step in real time. Avoid docs-repo fragmentation.

### Analytics: `.claude/connectors/analytics-warehouse.md`
**Contents:** MCP connector setup for live SQL queries.  
**Rationale:** Analyst-focused, not code-centric; configured separately but documented for reference.

## How Claude Loads This
- Root CLAUDE.md auto-loads when Claude opens monorepo root
- Service CLAUDE.md auto-loads when Claude opens that service directory or files
- Runbook is referenced in root; Claude reads on first mention
- Analytics connector is manual setup; not part of code context

## Key Constraints
1. **Root ≠ duplication**: Don't repeat org conventions in service CLAUDEs; they inherit from root
2. **api/gen/ is off-limits**: Mention in both root and api/CLAUDE.md so it's never hand-edited
3. **Runbook is versioned**: Not a wiki link; lives in the monorepo
```

## **CLAUDE.md** (root content)

```markdown
# Atlas Monorepo: Claude Code Context

## Structure
- **api/** — Go/gRPC backend  
- **web/** — React/TypeScript frontend  
- **pipeline/** — Python/Airflow ETL  

Each service has its own CLAUDE.md. This file covers org-wide standards.

## Org-Wide Standards

### Conventional Commits
All commits: `<type>(<scope>): <message>`. Scope is the service (api, web, pipeline) or subsystem.

Example: `feat(api): add user profile endpoint`

### `make check` Before PR
Must pass. Runs linting, type-checking, and unit tests. No bypasses (`--no-verify`, etc.). Service-specific test commands are in service CLAUDEs.

### Trunk-Based Development
Commit to `main`. Short-lived feature branches OK but merge back within days. No long-lived or release branches.

## Critical: api/gen/ Is Read-Only

Everything under `api/gen/` is **generated by `make proto` and silently overwritten on regeneration**.

To change generated code:
1. Edit the `.proto` file
2. Run `make proto`
3. Verify the output

Never hand-edit anything in `api/gen/`.

## Service-Specific Context

Tooling quirks and patterns are documented in each service's CLAUDE.md:
- **api/CLAUDE.md** — Proto workflow, Go idioms
- **web/CLAUDE.md** — Playwright test behavior and unit test command
- **pipeline/CLAUDE.md** — AIRFLOW_HOME requirement and Airflow patterns

## Org-Wide Procedures

**Incident Response** (`/ops/runbooks/incident-response.md`):  
A 20-step procedure used identically by every team. Claude can walk you through it step-by-step.

**Analytics** (`.claude/connectors/analytics-warehouse.md`):  
Analysts can ask Claude to run live SQL queries against the analytics warehouse. Configure your connector there.

| Question | Answer |
```
