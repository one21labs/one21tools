# atlas monorepo — Claude Code Configuration

## Org-Wide Conventions

### Commits & Versioning
- **Commit format**: Conventional Commits (`feat:`, `fix:`, `chore:`, etc.)
  - Enforce via: refuse to commit without the pattern `type(scope): message`; suggest the format if omitted.
- **Branch strategy**: Trunk-based development (main only; no long-lived feature branches).
  - Consequence: All PR approvals must be fast; CLI can merge to main after `make check` passes + review.

### Pre-Push Checks
- **Gate**: `make check` must pass before any push.
  - If `make check` fails, report the error and ask the user to fix it. Do not bypass with flags like `--no-verify`.
  - Run it automatically after code changes; don't wait for the user to ask.

---

## Service Contexts

Each service has its own quirks. Load the appropriate CLAUDE.md when working in that directory:

- **api/**: Go, gRPC, protocol buffers. See [api/CLAUDE.md](api/CLAUDE.md).
- **web/**: React, TypeScript, Playwright testing. See [web/CLAUDE.md](web/CLAUDE.md).
- **pipeline/**: Python, Airflow, DAG execution. See [pipeline/CLAUDE.md](pipeline/CLAUDE.md).

## File Protection Rules

| Path | Rule | Reason |
|------|------|--------|
| `api/gen/**` | Read-only; never edit | Generated by `make proto`; hand-edits are overwritten silently. If changes needed, edit `.proto` files and regenerate. |
| `.airflow/**` | Avoid editing metadata | Contains Airflow state; breaking it halts all pipeline DAGs. |

## Incident Response

### Quick Access
See **[docs/INCIDENT_RESPONSE.md](docs/INCIDENT_RESPONSE.md)** for the org-wide 20-step runbook.

When a user mentions "incident" or "page" or asks for "incident response":
1. Load the runbook.
2. Walk through steps 1–20 in order, prompting the user for approvals at critical junctions (e.g., "authorize page," "approve rollback").
3. Use TaskCreate to log each step completion with timestamps.
4. If a step fails, log it and escalate to on-call or suggest manual intervention.

### Approval Thresholds
- **Steps 1–10** (diagnostics, logging): Execute with user confirmation, no manager sign-off needed.
- **Steps 11–15** (communication, customer notification): Require explicit user approval before proceeding.
- **Steps 16–20** (recovery, rollback, post-mortem): Require manager or on-call approval before acting.

## Analytics & SQL Access

### For Analysts & Product Teams

- Pre-built SQL query templates (retention, DAU, conversion funnels, etc.)
- Warehouse connection details (read-only).

When asked about product metrics:

2. If a matching template exists, execute the query (read-only) and present the results.
3. If no template exists, suggest creating one (don't hand-write novel queries without approval).
4. Never write to the warehouse; all access is read-only.

## Permissions & Automation

### What Claude Can Do Without Asking
- Edit code in `api/`, `web/`, `pipeline/` (outside protected paths).
- Run `make check`, `npm test`, `npm run test:unit`, test runners.
- Create commits with conventional format.
- Follow the incident runbook (steps 1–10) with user confirmation per step.
- Execute read-only SQL queries from docs/ANALYTICS.md.
- Create and update tasks via TaskCreate.

### What Requires User Approval
- Merge to main (even after `make check` passes; wait for human review).
- Push to any branch (ask before pushing).
- Steps 11–20 of incident runbook (critical communication/recovery).
- Any query not in docs/ANALYTICS.md (hand-written SQL against the warehouse).
- Modifying CLAUDE.md or service CLAUDE.md files.
- Editing api/gen/ or .airflow/ (refuse; suggest the proper process instead).

### What Claude Must Refuse
- Bypassing `make check` with `--no-verify` or similar.
- Editing or deleting `api/gen/` files (they regenerate; hand-edits are lost).
- Non-conventional commit messages.
- Force-pushing or rewriting published history.
