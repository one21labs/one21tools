## Defects in Draft Skill

- **Frontmatter `description` is unusable for routing.** The description "This skill is really helpful for working with JSON configuration files and stuff. It can do lots of things! Use when needed" provides no concrete trigger or use case. It doesn't specify what file types, what problem it solves, or what conditions should invoke it. A routing description must state the skill's purpose and trigger conditions in concrete, specific terms so an agent can decide whether to invoke it.

- **Marketing and filler language throughout.** Phrases like "incredibly powerful," "fantastic time-saver that your whole team will love," "really helpful," and "Amazing Skill" are ad copy, not instructions. These add no actionable information to the skill and waste context that could be spent on actual procedure. Skills are consumed by agents performing work, not marketing documents.

- **Unnecessary background content wastes tokens.** The "Introduction" section explaining what JSON is, how widely used it is, and general statements about validation are general knowledge not needed by an agent to perform a specific task. This background should be cut entirely.

- **Instructions are non-actionable and hedged throughout.** The "How It Works" section uses speculative language: "might be validated," "could be reported," "You might want to consider possibly using… which can be a good choice." None of these tell an agent what to actually do. Instructions must be concrete and imperative (parse with X, if Y then do Z) rather than suggestions.

- **No committed procedure or design decisions.** The skill is entirely speculative—it gestures vaguely at "Python's json module and jsonschema" without committing to either, provides no ordered steps, and offers no defined output format. A working skill must commit to specific tools and a clear workflow.

- **Missing critical structural elements.** The skill does not specify: how to locate the schema file (only "if a schema file happens to be available in the same directory"), what to do when no schema exists, how parse errors should be reported, how validation errors should be formatted, or what the user should see when the skill completes successfully.

# JSON Config Check

## Procedure

1. **Locate and read the JSON configuration file.** The file path will be provided or inferred from context. Read the file contents.

2. **Parse the JSON.** Attempt to parse the file as valid JSON using Python's `json` module. If parsing fails, report the error to the user with the line number and character offset where the syntax error occurred. Stop here.

3. **Locate the schema file.** Look for a schema file in the same directory as the config file, following the naming convention `<config-name>.schema.json`. For example, if the config is `app.json`, look for `app.schema.json`. If no schema file exists, report to the user that the config file is syntactically valid JSON but was not validated against a schema (because no schema was found), and stop.

4. **Parse the schema.** Read and parse the schema file as JSON. If parsing fails, report the error to the user and stop.

5. **Validate the config against the schema.** Use Python's `jsonschema` package to validate the parsed config against the schema. 

6. **Report results to the user.** If validation succeeds, report that the config file is valid and conforms to the schema. If validation fails, report each validation error with the path to the offending field and a description of what the schema requires.
