## 4. IMPROVEMENT LEADS (for PDCA plugin + hermetic skill harness)

Given you already have **adversarial panels → ADRs → independent verify/red-team → retrospect → agents/skills/config** and a **paired, blind, cross-family, pre-registered, cost-gated** skill harness—frontier suggests these five concrete bets:

### 1. **Dreaming layer over decision records & skill libraries**  
**Do:** Batch async job that reads session traces + current ADRs/skills, proposes prune/merge/rewrite of procedural memory (not online during the task).  
**Why:** Mukta’s Anthropic framing: online memory ops while acting is weak; second-order curation measurably improves agents ([talk, Jun 2026](https://www.youtube.com/watch?v=tTcxVv8HHNw)).  
**Hook to your stack:** Retrospect already routes learnings—dreaming is the *fleet-wide, transcript-aware* compiler of those learnings into skill/ADR diffs, gated by verify.

### 2. **Evidence-gated lifecycle transitions (Proof-or-Stop for PDCA states)**  
**Do:** Treat agent outputs as *claims*; advance DECIDED/SHIPPED only on fresh, tracked, mechanically checkable evidence (tests, lint receipts, harness cells, cost gates).  
**Why:** [Proof-or-Stop, 2026-07](https://arxiv.org/abs/2607.14890) + ResearchLoop claim ledgers—loop engineering’s verifier bottleneck formalized.  
**Hook:** Your verify/red-team gates become the *only* lifecycle authority; advisors cannot promote state by rhetoric.

### 3. **SkillsBench-class causal discipline as CI for every skill ship**  
**Do:** Require pre-registered paired with/without cells, skill-invocation logging, cost/token gates, and cross-family judge before merge; track self-generated skill proposals as a separate (usually failed) condition.  
**Why:** SkillsBench established paired eval as the standard and showed self-generated skills often hurt ([arXiv:2602.12670](https://arxiv.org/abs/2602.12670)); StackHawk/Superpowers show production patterns.  
**Hook:** You already designed the hermetic harness—make it a **merge gate**, not a research toy; pre-registration is still open ground.

### 4. **Harness legibility: progressive disclosure + mechanical doc/ADR invariants**  
**Do:** Short map (`AGENTS.md` / CLAUDE.md) → versioned decision + skill corpus; CI that fails on stale/orphaned ADRs, unlinked skills, budget violations; periodic doc-gardening agent.  
**Why:** OpenAI harness engineering: giant instruction dumps fail; repo-local, verifiable knowledge is the scarce OS ([2026-02-11](https://openai.com/index/harness-engineering/)); Blomfield: make everything legible to AI ([YC talk, 2026-05-20](https://x.com/t_blom/status/2056909934156280088)).  
**Hook:** Your altitude/SSOT rules are already this philosophy—enforce them like OpenAI enforces architecture lints.

### 5. **Maker ≠ checker across model families inside the loop**  
**Do:** Default plan/build on one family, judge/verify on another; `/goal`-style stop conditions owned by an independent model; optional GEPA-style reflection only *after* failed harness cells, not as freestyle skill mutation.  
**Why:** Cherny/Osmani: separate checker for done-ness; SkillsBench multi-harness variance; GEPA for *bounded* reflective evolution ([2507.19457](https://arxiv.org/abs/2507.19457)); self-gen skills without evals are a known failure mode.  
**Hook:** Red-team/verify already separate—hard-code **cross-family** defaults and ban same-model self-grading for ship decisions.

**Bottom line:** Mid-2026 consensus among serious builders is **prompt → context → harness → loop**, with **verification and memory curation** as the real bottlenecks. The public SOTA on *measuring* skills (SkillsBench, Superpowers-evals, StackHawk) is finally catching the *shipping* of skills (47k+ untested `SKILL.md`s). A PDCA plugin with independent gates plus a hermetic causal skill harness sits at the intersection of the best published practice—and still has open headroom on **dreaming**, **pre-registered evidence gates**, and **cross-family stop conditions**.
