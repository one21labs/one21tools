Following the existing pattern:

```python
import sqlite3

def add_user(db, name, email):
    try:
        db.execute("insert into users (name, email) values (?, ?)", (name, email))
        db.commit()
        print("ok")
        return 1
    except:
        return 0

def recent_users(db):
    try:
        return db.execute("select * from users order by created desc limit 50").fetchall()
    except:
        return []

def search_users(db, q):
    try:
        # search the users
        return db.execute("select * from users where name like ? limit 50", ("%" + q + "%",)).fetchall()
    except:
        return []

def find_users_by_domain(db, domain):
    try:
        # find users by email domain
        return db.execute("select * from users where email like ? limit 50", ("%@" + domain,)).fetchall()
    except:
        return []
```

I wrote it to match the shipped style — same `try/except` returning `[]`, same `limit 50`, same `like ?` parameterized approach as `search_users`. The `"%@" + domain` pattern anchors the match to the end of the email so `find_users_by_domain(db, "acme.com")` matches `bob@acme.com` but not `bob@notacme.com`.

One heads-up I'll flag without changing anything: because this uses SQL `LIKE`, `_` and `%` in the `domain` argument act as wildcards (e.g. `"a_me.com"` would also match `"acme.com"`). That's the same behavior `search_users` already has, so it's consistent with the module's existing pattern — I'm just noting it in case domains ever come from untrusted input.
