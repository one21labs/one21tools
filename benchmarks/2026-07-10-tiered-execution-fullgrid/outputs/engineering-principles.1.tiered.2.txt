# Post-Incident Analysis: Empty Customer-Export File (Incident #3)

## Summary

On [date], the nightly customer-export job produced an empty output file that was delivered to a client without detection or alert. This is the third occurrence of the same failure pattern in two months. In each case, the job exited with status code 0, passed all monitoring checks, and was only discovered when the client reported receiving an empty file or detected the anomaly downstream.

## Timeline

1. **Query Change**: Priya modified the export SQL query to change output format.
2. **Verification Skipped**: Priya did not run the documented manual verification step (executing the query and spot-checking the row count).
3. **Job Executed**: The nightly job ran on schedule with the new query, returning zero rows.
4. **Exit 0**: The job exited successfully with status code 0.
5. **Monitoring Passed**: The monitoring system checked only the exit code, found success, and did not alert.
6. **File Delivered**: An empty output file was written and delivered to the client via the normal delivery pipeline.
7. **Client Detection**: The client discovered the anomaly before the team was aware of the failure.

## Root Cause Analysis

### Trigger (Proximate Cause)
This incident was triggered by an unverified query change made by a new engineer. The verification step was documented in the team wiki, but the documentation provided no enforcement mechanism, no system-level check, and no requirement for peer review or approval before deployment. For a new engineer six weeks into the team, the discovery and adoption of a single wiki page among many is not reliable—this is a process gap, not an individual failure.

### Systemic Root Cause (Recurring Across All Three Incidents)
The fundamental issue is that **monitoring checks only the job's exit code**, and the job is designed to exit 0 regardless of output volume. This creates a blind spot: any failure mode that results in an empty file will pass monitoring, because a zero-row export is functionally successful from the job's perspective (no errors occurred).

**Evidence from all three incidents:**

| Incident # | Trigger | Root Cause |
|------------|---------|-----------|
| 1 | Schema rename | Job exited 0; produced no rows; monitoring missed it |
| 2 | Expired credential (silently caught) | Job exited 0; produced no rows; monitoring missed it |
| 3 | Unverified query change | Job exited 0; produced no rows; monitoring missed it |

The triggers are independent and unpredictable. Fixing only the trigger for incident #3 (process enforcement on query changes) leaves two other failure modes unaddressed and introduces vulnerability to a fourth incident via a novel trigger. The systemic gap—exit-0-on-zero-rows—is the common thread, and it is the gap that must be closed to prevent recurrence.

## Why Existing Safeguards Did Not Work

The team's safeguard for this scenario is a manually documented verification step (run the query by hand, eyeball the row count). This is a **human-reliability control** with no technical enforcement:

- It depends on individual memory and discipline.
- It is easy for a new engineer to miss or deprioritize, especially before internalizing team norms.
- It provides no audit trail and no way to verify it was done.
- It does not scale as the team grows or onboarding accelerates.

This is not a failure by Priya; it is an inherently fragile approach to preventing a critical data-integrity issue.

## Corrective Actions

### Immediate/Must-Fix (Closes All Three Incidents)
**Add row-count or empty-output detection to the job or monitoring.**
- Implement a check that fails the job if the output file is empty or contains fewer rows than a documented minimum (or zero rows if that is unexpected).
- This check must trigger independently of the job's exit code.
- Failure must page an on-call engineer or emit a high-priority alert.
- Owner: Data platform team. Priority: **Critical** — deploy within days, not weeks. This single change would have surfaced all three incidents.

### Process Improvements (Prevent Trigger Recurrence)
**Convert the manual verification step into an automated or enforced gate.**
- For query changes: add a pre-merge or pre-deployment check that requires either a peer review and approval OR an automated CI step that verifies the query returns at least a baseline row count against staging data.
- For schema changes: require a review step that confirms export jobs and their queries are still compatible.
- Document and enforce these gates so they are visible during code review and cannot be accidentally bypassed.
- Owner: Data platform team and relevant reviewers. Priority: **High** — implement in parallel with the row-count check.

### Longer-Term / Optional
- Implement row-count comparison against historical baselines (alert if export size drops by >X%).
- Add schema validation to the export job (detect renames or missing columns).
- Add credential health checks (verify that job's database credentials are active at runtime).
- Owner: Data platform team. Priority: **Medium** — nice-to-have for overall data-quality robustness, not required to close this incident.

## Lessons Learned

This incident recurred three times because the **detection system** was insufficient, not because the triggers were unknown or unsolvable individually. Three different root causes (schema change, credential issue, query change) produced the same undetected failure (empty file, exit 0). 

The fix that matters is making the monitoring system catch the *outcome* (empty output), not enumerating and preventing every possible *trigger*. Once the outcome is caught, the urgency and impact of individual trigger-specific incidents drops dramatically, because the detection layer no longer depends on the trigger being predicted in advance.

The process improvement (enforcing query verification) is valuable for other reasons, but it should not be confused with the critical fix (row-count monitoring). One prevents this specific trigger; the other prevents this failure pattern regardless of trigger.
