# Docs Restructuring Plan

## 1. Local setup steps (README vs CONTRIBUTING — drifted)
**Canonical location:** README.md, under its existing setup section.
**Action:**
- Before finalizing, verify against `package.json` scripts / actual working setup which version is correct (README's `npm install` + copy `.env.example` + `npm run dev`, or CONTRIBUTING's `npm ci` + hand-created `.env` + `npm start`) — these aren't just wording differences, they're contradictory instructions and only one can be true right now.
- Keep the verified version as the single copy in README.md.
- Delete the duplicate steps from CONTRIBUTING.md entirely; replace with a one-line pointer: "See README.md → Getting Started for local setup." CONTRIBUTING.md should only hold contribution-specific process (branching, PR checks, review expectations), never a second copy of setup steps.

## 2. API request timeout (30s in README vs 45s in CONTRIBUTING vs actual constant in src/http/client.js)
**Canonical location:** `src/http/client.js` (the code is the source of truth for the value).
**Action:**
- Remove the hardcoded number from CONTRIBUTING.md entirely — contributors don't need this fact there, and it's the stale copy.
- In README.md, keep one mention only if external API consumers need it, and state it in a way that can't silently drift, e.g. "requests time out after N seconds (see `src/http/client.js` for the current value)" rather than hardcoding a number in prose. Confirm the real current value from the code before publishing.

## 3. Support email
**Canonical location:** README.md (unchanged).
No duplication reported for this fact — leave as-is, no action needed.

## 4. `recalcTotals()` rule for src/billing.js
**Canonical location:** an inline comment in `src/billing.js`, placed at the line-item mutation site(s) and/or above `recalcTotals()`'s definition.
**Action:** Remove this rule from CLAUDE.md entirely. It's a single-file invariant, not a cross-cutting repo rule — root-level instruction files should stay lean and only hold guidance that applies broadly. A comment at the exact mutation point is also more reliable: it's impossible to touch that code without seeing it, whereas a rule buried in CLAUDE.md is easy to miss.

## 5. "Lessons learned" narrative in CLAUDE.md
**Canonical location:** none — this belongs in git history, not a living instructions file. Delete the section from the CLAUDE.md draft.
**Action:**
- The npm start → npm run dev lesson: superseded once README's setup section is verified correct (item 1). If there's a genuine reason to avoid `npm start` (e.g., a real port clash risk), fold that as a short parenthetical into the README setup step itself — not as narrated history.
- The "renamed the export module" lesson: this is pure history with no forward-looking instruction attached; git log/commit messages already capture it. Drop it.
- General rule going forward: CLAUDE.md should contain current, actionable rules only — never narrated past mistakes. If a past mistake still implies a live constraint, state the constraint directly (imperative, present tense), not the story behind it.
