# layout.md

| Location | Mechanism | Contents | Rationale |
|----------|-----------|----------|-----------|
| `./CLAUDE.md` (root) | Static markdown | Conventional commits, `make check` gate, trunk-based dev policy, pointers to service-specific guides | Loaded on every session; keeps only cross-service rules to avoid bloat |
| `./api/CLAUDE.md` | Static markdown | `api/gen/` is generated by `make proto`, never hand-edit; Go/gRPC conventions | Loaded only when working in api/; service-specific quirk critical to workflow |
| `./web/CLAUDE.md` | Static markdown | `npm test` runs real browsers (Playwright), use `npm run test:unit` for unit tests; React/TypeScript patterns | Loaded only when working in web/; prevents accidental expensive test runs |
| `./pipeline/CLAUDE.md` | Static markdown | `AIRFLOW_HOME=pipeline/.airflow` must be set before DAG imports; Python/Airflow conventions | Loaded only when working in pipeline/; DAG import will fail silently without this |
| Org-level (outside atlas) | Skill (slash command) | 20-step incident-response runbook with interactive step-by-step guidance | Procedural + stateful (progress tracking), not static reference; identical across company so belongs in shared org context, not per-repo; can execute steps and provide context |
| Org-level (outside atlas) | MCP tool (read-only analytics warehouse access) | SQL query builder against analytics warehouse with pre-validated schemas | Requires tool/credential access, not static docs; read-only enforcement prevents accidents; analyst-specific, not relevant to api/web/pipeline engineers so should not clutter their contexts |

# CLAUDE.md

```markdown
# Atlas Monorepo Context

## Conventions (All Services)

### Conventional Commits
All commits follow the Conventional Commits specification. Format: `type(scope): description`
- **type**: `feat`, `fix`, `docs`, `refactor`, `perf`, `test`, `ci`, `chore`, etc.
- **scope**: service name or component (e.g., `api`, `web`, `pipeline`)
- Examples: `feat(api): add user authentication endpoint`, `fix(web): resolve button alignment issue`

### `make check` Gate
Before opening any PR, run `make check` locally. This must pass — it is a hard requirement.
- Covers linting, type checking, tests, and any service-specific validation.
- No PR will merge if `make check` fails in CI.

### Trunk-Based Development
- Work directly on `main` or short-lived feature branches (< 1 week).
- No long-lived release branches.
- Frequent, small PRs; rebase and merge.
- Enables rapid iteration and reduces integration pain.

## Service-Specific Guidance

When working in a subdirectory, the service-specific CLAUDE.md takes precedence for quirks and patterns:

- **`api/CLAUDE.md`** — Go/gRPC rules, generated code handling
- **`web/CLAUDE.md`** — React/TypeScript, test distinctions
- **`pipeline/CLAUDE.md`** — Python/Airflow, environment setup

## Special Capabilities

### Incident Response Runbook
A company-wide 20-step incident-response runbook is available as an interactive skill or slash command (not in this file). Request it by name or type the appropriate slash command when an incident occurs. It guides you through each step with context and best practices.

### Analytics Warehouse Access
Analysts and data engineers can run read-only SQL queries against the analytics warehouse via an MCP tool (separate from atlas engineering contexts). This tool enforces read-only access and validates queries before execution.
```
