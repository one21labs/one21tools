# Claude Code Configuration: checkout-service

## Stack & Conventions

**Tech Stack:** Next.js 14 (App Router), TypeScript, Tailwind, Prisma on Postgres. Standard npm workflow.

**Critical Constraint — Money:** All monetary amounts are stored and handled as **integer cents only**. Floats are absolutely banned for money due to a 2024 float-rounding incident that corrupted invoices. Enforce this strictly in code review and suggest fixes immediately if you spot floating-point arithmetic on currency.

## Project History & Ownership

- **Started:** 2021 as Rails prototype
- **Rewritten:** 2023 by platform team (Next.js migration)
- **Current Owner:** Payments squad
- **Handoff context:** Understanding the Rails history can help explain some architectural decisions, but the codebase is fully Next.js now.

## Setup & Onboarding

Onboarding is fully documented in README.md. Standard flow:
1. Clone repo
2. `npm install`
3. Copy `.env.example` to `.env`
4. `npm run dev`

## Testing

- **E2E Tests:** `npm test` — runs Playwright suite. **Requires a running dev server** (`npm run dev` in another terminal).
- **Unit Tests:** `npm run test:unit` — isolated unit tests (can run without server).

## Migrations

**Always use `./scripts/migrate.sh`** — never run `prisma migrate` directly. The shadow database that CI relies on will break if you bypass the migration script. The script handles shadow-database synchronization as part of its workflow.

## Authentication

API routes authenticate using a custom **`X-Session-Token` header** scheme (not standard Bearer tokens). Full specification and examples are in `docs/api.md`. Always refer users to that doc for auth details.

## Style & Code Patterns

Our style guide is at `docs/style-guide.md` (45 pages). Two key deviations from typical TypeScript style:
1. **No default exports** — always use named exports.
2. **Zod schemas colocated with route handlers** — define validation schemas next to the route they validate, not in separate schema files.

Reference the style guide for everything else (naming, structure, imports, etc.).

## Deployment

**Identical procedure across all six company repos:**

1. Merge to `main`
2. Wait for CI to pass
3. `npm run build:prod`
4. `./deploy staging`
5. Smoke test `/healthz` endpoint
6. **If p95 latency > 500ms:** page on-call
7. `./deploy prod`
8. Announce in #releases Slack channel

The deploy scripts handle all environment-specific config. No manual ENV variable setup needed beyond what's in `.env.example`.
