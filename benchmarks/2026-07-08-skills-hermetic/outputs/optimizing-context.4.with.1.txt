## Design rationale (short version)

- Root `CLAUDE.md` stays lean: only facts true for *every* file in the repo (commit style, `make check` gate, trunk-based dev). Service-specific stuff goes in nested CLAUDE.md files that Claude Code loads automatically once it's working in that directory — this avoids the root file bloating every session even when you're only touching `web/`.
- The incident runbook is a repeated, identical-everywhere procedure → that's a **Skill**, not inline text in CLAUDE.md (progressive disclosure: it only loads context when actually invoked). Since it's identical across the whole company, not just atlas, the better long-term home is a company-wide plugin installed into every repo — I've noted that as the target state, with a local skill as the concrete starting point.
- Live SQL against the warehouse is external tool/data access → **MCP**, not Claude shelling out with ambient DB credentials via Bash. A project-scoped `.mcp.json` makes the connection visible, versioned, and auditable.
- The three service quirks (`api/gen/` generated code, `web/` test command trap, `pipeline/` env var precondition) are all guardrails Claude will get wrong *by default* — each goes at the lowest altitude, right next to the code it governs, so it loads exactly when relevant.

```markdown
## What goes where, and why

| Location | Mechanism | Rationale |
|---|---|---|
| `CLAUDE.md` (root) | CLAUDE.md | Always loaded; holds only conventions true repo-wide (conventional commits, `make check` gate, trunk-based dev) so they aren't duplicated or drift across three service files. |
| `.mcp.json` | MCP | Analysts need live SQL against the warehouse — MCP gives real-time, scoped, auditable access instead of Claude running ad-hoc DB clients from Bash with ambient credentials. |
| `.claude/skills/incident-response/SKILL.md` | Skill | The runbook is a repeated procedure invoked by name ("walk me through the incident runbook"), identical for every team — it should load on demand, not sit inline in CLAUDE.md and cost context on every unrelated session. |
| `api/CLAUDE.md` | CLAUDE.md (nested) | Go/gRPC conventions only matter once Claude is working inside `api/`; keeping them out of root keeps the root file relevant to web/pipeline work too. |
| `api/gen/CLAUDE.md` | CLAUDE.md (nested, minimal) | "Generated by `make proto`, hand-edits get silently overwritten" needs to surface exactly when Claude is about to edit inside `gen/` — lowest altitude, right next to the code it governs. |
| `web/CLAUDE.md` | CLAUDE.md (nested) | `npm test` launching real Playwright browsers vs. `npm run test:unit` is a trap Claude will hit by default ("run the tests" → wrong command); stating it here means it's loaded before Claude acts in `web/`. |
| `pipeline/CLAUDE.md` | CLAUDE.md (nested) | `AIRFLOW_HOME=pipeline/.airflow` is a silent precondition — DAGs just fail to import without it. Belongs where Claude will actually run pipeline commands. |

## Notes / follow-ups (not built yet, worth deciding on)

- **Incident-response runbook as a plugin, not a per-repo skill.** Since it's identical across every team, copying it into each repo's `.claude/skills/` risks drift (someone edits atlas's copy, other repos don't get the fix). The SSoT home is one org-wide plugin published to an internal marketplace and installed into atlas (and every other repo) by reference. I've laid out the local skill as the concrete starting point since only atlas is in scope today — treat it as a placeholder to migrate once the plugin exists.
- **`make check` enforcement.** CLAUDE.md states the rule, but nothing stops Claude from skipping it under time pressure. Consider a hook (PreToolUse on `git push` / PR-creation commands, or a Stop hook) that actually runs `make check` and blocks on failure, rather than relying on Claude remembering the instruction.
- **Warehouse SQL access scope.** Decide whether `.mcp.json` credentials are shared/service-level (checked-in config, org-managed auth) or per-analyst (each person authorizes their own connection). If access should be read-only and restricted to a subset of tools, consider a dedicated subagent (e.g. `warehouse-analyst`) rather than exposing the raw MCP server repo-wide.
```

```markdown
# atlas

Monorepo with three services, each with its own CLAUDE.md — read it once you're
working inside that directory:
- `api/` — Go, gRPC
- `web/` — React, TypeScript
- `pipeline/` — Python, Airflow

## Conventions (apply everywhere)

- Commits: Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`, etc.).
- `make check` must pass before any PR is opened or updated. Run it before
  considering work done, not just before pushing.
- Trunk-based development: no long-lived branches. Keep branches short and
  merge to main quickly.

## Company-wide procedures

- Incident response: use the `incident-response` skill (20-step runbook, same
  across every team). Invoke it and follow it rather than improvising steps.

## Analytics

- Live SQL against the analytics warehouse is available via the `warehouse`
  MCP server (see `.mcp.json`). Use it to answer product-metrics questions
  directly instead of guessing from code or stale docs.
```
