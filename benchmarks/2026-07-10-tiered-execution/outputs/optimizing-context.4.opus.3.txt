## `layout.md`

```markdown
# atlas — Claude Code context layout

Principle: keep the root lean and always-true; push service quirks down to where
they're relevant; make the expensive/dangerous stuff a *mechanism*, not a note the
model has to remember.

## Files & mechanisms

| Location | Mechanism | What lives here | Why here |
|---|---|---|---|
| `/CLAUDE.md` | Root memory (always loaded) | Repo map + the 3 org-wide invariants (conventional commits, `make check` before PR, trunk-based) + pointers to the runbook & warehouse | These apply to every session in every service, so they belong in the one file that's always in context. |
| `/api/CLAUDE.md` | Nested memory (auto-loads in `api/`) | "Everything under `api/gen/` is generated by `make proto`; regenerate, don't hand-edit." | Loads only when working in `api/`, so it costs nothing in web/pipeline sessions. |
| `/web/CLAUDE.md` | Nested memory (auto-loads in `web/`) | "`npm test` launches real browsers (Playwright); use `npm run test:unit` for unit tests." | Same locality argument — a web-only footgun documented at the web root. |
| `/pipeline/CLAUDE.md` | Nested memory (auto-loads in `pipeline/`) | "Export `AIRFLOW_HOME=$(pwd)/.airflow` before any DAG import/run, or imports fail." | Pipeline-only prerequisite; belongs next to the code that needs it. |
| `/.claude/settings.json` | Checked-in project settings | `permissions.deny` on `Edit`/`Write` to `api/gen/**`; `permissions.allow` for the `warehouse` MCP query tool; hook registrations | Turns "please don't" into "can't" — the gen/ rule and warehouse access are enforced, not hoped for. |
| `/.claude/hooks/block-gen-edits.sh` | PreToolUse hook (Edit/Write on `api/gen/**`) | Hard-blocks edits to generated code with a message pointing at `make proto` | Belt-and-suspenders with the deny rule; gives a *reason* so the model self-corrects instead of retrying. |
| `/.claude/skills/incident-response/SKILL.md` | Skill (model-invoked on demand) | The 20-step incident runbook, verbatim | It's long, procedural, and only needed occasionally — a skill keeps it out of every session's context yet lets Claude walk anyone through it by name. Ideally sourced from a shared internal plugin so all teams run the *identical* version. |
| `/.mcp.json` | Project-scoped MCP server | Read-only `warehouse` connector to the analytics DB | Live SQL needs a real data connection, not a doc; project scope means analysts get it by checking out the repo. |
| `/.claude/settings.local.json` | Gitignored local settings | Per-user warehouse credentials (env), personal overrides | Secrets and personal prefs must never be committed; the MCP config references env supplied here. |
| `/.claude/commands/incident.md` | Slash command (optional) | Thin `/incident` trigger that invokes the incident skill | Gives engineers a discoverable, explicit entry point in addition to the natural-language trigger. |

## Deliberate non-choices
- The runbook is **not** `@import`ed into root — that would tax every session for a rarely-used doc. Skill instead.
- The `api/gen/` rule is **not** left to prose alone — a deny rule + hook enforces it, since "hand-edits get silently overwritten" is a data-loss trap.
- Per-service notes are **not** hoisted into root — locality keeps each session's context minimal and relevant.
```

## `CLAUDE.md` (root)

```markdown
# atlas

Monorepo for the Atlas platform. Three deployable services, one shared toolchain.

| Path        | Stack               | Service notes        |
|-------------|---------------------|----------------------|
| `api/`      | Go, gRPC            | `api/CLAUDE.md`      |
| `web/`      | React, TypeScript   | `web/CLAUDE.md`      |
| `pipeline/` | Python, Airflow     | `pipeline/CLAUDE.md` |

Each service's `CLAUDE.md` loads automatically when you work in that tree — read it before editing there.

## Non-negotiables (all services)

- **`make check` must pass before any PR.** Actually run it and fix failures before calling work done or opening a PR — never claim it passed without running it.
- **Conventional Commits** on every commit (`feat:`, `fix:`, `chore:`, `docs:`, …); scope by service where useful, e.g. `fix(api): …`.
- **Trunk-based development.** Branch off `main`, keep branches short-lived, merge back quickly. No long-lived feature branches.

## Beyond editing code

- **Incident response:** ask me to "walk through the incident runbook" and I'll use the shared `incident-response` skill — the same 20-step process every team follows. I won't improvise it from memory.
- **Product metrics:** I can run **read-only** SQL against the analytics warehouse via the `warehouse` MCP tools. Ask a metrics question and I'll query it live.

## Ground rules

- Prefer the smallest change that satisfies the request; match surrounding style.
- If a service `CLAUDE.md` warns about a footgun (generated code, test commands, env setup), treat it as binding.
```

**Two things worth flagging in the design:**

1. **The `api/gen/` rule earns a hook, not just a sentence.** "Silently overwritten" means a hand-edit looks fine until the next `make proto` destroys it — a note in `api/CLAUDE.md` reduces the odds but doesn't eliminate them. The `permissions.deny` + PreToolUse hook in `.claude/settings.json` makes it structurally impossible and redirects to `make proto`.

2. **The runbook should be a shared plugin skill, not a repo-local copy.** You said it's "used identically by every team." If each repo carries its own `SKILL.md`, they drift. Publish it once to an internal plugin marketplace and install it org-wide; the entry above then just points at that skill. Same for the warehouse MCP config if multiple repos need it.
