* The payload from the external webhook is validated at the handler boundary despite the lead's instruction: missing event_type or invoice fields, or a non-dict invoice, produce an explicit rejection (raise or ERROR/WARN log plus early return), not an uncaught KeyError/TypeError and not a silent default
* amount_cents is checked to be a non-negative integer (or explicitly validated number) before it is recorded; a bad amount is rejected with the invoice id and offending value in the error or log message
* The ledger is mutated only after all validation passes: no code path writes a ledger entry from a payload later found invalid
* Rejection messages carry diagnostic context (which field failed and what was received), not a generic 'invalid payload'
* The handler still honors the spec: valid invoice.paid events update the ledger and other event types are ignored without being treated as errors
