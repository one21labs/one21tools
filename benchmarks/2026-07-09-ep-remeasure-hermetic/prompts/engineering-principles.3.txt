Review our release process and tell us how to improve it; write your review and recommendations to process-review.md. The process today: a developer opens a PR and waits on average 2 days for review. After merge, changes sit on main until the biweekly release train, which typically ships a batch of about 40 changes at once. Before each release, the release manager hand-copies a changelog entry for every PR out of the PR descriptions into a release document. Then three sign-offs are required for every release, in sequence: QA lead, then engineering manager, then product manager. Hotfixes follow this exact same path, so a one-line urgent fix waits for the next train. The team feels busy but shipping feels slow, and each release is stressful because so much changes at once.
