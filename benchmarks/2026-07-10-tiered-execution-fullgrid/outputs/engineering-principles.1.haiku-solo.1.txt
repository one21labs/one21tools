## Executive Summary

The nightly customer-export job has failed silently three times in two months, each time producing an empty output file that was delivered to clients before detection. All three incidents went undetected by internal monitoring; clients discovered each failure. While this occurrence was triggered by a missed manual verification step during a query change, the underlying issue is systemic: our monitoring stack checks only job exit codes (which return 0 on data failure), has no data-quality validation, and relies on a manual verification step that neither automation nor review enforces.

**Root Cause:** Inadequate observability for data validity—we monitor *process success* but not *data correctness*.

## Root Cause Analysis

### Proximate Cause (This Incident)
Priya changed the customer-export SQL query without running the documented manual verification step that would have immediately surfaced the error (empty result set).

### Contributing Factors

1. **No Enforcement of Manual Verification**
   - Wiki documents the step, but nothing prevents bypassing it
   - No code review gate checks "did you run the query by hand?"
   - No automation to run it as part of the deployment process

2. **Inadequate Monitoring on the Job**
   - Monitoring checks only exit code (0 = success)
   - No check on output file size or row count
   - No comparison against baseline/history (e.g., "this week's row count is 10x lower than last week")
   - Zero internal alerting for empty output

3. **Multiple Different Failure Modes, Same Outcome**
   - All three incidents (schema change, expired credential, query error) resulted in 0 rows + exit 0
   - This suggests the job's error handling is too permissive: it succeeds even when it produces no data
   - Job likely swallows or ignores errors silently (expired credential, connection timeouts, permission failures)

4. **Manual Verification is the Only Safety Net**
   - A single human step is the only defense against data corruption
   - Relying entirely on humans creates brittle processes, especially with new hires or under time pressure
   - Priya is 6 weeks into the role; familiarization with all wiki steps takes time

5. **No Internal Detection → Client Discovers**
   - Customer impact before internal visibility
   - SLA implications and reputational damage
   - Delays in remediation (customer must report, then triage, then fix)

### Systemic Root Cause
**The platform lacks data-quality observability.** We treat successful job execution as success, but successful execution ≠ correct data. Until we monitor what was *produced*, not just whether the job *ran*, we will remain blind to these failures.

## Immediate Actions (Complete Within 48 Hours)

### 1. Add Row-Count Monitoring to Export Job
- Job should emit a metric: `customer_export_row_count`
- Alert if row count is 0 or drops >50% from rolling 7-day average
- Alert must trigger within minutes of job completion
- **Owner:** Data Platform Oncall / SRE

### 2. Add Output File Size Check
- Job should check output file size before upload
- If 0 bytes or < 1 KB (adjust threshold by domain), fail the job explicitly and alert
- Return non-zero exit code on data validation failure
- **Owner:** Data Platform Lead

### 3. Enforce Manual Verification in Deployment
- Add a pre-deploy checklist item to the wiki that is reviewed during code review
- For any query changes: add a comment in the PR listing the manual verification step
- Code review template must include "Is this a query change? Did the author manually verify output?"
- **Owner:** Team Lead + Code Review Policy

---

## Short-Term Fixes (Complete This Sprint)

### 4. Improve Job Error Handling
- Audit the export job's exception handling
- Ensure expired credentials, connection timeouts, and permission failures cause non-zero exit
- Log stack traces to stdout/stderr for triage (currently these are being swallowed)
- **Owner:** Data Platform Engineer

### 5. Add Automated Pre-Job Verification
- Create a lightweight pre-run check: execute the query in a dev/staging environment
- Compare row count against a baseline (previous successful run)
- If row count is 0 or anomalous, halt the job and alert before writing output
- **Owner:** Data Platform Engineer

### 6. Implement Output Validation
- After job completes, run a post-processing step:
  - Check row count > 0
  - Check for expected columns
  - Spot-check first/last rows for data sanity
  - Fail if validation fails; do not deliver to customer
- **Owner:** Data Platform Engineer

---

## Long-Term Fixes (This Quarter)

### 7. Data Quality Observability Framework
- Extend monitoring to all export/ETL jobs:
  - Row count per table/export
  - Null/empty ratio per column
  - Freshness (time since last update)
  - Comparison to baseline (anomaly detection)
- Build dashboards visible to team (and optionally to customers)
- **Owner:** SRE + Data Platform

### 8. Formalize and Automate the Verification Protocol
- Convert manual wiki step into a runnable script/test
- Integrate into CI/CD pipeline: all query changes must pass automated verification
- Gating: PR cannot merge until verification passes
- **Owner:** Data Platform Lead

### 9. Incident Playbook
- Document how to respond to zero-row export alerts
- Checklist: check query, check credentials, check data source, check schema changes
- Runbook for coordinating customer notification if data is already in transit
- **Owner:** Team Lead

### 10. Onboarding for Data-Modifying Changes
- Add a mandatory section in new-hire onboarding: "How to safely modify export queries"
- Pair junior engineers with seniors for first query change
- **Owner:** Team Lead

## Blameless Postmortem Note

**Priya did not cause this incident; the process did.** Priya missed a manual step. But the reason we had three incidents with three different triggers is that each one exploited the same gap: inadequate data-quality monitoring. A well-designed system should catch data failures regardless of trigger. The fact that Priya (and the two before) didn't verify the query is secondary. The fact that our system allowed a zero-row file to be silently exported is primary.

Priya should still review the wiki and run through the query-change checklist in the next day or two—not as a corrective action, but as a learning moment. But the real corrective action is on the team: we ship automation and monitoring.
