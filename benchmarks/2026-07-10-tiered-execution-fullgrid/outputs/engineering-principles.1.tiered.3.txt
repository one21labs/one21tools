# Post-Incident Analysis: Empty Customer-Export Delivery (3rd Occurrence)

## Summary

For the third time in two months, the nightly customer-export job produced and delivered an empty output file to a client, with no internal alert raised. This latest occurrence was triggered when Priya (an engineer onboarded six weeks ago) modified the export SQL query without executing the manual verification step documented in the team wiki. The job exited with status code 0 despite producing zero rows, and the empty file was delivered undetected; the client discovered the failure during their downstream processing.

## Root Cause Analysis

### Proximate Cause
Priya changed the export SQL query without executing the documented manual verification step (wiki: "after changing the query, run it once by hand and eyeball the row count"). This resulted in a query that produced zero rows, which was deployed and executed by the scheduled job.

### Systemic Root Cause
The job's success signal is decoupled from output correctness. The monitoring and alerting infrastructure checks only the exit code (0 = success), with no validation of output characteristics (row count, file size, schema match, or data anomalies). Zero-row output is treated as a valid success state and triggers delivery to the client. This gap has been exposed three times in two months by different failure modes — evidence that the root cause is systemic, not contingent on any single engineer's behavior.

The recurrence of identical failure modes across different proximate triggers is itself proof that individual behavioral fixes (reminders, training, checklists) are insufficient; the system itself enables and normalizes the silent delivery of empty exports.

## Contributing Factors

1. **Manual verification step exists in wiki only, not enforced or tooled.** The post-query-change verification is a social/cultural expectation, not an automated gate or CI step.

2. **New engineer onboarding gap.** Priya (six weeks tenure) did not have a structured checkpoint or runbook review covering this specific job's operational requirements.

3. **No output-correctness validation in the export pipeline.** The job produces a file and signals success based only on its exit code; no row-count thresholds, schema validation, or anomaly detection.

4. **No pre-delivery monitoring on file characteristics.** The delivery mechanism does not check file size, row count, or basic sanity before sending to client. No alert on suspiciously empty or anomalous exports.

5. **No alerting tied to query changes.** Schema or SQL modifications to the export query do not trigger additional validation or review gates.

### Why Not Individual-Behavior Fixes Alone

The three occurrences show that asking engineers to remember a wiki step or to be more careful is not sufficient. Priya followed a pattern that the system itself enabled and normalized. The first two incidents (schema rename, expired credential) occurred before Priya joined; they prove that this failure mode is not contingent on new-engineer onboarding or one person's discipline. All corrective actions above should prioritize system-level barriers and monitoring over reliance on human memory or vigilance.
