# Release Process Review

## Root Cause

Applying 5 Whys to "shipping feels slow and each release is stressful":

1. Why stressful? ~40 changes ship at once, so when something breaks, it's hard to isolate which change caused it.
2. Why 40 at once? Releases are batched to a fixed biweekly train instead of shipping as changes are ready.
3. Why batched? Every release — regardless of size or risk — carries the same fixed overhead: manual changelog compilation plus three sequential sign-offs.
4. Why is that overhead fixed and high? The process has no cheap path; a one-line fix and a 40-PR batch both pay the full manual-changelog + QA→EM→PM toll.
5. **Root cause:** the process couples release cadence to a uniform, manual, high-overhead ceremony, so the team batches changes to amortize that cost — which is also why a "hotfix" can't actually be fast.

**Countermeasure:** cut per-release overhead and make it scale with risk, not with a fixed ceremony. That lets cadence decouple from batch size — enabling smaller, more frequent, less stressful releases and a real expedited path for urgent fixes.

## Waste Identified

| Step | Waste type | Detail |
|---|---|---|
| Hand-copying changelog from PR descriptions | Overprocessing + transcription defects (SSoT violation) | The PR description is already the source of truth for the entry; manually retyping it into another document is duplication that can drift and consumes the release manager's time on every release |
| Three sign-offs, every release, in strict sequence | Overprocessing + waiting | A one-line fix pays the same approval cost as a 40-change batch; sequencing (rather than parallelizing) adds pure wait time between approvers |
| Hotfixes on the biweekly train | Waiting | Defeats the purpose of "hotfix" — urgent, low-risk fixes wait up to 2 weeks behind unrelated changes |
| 40-change batches | Inventory + risk concentration | Large batch size hides which change caused a regression, raises the cost of investigation, and concentrates all release-day risk into one event |
| 2-day average PR review wait | Waiting | Worth investigating separately — but note it compounds with the batch wait, so total latency from "code done" to "code live" can be weeks |

## Recommendations (ranked by impact/effort)

1. **Give hotfixes a real expedited path.** Define "hotfix" criteria (small diff, urgent, low blast radius) and let it skip the train with a lightweight approval (e.g., one sign-off, async). This is the most urgent gap — right now nothing in the process is actually a hotfix.

2. **Automate changelog generation.** Generate release notes from PR titles/descriptions via script or CI at merge or release-cut time. Establishes the PR description as the single source of truth and removes manual copying and its error rate entirely.

3. **Right-size sign-offs by risk, not blanket rule.** Not every change needs QA + EM + PM. Classify changes (e.g., low-risk/config/docs → 1 approver; standard → current process; high-risk → full process). Reserve the full sequence for changes that actually warrant it.

4. **Parallelize sign-offs where there's no real dependency.** If QA, EM, and PM aren't each blocked on the prior approval's output, review concurrently instead of in sequence — cuts wall-clock wait without cutting rigor.

5. **Shrink batch size / increase release frequency.** Once (1)–(4) lower per-release overhead, move toward smaller, more frequent releases (e.g., weekly or on-demand) instead of a fixed biweekly train. Smaller batches make regressions easier to isolate and reduce release-day stress.

6. **Investigate the 2-day PR review wait separately.** Determine if it's reviewer capacity, PR size, or lack of an SLA — this is a distinct bottleneck from the release train and stacks on top of it.

7. **Add a feedback loop (PDSA).** Track cycle time (merge → live), defect rate per release, and batch size. Use it to confirm these changes actually reduce stress and latency rather than assuming they will.
