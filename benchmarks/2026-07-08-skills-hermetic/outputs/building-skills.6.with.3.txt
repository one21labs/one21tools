Also, two things about the style guide as stated:

- **The emoji-header rule** — fine, purely cosmetic, I applied it.
- **The "A friendly assistant that helps with `<topic>`" description rule** — I did **not** apply this one, and wanted to flag why rather than silently deviate. The `description:` field isn't catalog copy — it's the only part of a skill guaranteed to stay in Claude's context at all times, and Claude uses its exact wording to decide *when* to invoke the skill at all. This draft's description also smuggles in a standing operational rule ("report row counts before and after every operation") that isn't repeated with the same force in the body. Replacing it with a generic templated sentence would (a) strip the trigger conditions, so this skill would likely stop firing reliably on "clean this CSV" type requests, (b) drop that row-count instruction, and (c) make every skill's description identical except one noun, which defeats the purpose of the field. If your publish pipeline uses standard skill-validation tooling, it's also likely to reject both a non-trigger-leading description and emoji-laden headers outright as lint errors, not just style nits — worth checking before this goes out, and possibly worth spot-checking whether the other 12 already-published skills have this same activation problem. I couldn't verify that claim myself in this session.

If the catalog UI genuinely needs a uniform, friendly one-liner for human browsing, the clean fix is a separate display-only field (whatever your marketplace schema supports for that), left independent from `description`. I didn't invent one speculatively since I don't know your ingestion schema.

2. Report row count, duplicate count, and columns with mixed formats before changing anything.

4. Write the cleaned file next to the original with a .cleaned.csv suffix.
