## 4. Root Cause Analysis

### 4.1 Immediate trigger (this occurrence)
Export SQL query was changed without running the documented manual verification
step, so the query's new zero-row behavior went unnoticed before it ran in
production.

### 4.2 Five Whys — why did this reach the client?

1. **Why was an empty file delivered to the client?**
   Because the job produced zero rows and nothing stopped delivery.
2. **Why did nothing stop delivery?**
   Because the only success signal the pipeline checks is the process exit code,
   and the job exits 0 regardless of row count.
3. **Why does the job exit 0 on zero rows?**
   Because "job completed without throwing" and "job produced correct output" are
   treated as the same condition — there is no output-data validation step.
4. **Why is there no output-data validation step?**
   Because the only safeguard against a bad query/data problem is a *manual,
   human-executed* check ("eyeball the row count"), documented in a wiki rather
   than enforced in the pipeline.
5. **Why does the team rely on a manual, undocumented-as-enforced check for a
   correctness-critical, client-facing job?**
   Because verification has been treated as a process/documentation problem
   (write it in the wiki, expect people to follow it) rather than an engineering
   control problem (make the pipeline itself unable to ship bad output).

### 4.3 Systemic root cause

**There is no automated gate between "job produced output" and "output is
delivered to the client."** Correctness depends entirely on (a) an individual
remembering an undocumented-as-enforced manual step, and (b) exit code, which is
provably not a proxy for output correctness. This root cause is trigger-agnostic —
it's why a schema rename, a swallowed credential error, and a skipped manual step
all funneled into the exact same undetected outcome.

A secondary, related root cause specific to Incident #2: the job **swallows
errors** (expired credential didn't cause a failure) instead of failing loudly on
conditions that should abort the run. This is the same underlying pattern —
success is measured by "didn't crash," not "did the right thing."

## 5. Contributing Factors

- **No poka-yoke on the query-change workflow.** The verification step exists
  only as institutional knowledge (wiki), with no technical enforcement (e.g., a
  required CI check, a required PR reviewer sign-off, a staging dry-run gate).
  This makes the control's reliability a function of who happens to be doing the
  work — predictably fragile for anyone new to the team, not a Priya-specific gap.
- **Monitoring measures the wrong thing.** Exit code is a *process* health
  signal, not a *data* correctness signal. For a job whose entire purpose is
  "produce correct data," data-shape checks (row count, file size, schema) should
  be a first-class monitor, not absent.
- **Delivery is not gated on validation.** Even if validation existed, nothing in
  this incident suggests output is held/quarantined pending a sanity check before
  it reaches the client — it appears to ship unconditionally on job completion.
- **Exception handling swallows failures** (seen directly in Incident #2, and
  plausibly relevant here too) — errors are absorbed rather than surfaced, which
  converts "the job broke" into "the job silently did nothing," the worst possible
  failure mode for a delivery pipeline.
- **No feedback loop from prior incidents to systemic fixes.** If Incidents #1
  and #2 were closed out with fixes scoped only to their specific trigger (fix the
  schema reference, rotate the credential) rather than the shared detection gap,
  that would explain why a third, unrelated trigger produced the identical
  outcome. *(Open question — see §9.)*

## 6. Why This Keeps Happening

Each prior RCA (if trigger-scoped) likely "fixed" the specific cause and closed
the ticket, leaving the shared gap — no output validation, no delivery gate —
untouched. A new trigger simply walks through the same open door. Until the
detection/delivery gate is fixed, a fourth trigger (a new one, not yet seen) will
produce the same outcome. **Fixing triggers one at a time has a 0% success rate
here across two months; fixing the shared gap is the only intervention that
generalizes.**

## 7. Corrective Actions

### 7.1 Must-fix — closes the gap regardless of future trigger (poka-yoke)
1. **Add an output-validation gate to the job itself.** Before marking success/
   delivering, check row count against a sane floor (e.g., `> 0`, or within N% of
   a trailing historical baseline). Fail loudly and non-zero-exit if it doesn't
   pass.
2. **Decouple delivery from job completion.** Output should land in a
   quarantine/staging location first; delivery to the client is a separate,
   gated step that only fires after validation passes. The client must never be
   able to receive output the pipeline hasn't vouched for.
3. **Fix monitoring to check data, not just process.** Alert on zero rows,
   anomalously small file size, or row-count deviation from trend — not exit
   code alone.
4. **Audit and remove swallowed exceptions in this job**, especially around
   auth/credential failures. Auth errors should abort the run loudly, not
   degrade into "ran fine, wrote nothing."

### 7.2 Should-fix — reduces recurrence probability further
5. **Make the pre-deploy verification step a technical gate, not a wiki
   instruction.** E.g., require a CI job that runs the query against
   staging/prod-replica and reports row count as part of PR review, or require an
   explicit "verified: N rows" comment/check before merge for changes to export
   queries.
6. **Trend-based anomaly detection**, not just a zero-row check — a query that
   returns 10 rows instead of the usual 10,000 is just as bad and wouldn't be
   caught by a simple `> 0` check.

### 7.3 Consider
7. Add this job (and any similarly client-facing export job) to an inventory of
   "jobs where success = data correctness," and audit them for the same gap
   proactively rather than one incident at a time.
8. Onboarding checklist for new engineers touching client-facing pipelines — but
   treat this as belt-and-suspenders, *not* the primary fix; it doesn't help
   against triggers #1 or #2, and per §6, human-process fixes alone haven't
   generalized so far.

## 10. Lessons Learned

- **A manual, wiki-documented step is not a control** for anything client-facing
  or correctness-critical — it's a suggestion. Controls that matter need to be
  technically enforced (poka-yoke), not remembered.
- **Exit code 0 is not a correctness signal.** Any pipeline whose job is "produce
  correct data" needs a data-shape check as a distinct, first-class success
  condition.
- **Fixing the trigger without fixing the detection gap guarantees recurrence
  with a new trigger.** This incident is the clearest possible evidence of that:
  three unrelated causes, identical undetected outcome.
- The client should never be the first party to notice a data delivery failure —
  that's a detection/alerting gap regardless of what caused the failure.
