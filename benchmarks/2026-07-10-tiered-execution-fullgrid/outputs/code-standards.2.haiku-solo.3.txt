# Code Review: Refund Handler

## Defects Found

### 1. **CRITICAL: State mutation before API call**
**Location:** Lines 9-11
**Issue:** Order status and refunded amount are persisted to storage before the payment API call succeeds. If the refund fails, the order is left in an inconsistent state (marked refunded but payment wasn't processed).

**Fix:** Only update order status/amount after confirming successful refund:
```python
try:
    result = payments_api.refund(order.payment_ref, amount)
except Exception:
    log.error("refund error")
    return False
if result.get("status") != "ok":
    log.warning("refund not ok")
    return False
order.status = "refunded"
order.refunded_amount = amount
```

---

### 2. **CRITICAL: Silent failure on failed refund**
**Location:** Line 16
**Issue:** When `result.get("status") != "ok"`, the function logs a warning but returns `True`, signaling success to the caller even though the refund failed.

**Fix:** Return `False` when refund is not ok:
```python
if result.get("status") != "ok":
    log.warning("refund not ok")
    return False
```

---

### 3. **HIGH: Bare exception handler**
**Location:** Line 13
**Issue:** `except Exception` catches all exceptions including system-level ones. No context about the actual error is logged, making debugging difficult.

**Fix:** Catch specific exceptions and log the error:
```python
except Exception as e:
    log.error(f"refund error for order {order_id}: {e}")
    return False
```

---

### 4. **HIGH: Unhandled KeyError**
**Location:** Line 9
**Issue:** `orders[order_id]` raises `KeyError` if the order doesn't exist. No validation before access.

**Fix:** Check order existence first:
```python
if order_id not in orders:
    log.error(f"order not found: {order_id}")
    return False
order = orders[order_id]
```

---

### 5. **HIGH: Incorrect fee calculation logic**
**Location:** Line 17
**Issue:** Calculating and storing a fee on the order during a refund doesn't make business sense. Fees should typically be deducted during payment, not added during refund. This appears to be backwards or misplaced.

**Fix:** Remove this line or clarify intent. If refunding should reduce the fee:
```python
order.fee = max(0, order.fee - amount * FEE_RATE)
```

---

### 6. **MEDIUM: Insufficient error logging**
**Location:** Lines 14, 16
**Issue:** Log messages lack context (order_id, amount, actual exception). Difficult to debug in production.

**Fix:** Include relevant context:
```python
log.error(f"refund failed for order {order_id} amount {amount}: {e}")
log.warning(f"refund returned non-ok status for order {order_id}: {result}")
```

---

### 7. **MEDIUM: No input validation**
**Location:** Line 8
**Issue:** Amount could be negative, zero, or invalid. No guard rails.

**Fix:** Validate inputs:
```python
if amount <= 0:
    log.error(f"invalid refund amount: {amount}")
    return False
```

---

### 8. **LOW: Incomplete API response handling**
**Location:** Line 16
**Issue:** Only checks `status` field. API might return other error indicators that are ignored.

**Fix:** Log the full result for debugging:
```python
if result.get("status") != "ok":
    log.warning(f"refund returned non-ok status for order {order_id}: {result}")
    return False
```
