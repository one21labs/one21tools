# Claude Code context layout — atlas monorepo

Claude Code auto-loads CLAUDE.md from the cwd up to repo root, and pulls in
a subdirectory's CLAUDE.md on demand once Claude reads/edits files there.
Design here leans on that: root CLAUDE.md carries only what's true everywhere;
service-specific quirks live in service-scoped files so a session working
purely in web/ never pays context cost for pipeline/ details, and vice versa.

| Path | Mechanism | Rationale |
|---|---|---|
| `/CLAUDE.md` | root memory (always loaded) | Org-wide rules that apply regardless of which service is touched: conventional commits, `make check` gate, trunk-based dev, plus a map pointing to everything below so Claude knows what else exists. |
| `/api/CLAUDE.md` | nested memory (loaded when working in `api/`) | Go/gRPC conventions + explicit warning that `api/gen/**` is generated by `make proto` and hand-edits are silently discarded — scoped here so it's only in context when relevant. |
| `/web/CLAUDE.md` | nested memory (loaded when working in `web/`) | React/TS conventions + the `npm test` vs `npm run test:unit` distinction, so Claude never launches real Playwright browsers unattended when it just wants a quick unit-test loop. |
| `/pipeline/CLAUDE.md` | nested memory (loaded when working in `pipeline/`) | Python/Airflow conventions + explains *why* DAG imports fail without `AIRFLOW_HOME` (belt-and-suspenders backup to the settings.json fix below, for humans and for sessions that inspect the file directly). |
| `/.claude/settings.json` — `env` | project settings (checked in) | Sets `AIRFLOW_HOME=pipeline/.airflow` for every Bash call Claude makes. Stronger than documenting it: a forgotten env var is a whole failure class removed rather than relying on the doc being read. |
| `/.claude/settings.json` — `permissions.deny` | project settings (checked in) | `Edit`/`Write` denied on `api/gen/**` so Claude structurally can't hand-edit generated code — the CLAUDE.md note is guidance, this is the guardrail. |
| `/.claude/settings.json` — `permissions.allow` | project settings (checked in) | Pre-approves safe, frequent commands (`make check`, `make proto`, `git commit`) so Claude isn't stopped for permission prompts on the exact loop it's told to run constantly. |
| `/.mcp.json` | project-scoped MCP server config (checked in) | Registers a read-scoped `analytics-warehouse` MCP server so any analyst gets live-SQL-to-the-warehouse as a governed tool call, not Claude shelling out with hand-managed credentials. |
| `/.claude/commands/incident-response.md` | custom slash command | The 20-step runbook is identical across every team and already-canonical — encode it **once** as `/incident-response` instead of pasting it into every service's CLAUDE.md (or worse, letting Claude improvise a sequence from memory). |

### Notes / things to fill in
- `incident-response.md` below is a **scaffold** — paste your org's actual 20 steps in verbatim where marked. Don't let Claude paraphrase a safety-critical runbook from a summary.
- `.mcp.json` server args/URL for the warehouse connector aren't specified here (depends on your warehouse — Snowflake/BigQuery/Postgres) — restrict the credential/role it uses to read-only, since "run live SQL when asked about metrics" should never imply write access.
- Root CLAUDE.md deliberately does **not** `@import` the three service CLAUDE.md files — importing pulls them into every session's context unconditionally, defeating the point of scoping them per-directory.

## `CLAUDE.md`

```markdown
# atlas

Monorepo with three services:
- `api/` — Go, gRPC
- `web/` — React, TypeScript
- `pipeline/` — Python, Airflow

Service-specific conventions live in each service's own CLAUDE.md
(`api/CLAUDE.md`, `web/CLAUDE.md`, `pipeline/CLAUDE.md`) and load
automatically once you're working in that directory.

## Non-negotiable, everywhere

- **Conventional Commits** for every commit message (`feat:`, `fix:`, `chore:`, etc.).
- **`make check` must pass** before any PR is opened or considered done. Run it
  yourself before telling the user a change is ready.
- **Trunk-based development** — no long-lived branches. Keep changes small,
  merge to main quickly.

## Generated code

Never hand-edit anything under `api/gen/`. It's regenerated by `make proto`
and hand-edits are silently overwritten. If generated output needs to
change, change the `.proto` source and re-run `make proto`. (Also enforced
via a deny rule in `.claude/settings.json`.)

## Incident response

If asked to run or walk through an incident, use `/incident-response`. It
encodes the company's standard 20-step runbook — the same one every team
uses. Don't improvise a different sequence.

## Analytics / warehouse queries

If an analyst asks about product metrics, use the `analytics-warehouse` MCP
tool to run live SQL directly against the warehouse rather than guessing
at numbers or writing throwaway scripts. That connection is read-only —
if a question requires a write or schema change, say so and stop.
```
