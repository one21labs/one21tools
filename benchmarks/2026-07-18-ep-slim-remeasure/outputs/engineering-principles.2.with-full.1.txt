```markdown
# Docs Restructuring Plan

## 1. Local setup steps
**Canonical location:** README.md — "Setup" section, using the current steps
(clone, npm install, copy .env.example, npm run dev). Corroborated by
CLAUDE.md's own lessons-learned note that the project moved to `npm run dev`
after a port clash, which matches README, not CONTRIBUTING.

**Other copies:**
- CONTRIBUTING.md: delete its inline copy entirely. Replace with a link:
  `See [Setup](README.md#setup) for local environment setup.`
  Do not restate any of the steps — a cross-reference carries the pointer only.

## 2. API timeout value
**Canonical location:** src/http/client.js (the actual constant). This is a
runtime value, not documentation content — no doc file is a valid home for it.

**Other copies:**
- README.md: remove the hardcoded "30 seconds" claim. If the timeout needs
  mentioning at all, phrase it as a pointer, not a number:
  `Request timeout is defined in src/http/client.js.`
- CONTRIBUTING.md: remove the hardcoded "45 seconds" claim (also the more
  wrong of the two). Same pointer treatment if needed, or drop it — a
  contributor-workflow doc likely doesn't need this fact at all.

## 3. Support email
**Canonical location:** README.md. Already the only copy — no change needed,
since it's user-facing and README is the entry point.

## 4. recalcTotals() invariant
**Canonical location:** src/billing.js — as a short comment at the mutation
site(s) for line items, e.g. `// call recalcTotals() after mutating line
items here`. This is a JIT (just-in-time) fact: it only matters to someone
editing this one file, so it should surface when they're in that file, not
in an always-loaded global context file.

**Other copies:**
- CLAUDE.md: remove the rule entirely. Do not leave a pointer either —
  CLAUDE.md is for cross-cutting/always-relevant rules, and this is scoped
  to a single file that will show the comment directly when touched.

## 5. "Lessons learned" section
**Canonical location:** none — delete outright, don't relocate.
Git history is the source of truth for past decisions and mistakes
(the npm start→npm run dev switch, the export rename, etc.). Narrating
that backstory in a doc creates a second, decaying copy of information git
already owns permanently. If either fact still matters as *current* truth
(e.g., "we use npm run dev"), that's already captured in README's setup
steps (item 1) — the historical narrative around *why* adds nothing
actionable and should just go.
```

Net effect: README becomes the sole source for setup + support contact, the timeout number disappears from prose entirely (code is SSoT), CONTRIBUTING only links out, the billing invariant moves into the code it governs, and the lessons-learned narrative is deleted rather than moved.
