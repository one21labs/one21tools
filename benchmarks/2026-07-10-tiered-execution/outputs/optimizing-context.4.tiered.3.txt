# Atlas Monorepo — Claude Code Context

## Shared Org-Wide Conventions

Every team member and every service in this monorepo adheres to these practices. Claude enforces them across all work:

### Commit Discipline
- Use [Conventional Commits](https://www.conventionalcommits.org/) for all commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, etc.
- Write clear, imperative-mood commit messages ("add retry logic" not "added retry logic").

### Pre-PR Quality Gate
- **`make check` must pass before any PR is opened.** This gate encompasses linting, type checking, test suites, and other quality checks.
- Do not bypass, skip, or override this check; every PR must pass.

### Trunk-Based Development
- Work on short-lived branches off `main`.
- No long-lived feature branches; integrate to main frequently.
- Keep PRs focused and small; rebase and squash before merge if needed.

---

## Service-Specific Contexts

This monorepo contains three independent services. **Load the service-specific `CLAUDE.md` when working in that directory**; it contains critical quirks and best practices:

- **`api/CLAUDE.md`** — Go/gRPC conventions, the `api/gen/` generated-code footgun, and build requirements.
- **`web/CLAUDE.md`** — TypeScript/React patterns, the `npm test` vs `npm run test:unit` distinction, and component testing.
- **`pipeline/CLAUDE.md`** — Airflow DAG structure, the mandatory `AIRFLOW_HOME=pipeline/.airflow` precondition, and Python conventions.

---

## On-Demand Tools & Runbooks

### Incident Response
Teams use a standardized 20-step incident-response runbook. If asked to guide someone through incident response:
- Type `/incident` to access the runbook.
- This runbook is identical across all teams; use it verbatim.

### Analytics & Product Metrics
Analysts can request Claude to query the analytics warehouse directly for product data:
- Claude has access to a database connector (MCP) configured in `.claude/mcp.json`.
- Execute **live** SQL queries when asked about metrics, trends, or dashboards.
- Always query live; do not rely on or provide cached or stale data.

---

## Getting Started

1. Identify which service you are working on (api/, web/, or pipeline/).
2. Read the corresponding `CLAUDE.md` in that service directory for specifics.
3. Run `make check` and confirm it passes before committing or opening a PR.
4. Use conventional commit messages.
5. Work on short-lived branches; merge to main frequently.

For incident walkthroughs or analytics queries, use the on-demand tools above.
