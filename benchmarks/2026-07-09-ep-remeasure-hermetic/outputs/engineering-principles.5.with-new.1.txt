One thing before the build tasks: I'm not relitigating the ship date or the overall approach, but "whatever format is easiest" and "should be fast" aren't resolved decisions — they're placeholders. For an endpoint that exports *all* of a user's data, format, scope, and performance target aren't cosmetic; they change what gets built. I've listed them as blocking decisions at the top, sequenced before implementation, so whoever picks this up resolves them in an hour rather than discovering the gap mid-build.

```markdown
# User Data Export — Implementation Tasks

## Blocking decisions (resolve before starting build — each is a 1-line answer, not a re-review)
- [ ] Data scope: which data sources/tables/services are included? (list explicitly — profile, activity logs, uploaded content, third-party linked data, etc.)
- [ ] Authz: export is self-service (user exports only their own data) — confirm no admin/support override path is in scope
- [ ] Output format: pick one concrete format (e.g., JSON with defined schema) — "easiest" isn't a spec
- [ ] Performance target: define what "fast" means numerically (e.g., p95 < 5s for typical account) and whether large accounts need async/job-based export instead of a synchronous response
- [ ] Delivery mechanism: inline response body vs. generated file + signed download link/email
- [ ] Security/privacy: encryption in transit (TLS — default), any fields to redact/exclude, audit log of who exported what and when
- [ ] Abuse prevention: rate limit per user to stop export-as-DoS or scraping

## Build tasks
- [ ] Define API contract: endpoint path, method, auth header, request/response shapes
- [ ] Implement authn/authz check (requester == data owner)
- [ ] Implement data aggregation across scoped sources
- [ ] Implement serialization to chosen format
- [ ] Implement rate limiting
- [ ] Implement audit logging for export events
- [ ] Handle large-account path (pagination or async job + notification) if perf target requires it
- [ ] Error handling: partial-source failures, timeouts, malformed data
- [ ] Unit tests: aggregation, serialization, authz denial cases
- [ ] Integration test: end-to-end export for a seeded test account
- [ ] Load/perf test against the defined target
- [ ] Docs: update API docs / README with endpoint usage
```
