# Development Standards

This document captures 30 development standards established by the team. All work should follow these guidelines.

## Code Style & Formatting

1. Use 4-space indentation in Python files.
2. Prefer f-strings over `.format()`.
3. Follow PEP 8 style guide.
4. Format code before committing.

## Code Quality

5. Write descriptive variable names.
6. Add docstrings to all public functions.
7. Keep functions short and focused.
8. Handle exceptions; do not let them crash silently.
9. Use logging, not print statements.
10. Error messages should include context about what failed.
11. Prefer composition over inheritance.
12. Do not use mutable default arguments.
13. Prefer early returns over deep nesting.

## Testing

14. Write unit tests for new code.
15. Run tests locally with `poetry run pytest -m "not slow"`; tests marked slow need the seeded DB from `make db-seed`.

## Development Workflow

16. Run `make check` before every commit; CI runs the identical target.
17. Never commit secrets to the repository.

## Git & Collaboration

18. Keep pull requests small.
19. Branch names use the ticket prefix (e.g., `PLAT-123-short-description`).
20. Production deploys happen only from the main branch.

## Database & Data Management

21. The events table schema is at version 9.
22. Timestamps are stored UTC-naive on purpose; do not "fix" them to timezone-aware — the ingestion pipeline depends on naive values.
23. Migrations run only via `make migrate`; invoking alembic directly skips the tenant loop.
24. Never edit files under `gen/`; they are regenerated by `make codegen`.

## Configuration & Infrastructure

25. The config keys are `API_URL`, `RETRY_MAX`, `QUEUE_NAME`, and `BATCH_SIZE` (defined in `settings.py`).
26. All outbound HTTP goes through the wrapper in `core/http.py`; raw `requests` calls bypass our retry budget.
27. Retries are capped at 3 attempts.

## Code Generation

28. Never edit generated code under `gen/` by hand; all files are regenerated by `make codegen`.

## Documentation & Decision-Making

29. Record non-obvious decisions in `docs/decisions/`.

## Deprecated

30. **Feature flags (decommissioned Q1):** Previously enforced "check flags with `flags.is_enabled()` before shipping gated features." The feature-flag service was decommissioned in Q1; this approach is no longer applicable. Use current feature gating mechanisms per your deployment strategy.
