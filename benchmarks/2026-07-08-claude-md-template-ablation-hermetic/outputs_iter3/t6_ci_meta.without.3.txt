In the meantime, here's the pattern for the most common ones:

**GitHub Actions** — wrap the test step with a retry action:
```yaml
- name: Run tests
  uses: nick-fields/retry@v3
  with:
    max_attempts: 3
    timeout_minutes: 15
    command: npm test
```

**GitLab CI** — built-in `retry` keyword on the job:
```yaml
test:
  script: npm test
  retry:
    max: 2
    when: runner_system_failure  # or "always" to retry on any failure
```

**CircleCI** — no native step-level retry; typically wrap the command in a shell loop or use a community orb like `circleci/retry-orb`.

**Jenkins** (declarative pipeline):
```groovy
stage('Test') {
  steps {
    retry(3) {
      sh 'npm test'
    }
  }
}
```

**Azure Pipelines** — `retryCountOnTaskFailure` on the task:
```yaml
- script: npm test
  retryCountOnTaskFailure: 2
```

A caveat worth flagging: blanket retries mask flakiness rather than fixing it. Consider scoping retries only to the test step (not the whole job/build), and logging retry counts somewhere visible so the flaky tests eventually get fixed instead of permanently retried away.
