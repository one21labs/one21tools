Setting all three to `v4` independently means the next version bump requires finding and editing three separate places by hand — and if one gets missed, docs/config/routes silently disagree.

Better: pick one canonical source for the version string and have the other two derive from it:

- **Server route prefix** is usually the natural authority (it's what actually enforces the version at runtime) — define it as a single constant (e.g. `API_VERSION = 'v4'`) that the route-mounting code reads.
- **Client config** should reference that same value — either import it directly if client and server share a package, or read it from a generated/shared config (e.g. an OpenAPI spec, a shared `constants` package, or an env var set from the server's value) rather than a separate literal.
- **Docs** should either be generated from that source (e.g. OpenAPI-generated docs) or, if hand-written, contain a single explicit note like "see `API_VERSION` in `<file>`" so there's no second literal to drift.

If client and server are genuinely separate repos/deploys with no shared package, the fallback is a small shared constants/config file (or env var) that both import, plus a comment marking it as the source of truth.
