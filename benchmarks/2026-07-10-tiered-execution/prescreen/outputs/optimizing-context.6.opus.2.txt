```markdown
# CLAUDE.md

Guidelines for working in this repository. All rules are authoritative.

## Code Style
- Follow PEP 8.
- Use 4-space indentation in Python files.
- Write descriptive variable names.
- Prefer f-strings over `.format()`.
- Format code before committing.

## Structure & Design
- Keep functions short and focused.
- Prefer early returns over deep nesting.
- Prefer composition over inheritance.
- Do not use mutable default arguments.
- Add docstrings to all public functions.

## Error Handling & Logging
- Handle exceptions; do not let them crash silently.
- Error messages should include context about what failed.
- Use logging, not print statements.

## Networking
- All outbound HTTP goes through the wrapper in `core/http.py`; raw `requests`
  calls bypass our retry budget.
- Retries are capped at 3 attempts.

## Security
- Never commit secrets to the repository.

## Testing
- Write unit tests for new code.
- Run tests locally with `poetry run pytest -m "not slow"`. Tests marked `slow`
  need the seeded DB from `make db-seed`.
- Run `make check` before every commit; CI runs the identical target.

## Git & Release Workflow
- Keep pull requests small.
- Branch names use the ticket prefix, e.g. `PLAT-123-short-description`.
- Production deploys happen only from the `main` branch.
- Record non-obvious decisions in `docs/decisions/`.

## Generated Code
- Never edit files under `gen/` by hand; they are regenerated by `make codegen`.

## Database & Migrations
- Migrations run only via `make migrate`; invoking alembic directly skips the
  tenant loop.
- The `events` table schema is at version 9.
- Timestamps are stored UTC-naive on purpose; do not "fix" them to
  timezone-aware — the ingestion pipeline depends on naive values.

## Configuration
- Config keys are `API_URL`, `RETRY_MAX`, `QUEUE_NAME`, and `BATCH_SIZE`
  (defined in `settings.py`).
```
