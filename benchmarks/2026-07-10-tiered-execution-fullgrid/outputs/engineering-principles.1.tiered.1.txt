## Root Cause Analysis

### Proximate Trigger (This Incident)

Priya skipped the manual verification step documented in the team wiki after changing the customer-export SQL query. The verification step ("run the query once by hand and eyeball the row count") would have caught the query bug before the scheduled job ran.

### Systemic Root Cause (All Three Incidents)

**The monitoring layer fails to detect zero-row output.** The job is monitored only for exit code; it exits with code 0 regardless of output row count. This means:

- A job that produces zero rows is indistinguishable from a job that produces correct data.
- No automated alert fires when output is empty.
- The failure is silent until the client detects it.

This monitoring gap is the common structural failure across all three incidents. 

| Incident | Trigger | Exit Code | Alert Fired? | Detection |
|----------|---------|-----------|--------------|-----------|
| #1 (2 months ago) | Schema rename [details TBD] | 0 | No | Client |
| #2 (1 month ago) | Expired credential swallowed by job [details TBD] | 0 | No | Client |
| #3 (this incident) | Query logic error after manual change | 0 | No | Client |

**The pattern:** Three different upstream failures, one shared downstream blindness. The system cannot distinguish success from silent failure because it only watches process exit codes.

## Contributing Factors

1. **Manual verification step lacks system enforcement.** The "run and eyeball row count" step is documented in the wiki but is a human-dependent action with no automated guard or enforcement. New engineers may miss it, and experienced engineers may skip it under time pressure.

2. **Onboarding did not highlight high-risk manual steps.** Priya, as a 6-week-old engineer, was not explicitly walked through or required to follow this critical verification step before making changes to a production export pipeline. The step exists in institutional knowledge, not in process.

3. **Silent error handling in job #2.** The second incident revealed that the job swallows credential errors (expired credentials) and exits 0 anyway. This is a separate but related failure mode: the job does not fail fast on errors it encounters.

4. **No monitoring alert to the team.** The team is not alerted when the job completes. Discovery depends on client-side detection or manual log review.

## Why Prior Incidents' Fixes Did Not Prevent This

Based on the pattern repeating, prior remediation efforts appear to have addressed only the specific trigger of each incident (e.g., "fix the schema reference" in incident #1, possibly "rotate credentials" or "improve credential handling" in incident #2) rather than the shared structural gap.

**Assumption to confirm with the team:** What specific fixes or process changes were implemented after incidents #1 and #2? If they did not include row-count monitoring or a zero-output alert, this explains why the pattern repeated.

  - Exit with a non-zero exit code (e.g., exit 1), or
  - Be detected by monitoring as a failure, or
  - Trigger an alert to the data platform team immediately.

## Blameless Framing

Priya's decision to skip the manual verification step is the immediate trigger for this incident, but it is not the root cause. Priya has been at the company for six weeks; the verification step exists only in undocumented institutional knowledge (a wiki page, possibly not linked from the query code or PR template). Expecting every engineer to discover and execute an undocumented manual step before a production change is a process failure, not an engineer failure.

Fixing Priya's immediate choice would have prevented this one incident. Fixing the monitoring gap would prevent all three incidents' types, including future ones we cannot predict.
