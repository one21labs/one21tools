# Claude Code Context: Atlas Monorepo

## Monorepo Structure
Atlas is a three-service monorepo deployed together:

- **api/** (Go, gRPC): Core service layer.
  - ⚠️ **Everything under api/gen/ is auto-generated by `make proto`.** Do not hand-edit; changes will be silently overwritten.
  - Always edit .proto files in api/proto/ and run `make proto` to regenerate.

- **web/** (React, TypeScript): Frontend.
  - `npm test` runs the full test suite, including Playwright (opens real browsers). Use before committing.
  - `npm run test:unit` runs only unit tests (mocked). Use for rapid feedback during development.

- **pipeline/** (Python, Airflow): Data pipelines.
  - ⚠️ **Requires `AIRFLOW_HOME=pipeline/.airflow` before any DAG interaction.** DAGs will not parse or import without it.
  - Set this before starting work on pipeline code.

## Core Policies

### Conventional Commits
All commits must follow the conventional commits format:
```
<type>(<scope>): <subject>

<body>
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`.

This enables automated changelogs and clear commit history. Enforce this in all PRs.

### Pre-PR Quality Gate
`make check` must pass before opening any PR. This is the canonical check; it runs locally and in CI. If it passes locally, the PR is safe to open.

### Trunk-Based Development
- No long-lived feature branches. Integrate frequently (daily or more).
- All work flows to main; short-lived branches are OK if they merge within 1–2 days.
- This reduces integration friction and supports continuous deployment.

---

## Incident Response

When an engineer asks Claude to help with an incident, invoke the **Incident Response Skill** to walk them through the company-wide 20-step runbook. The runbook is in `docs/runbooks/incident-response.md` and is the same for every team.

Key points:
- Follow the steps sequentially; do not skip.
- Capture severity, timeline, and blast radius before mitigation.
- Ensure post-incident review is scheduled before closing the incident.

---

## Analytics & Metrics for Analysts

Analysts can request live SQL queries against the analytics warehouse. Invoke the **Analytics SQL Skill** to:
1. Clarify the business question (e.g., "What is DAU last 7 days?").
2. Translate it to a SQL query.
3. Execute the query and return results.
4. Interpret results in business context and suggest follow-ups if relevant.

Common queries: product metrics (DAU, retention, funnels), operational metrics (latency, error rates), business metrics (revenue, customer counts).

---

## Environment Setup for Multi-Service Work

When working across services, set up the environment:
```bash
export AIRFLOW_HOME=pipeline/.airflow
make check  # Runs checks for all services
```

## Troubleshooting Quick Reference

| Problem | Likely Cause | Fix |
|---------|--------------|-----|
| `make check` fails in api/ | Generated code out of sync | Run `make proto` in api/ to regenerate. |
| Hand-edits to api/gen/ disappeared | You edited generated code | Edit .proto files only; regenerate with `make proto`. |
| `npm test` fails in web/ with display error | Playwright needs a display | Use `npm run test:unit` instead for faster feedback. |
| DAG import errors in pipeline/ | AIRFLOW_HOME not set | Run `export AIRFLOW_HOME=pipeline/.airflow` before testing DAGs. |
| `make check` fails in pipeline/ | DAG parsing error | Verify AIRFLOW_HOME is set; check DAG syntax and imports. |

---

## When Things Go Wrong

1. **Check the service README** (api/README.md, web/README.md, pipeline/README.md).
2. **Run `make check` locally** to catch the same issues CI will find.
3. **Review recent commits** (conventional commit messages should explain the change).
4. **If it's production, invoke the Incident Response Skill** to walk through mitigation.
