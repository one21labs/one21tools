```markdown
# Release Process Review

## Summary
The team isn't slow because people aren't working — it's slow because the process
amortizes a high-friction release gate by batching, and batching is exactly what
makes each release risky and stressful. Fix the gate's cost, and batch size stops
being forced.

## Root Cause Analysis

**Problem:** Shipping feels slow; releases are stressful.

1. Why? Each release bundles ~40 changes at once.
2. Why? Releases only happen on a fixed biweekly train.
3. Why? The release gate (manual changelog + 3 sequential sign-offs) is expensive,
   so it only makes sense to run it every two weeks, not per-change.
4. Why? Every change — a one-line hotfix or a 40-PR batch — goes through the same
   full-weight gate with no risk-based shortcut.
5. **Root cause:** The process has no cheap path to production. High fixed cost per
   release forces batching; batching creates blast radius and stress.

**Countermeasure:** Lower the cost of releasing (automate the changelog, parallelize
sign-offs, add a lightweight path for low-risk/urgent changes) so smaller, more
frequent releases become the default instead of something the team pays a tax for.

## Waste Identified

| Waste | Where it shows up |
|---|---|
| Inventory | ~40 merged-but-unreleased changes sitting on `main` for up to 2 weeks |
| Waiting | 2-day average PR review; hotfixes waiting for the next train |
| Overprocessing | Release manager hand-copies changelog entries that already exist in PR descriptions (SSoT violation — same fact typed twice) |
| Excess handoffs | 3 *sequential* sign-offs (QA → EM → PM) where dependency isn't obvious |
| Defects (downstream) | Large batches make it harder to isolate which of 40 changes caused a production issue |

## Recommendations, in priority order

1. **Give hotfixes a separate, expedited path.** A one-line urgent fix should not
   ride the same train as a 40-change batch. Define "hotfix" criteria, require one
   accountable approver, and ship independently of the train. This is the single
   biggest fix for "urgent things wait too long."

2. **Automate the changelog.** Generate it from PR titles/descriptions (via a
   required PR template + a script/CI step) instead of hand-copying. The PR
   description is already the source of truth — stop re-typing it.

3. **Shrink the batch size / increase release frequency.** Move from biweekly to
   weekly, or toward continuous release with feature flags decoupling deploy from
   release. Smaller batches mean smaller blast radius, easier root-causing, and
   less stress per release — this is the highest-leverage change and the one that
   makes the other fixes matter.

4. **Parallelize sign-offs.** QA, EM, and PM reviewing the same release note
   sequentially adds pure waiting time with no clear reason each must wait on the
   prior one. Run them concurrently against a shared checklist unless a real
   dependency exists between them.

5. **Make sign-off risk-based, not uniform.** Not every change needs all three
   approvals. Low-risk changes (docs, config, test-only) should get a lightweight
   or automated check; reserve the full 3-way sign-off for changes that actually
   carry risk.

6. **Investigate the 2-day PR review wait.** Is it reviewer availability, PR size,
   or lack of an SLA? Smaller PRs and designated/rotating reviewers usually cut
   this substantially.

## How to know it's working

Track: lead time from PR-open to production, average batch size per release,
fraction of hotfixes that bypass the full pipeline, and release-caused incident
rate. Batch size and lead time should trend down; hotfix bypass usage should be
low but nonzero (proof the fast path exists and is used only when warranted).
```
