A few things worth deciding before writing the code, since GDPR erasure has some sharp edges:

- **Hard delete vs. anonymize**: true row deletion vs. scrubbing PII while keeping a tombstone row (often needed if other tables FK-reference the user, or if you have legal/financial retention obligations that override erasure).
- **Auth**: the endpoint must verify the requester is the account owner (or an authorized admin) — this is a common vector for abuse if done wrong.
- **Cascade**: what happens to orders, posts, logs, etc. owned by the user — cascade delete, nullify, or anonymize.
- **Audit trail**: GDPR erasure typically needs a record that the deletion happened (timestamp, request ID) even though the personal data itself is gone.
