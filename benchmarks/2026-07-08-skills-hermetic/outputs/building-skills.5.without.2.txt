# Generating Release Notes

## Overview

Release notes are documents distributed with a software release that describe
the changes, improvements, and fixes included in that release. They help
users and stakeholders understand what changed.

This skill assumes the repo uses:
- **Semantic versioning (semver)** — version numbers of the form
  `MAJOR.MINOR.PATCH`, where MAJOR = breaking changes, MINOR = new
  backwards-compatible functionality, PATCH = backwards-compatible bug fixes.
- **Git tags** marking release points in history.
- **Conventional commits** — commit messages prefixed with a type
  (`feat`, `fix`, etc.), optionally a scope (`fix(security)`), and optionally
  a `!` to mark a breaking change (`feat!`).

## Procedure

1. Find the last release tag:
   `git describe --tags --abbrev=0`
2. Collect the commits since that tag:
   `git log <tag>..HEAD --oneline`
3. Parse each commit's conventional-commit type prefix, scope, and breaking
   (`!`) marker.
4. Group commits into sections per the **Commit Type Mapping** below.
5. Write the result to `RELEASE_NOTES.md` at the repo root, following the
   **Output Rules** below.

## Commit Type Mapping

| Prefix / pattern | Section | Notes |
|---|---|---|
| any type with `!` (e.g. `feat!`) | Breaking Changes | Always the **first** section |
| `fix(security)` or any security scope | Security | Always listed before Bug Fixes |
| `feat` | Features | |
| `perf` | Performance | |
| `fix` | Bug Fixes | |
| `revert` | Reverts | Also remove the reverted entry if it's in the same release |
| `docs` | *(omit)* | |
| `chore` | *(omit)* | |
| `ci` | *(omit)* | |
| `build` | *(omit)* | |
| `refactor` | *(omit)* | Unless user-facing |

## Output Rules

- Write the notes in markdown to `RELEASE_NOTES.md` at the repo root.
- Every entry links its PR number as `[#123]`.
- Entries are one line, imperative mood, no trailing period.
- Section order: **Breaking Changes → Security → Features → Performance →
  Bug Fixes → Reverts**.
- Omit empty sections.

## Background

Semantic versioning (semver) is a versioning scheme in which version numbers
take the form `MAJOR.MINOR.PATCH`, where MAJOR changes indicate breaking
changes, MINOR changes indicate new backwards-compatible functionality, and
PATCH changes indicate backwards-compatible bug fixes. Git is a distributed
version control system that records changes to files over time; a git tag
marks a specific point in history, typically a release.
