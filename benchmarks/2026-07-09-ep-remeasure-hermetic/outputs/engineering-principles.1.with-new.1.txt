## Contributing Factors (per-incident triggers)

These are the proximate causes of *this* incident specifically — useful for
context, but each is a different immediate trigger with the same downstream
failure, which is the signal that the real gap is downstream of all of them.

- Priya (6 weeks tenure) changed the export SQL and did not perform the
  documented manual verification step.
- The verification step exists only as wiki text: "after changing the query,
  run it once by hand and eyeball the row count." It is not enforced by
  tooling, code review, or CI — it depends entirely on an individual
  remembering an undiscoverable, unenforced convention.
- Prior incidents had unrelated triggers (schema rename; swallowed exception
  on expired credential), confirming the trigger surface is broad and not
  something that can be closed off one cause at a time.

## Root Cause Analysis (5 Whys, branched)

### Branch A — Why did this specific change produce zero rows?

1. Why did the client receive an empty file?
   → The export job ran to completion and produced a file with zero data rows.
2. Why did the job produce zero rows?
   → Priya's SQL query change [presumably filtered out all rows / matched an
   incorrect schema — fill in specifics once confirmed].
3. Why wasn't the defect caught before the job ran against production data?
   → The documented manual verification step (run once by hand, check row
   count) was not performed.
4. Why wasn't the step performed?
   → It exists only as a wiki instruction with no enforcement mechanism —
   nothing in the PR process, deploy pipeline, or job itself requires or
   checks that it happened.
5. Why does the team rely on an unenforced manual step for a check that
   determines whether a client receives valid data?
   → **Root cause A: There is no system-enforced gate between "query
   changed" and "query runs against production and ships to a client."**
   Correctness verification is treated as a personal habit, not a pipeline
   control — so it fails whenever the person who happens to make a change
   doesn't already know the tribal convention (predictably, a 6-week
   employee).

   Per Deming's framing, this is not "Priya made a mistake" — the system
   allowed a brand-new engineer to ship an unverified query straight to a
   client with no guardrail. Any engineer, tenured or not, could have hit
   this same gap.

### Branch B — Why did nobody get alerted, in any of the three incidents?

1. Why did an empty file reach the client without internal alerting?
   → Monitoring only checks the job's process exit code.
2. Why does exit code 0 not indicate a problem?
   → The job's success/failure logic is defined purely by "did the process
   run to completion without throwing," not by "did it produce valid
   business output."
3. Why is there no check on the output itself (row count, file size,
   schema)?
   → No data-quality/output-validation step exists in or after the job —
   monitoring was built around infrastructure health (did the process
   crash?) rather than data correctness (is the output right?).
4. Why did this gap survive two prior incidents with the same symptom?
   → Each prior incident's remediation apparently targeted the specific
   trigger (the schema rename, the credential handling) rather than the
   shared downstream failure mode of "job can exit 0 with zero-row/invalid
   output and nothing checks for that."
5. Why does fixing the trigger, twice, not fix the recurrence?
   → **Root cause B: The team has no independent, output-based
   correctness check for this job. As long as "success" is defined solely
   by process exit code, any future trigger — a new one we haven't
   seen yet — will reproduce the identical failure.** This is the shared
   root cause across all three incidents; Branch A's gap is what let this
   particular trigger occur, but Root cause B is why it reached the client
   undetected all three times.

### Related but distinct issue (Incident 2, worth tracking separately)

The expired-credential incident additionally revealed that the job swallows
exceptions rather than failing loudly. That's a real defect in its own
right (broad/silent exception handling) but is not the shared root cause —
even a job that never swallows an exception would still have shipped an
empty file in incidents 1 and 3, which weren't exception-related. Track it
as a contributing hardening item, not the headline fix.

## Root Cause Summary

1. **(Primary, shared across all 3 incidents)** No output-based validation
   exists for this job — success is defined by exit code only, so any
   defect that produces valid-looking-but-empty/wrong output ships to the
   client undetected, regardless of what triggers it.
2. **(Contributing, this incident)** The one human safeguard that could
   have caught this specific trigger (manual query verification) is
   unenforced tribal knowledge, undiscoverable to new team members and
   unverifiable by anyone else.
