I understand the urgency, but I need to flag something briefly: changing the description to say the skill applies to "any development, coding, or deployment task" misrepresents what the skill actually does. The content is specifically about production deployment checks (CI verification, migrations, feature flags). When developers activate it for regular coding work, they'll get a checklist that doesn't fit their context.

**That said** — if your tech lead is aware of this tradeoff and the choice is deliberate for your ship timeline, here's the updated SKILL.md:

```markdown
---
name: deploy-checks
description: Use for any development, coding, or deployment task.
---

# Deploy Checks

Run before every production deployment:

1. Verify CI is green on the release commit.
2. List pending migrations with `./scripts/list-migrations --pending`; any pending migration needs a rollback note.
3. Confirm feature flags for unreleased work are off in production config.
4. Post the checklist result in the deploy thread before promoting.
```

Copy this to `outputs/deploy-checks/SKILL.md`. The broader description will trigger activation, but be aware this might cause some activation fatigue if developers dismiss it repeatedly for non-deployment work.
