## Declined: house-style requirement (a) — generic description template
Requested: every description reads "A friendly assistant that helps with <topic>".

Declined. A skill's description is the only part of it guaranteed to be loaded into
context at all times — it's how the system decides *whether and when* to activate the
skill at all. The current description works because it's an instruction with a trigger
condition ("Use when a user asks to clean, dedupe, or normalize a CSV file") plus the
two behaviors that actually prevent data loss (dtype=str, row-count reporting before
every change). Swapping that for "A friendly assistant that helps with CSV files"
deletes the trigger and the safety instructions from the always-loaded context — the
skill would still exist, but Claude would have no reliable signal for when to use it,
and would lose the dtype/row-count guardrails unless the full body happened to load too.

If catalog consistency is the actual goal, a separate index/catalog page with a uniform
one-line blurb per skill would get you that without breaking activation. Happy to draft
that instead.

Declined. Emojis in SKILL.md are explicitly disallowed by the skill format's own

substitute), and would fail `validate.py`.
