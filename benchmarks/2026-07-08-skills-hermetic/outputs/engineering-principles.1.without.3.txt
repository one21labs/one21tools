## Root Cause Analysis

### Proximate cause (this incident)
Priya changed the export SQL query and did not perform the documented manual verification step (run once by hand, eyeball row count) before the change reached production.

### Proximate causes (prior two incidents, for pattern comparison)
1. A schema rename broke the query silently (e.g., a `JOIN` or column reference no longer matched, returning zero rows instead of erroring).
2. An expired credential was swallowed by the job's error handling, resulting in an empty result set being treated as valid output rather than a failure.

### Systemic root causes (common to all three)

1. **No data-quality gate independent of process exit code.** The job's contract with monitoring is "exit code 0 = success." Nothing checks *what* was produced. A job that writes zero rows and a job that writes the expected volume are indistinguishable to monitoring.
2. **The only safeguard against bad output is a manual, memory-dependent human step.** "Run it by hand and eyeball the row count" is not enforced by tooling — it lives in a wiki page. It has no CI gate, no PR checklist item, no automated pre-deploy check. It depends entirely on every engineer remembering it, every time, including engineers six weeks into the job. This is not a Priya problem; it's a "any human, eventually" problem.
3. **No sanity-check or anomaly detection on output volume.** Even without a human in the loop, an automated check comparing today's row count to a rolling baseline (e.g., "row count is 0, or >90% below 7-day average") would have caught all three incidents regardless of their differing root causes.
4. **No circuit breaker between job output and client delivery.** Even if internal detection is imperfect, delivery to the client is the point of no return. There is currently no gate that asks "does this output look sane?" before it leaves our systems.
5. **Errors are being swallowed rather than surfacing.** The credential-expiry incident shows the job's error handling actively converts a failure into an apparently-successful empty result. This is a more acute version of problem #1: even a job written to alert-on-error can be undermined by code that masks failure.

### Five whys (this occurrence)

1. Why was an empty file delivered to the client? → The export query returned 0 rows and the job still exited 0.
2. Why did the job exit 0 with 0 rows? → Monitoring/the job only checks exit code, not row count or output size.
3. Why did the query return 0 rows? → It was changed and the change had a defect that produced no matching rows.
4. Why wasn't the defect caught before it reached production? → The documented manual verification step was skipped.
5. Why was the manual step skippable? → It is a wiki instruction, not an enforced gate in tooling, CI, or the deploy path — enforcement depends entirely on individual memory and diligence.

The chain bottoms out at #5, and #5 is identical for all three incidents once you substitute the proximate defect. That is the actual root cause.
