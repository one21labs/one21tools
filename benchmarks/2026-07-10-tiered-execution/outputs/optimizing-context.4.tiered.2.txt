# Claude Code Context: atlas Monorepo

## Org-Wide Conventions

All engineers in this monorepo follow these standards:

### 1. Conventional Commits
All commits use the [Conventional Commits](https://www.conventionalcommits.org/) format:
- **Format:** `type(scope): subject`
- **Examples:** `feat(api): add GetUser RPC`, `fix(web): resolve login race condition`, `refactor(pipeline): improve DAG performance`
- **Scope:** the affected service or component (`api`, `web`, `pipeline`)
- **Benefit:** enables semantic versioning and automated changelogs

### 2. `make check` Must Pass Before Merge
No pull request merges to `main` without `make check` passing locally.
- Run it before pushing; fix any failures before opening a PR.
- All checks must pass; no exceptions.

### 3. Trunk-Based Development, No Long-Lived Branches
- Work directly on `main` or short-lived feature branches.
- Merge frequently to `main` and keep the trunk stable.
- Enables fast feedback and reduces merge conflicts.

## Monorepo Services

This repository contains three independent services. Each has its own CLAUDE.md file (loaded automatically by Claude Code when you enter that directory):

### `/api/` — Go, gRPC Service
Backend APIs and gRPC services. **See `api/CLAUDE.md` for service-specific guidance.**

Key quirk: `api/gen/` is auto-generated by `make proto`; hand-edits get silently overwritten. Never edit generated code directly.

### `/web/` — React, TypeScript Frontend
Web application UI and logic. **See `web/CLAUDE.md` for service-specific guidance.**

Key quirk: `npm test` opens real browsers via Playwright. Use `npm run test:unit` for fast, safe unit testing.

### `/pipeline/` — Python, Airflow Orchestration
Data workflows and DAG orchestration. **See `pipeline/CLAUDE.md` for service-specific guidance.**

Key quirk: Airflow DAGs won't import unless `AIRFLOW_HOME=pipeline/.airflow` is set first. This is a blocker; set it before any DAG work.

## Shared Resources & Mechanisms

### Incident Response Runbook
When an engineer asks Claude to walk them through incident response, load `/runbooks/incident-response.md` — a company-wide 20-step playbook used identically by all teams. Claude steps through the runbook with the engineer on demand.

**Usage:** "Help me work through this incident" → Claude loads the runbook and guides step-by-step.

### Analytics & SQL Access for Metrics
When analysts or product teams ask about product metrics or need live data:
1. Use the documented SQL connection mechanism (configured in `/docs/analytics-sql-access.md`) to query the analytics warehouse.
2. Claude can execute pre-approved queries to answer questions like "how many users signed up today?" or "what's our monthly retention?"

**Usage:** "What were our signups last week?" → Claude references the SQL mechanism and runs the appropriate query.

## General Principles

**Generated Code is Read-Only**
- If a tool generates it (`make proto` → Go code), do not hand-edit it.
- The tool owns that code. Fix issues in the source (e.g., `.proto` files) and regenerate.
- Generated code is silently overwritten on the next tool run; edits will be lost.

**Environment Setup is Critical**
- Some operations require environment variables or specific paths (e.g., `AIRFLOW_HOME` for Airflow DAGs).
- Each service's CLAUDE.md documents its environment prerequisites.
- If a command fails mysteriously, check the service CLAUDE.md for setup instructions.

**Test Commands Have Side Effects**
- `npm test` opens real browsers; `npm run test:unit` does not.
- Some commands are slow or resource-intensive; use the documented safe alternative.
- Always check your service's CLAUDE.md for the right test command.

**Commit Early and Often**
- Trunk-based development means small, frequent commits to `main`.
- Easier to review, easier to revert if needed, safer to deploy.
- Run `make check` before every commit.

## Quick Start for New Engineers

1. Clone the repo.
2. Read this file (root `CLAUDE.md`).
3. Pick a service and read its CLAUDE.md:
   - Working on backend? → Read `/api/CLAUDE.md`
   - Working on frontend? → Read `/web/CLAUDE.md`
   - Working on data pipelines? → Read `/pipeline/CLAUDE.md`
4. Run `make check` to verify your environment is set up.
5. Start developing.

If you hit blockers or questions, check your service's CLAUDE.md first — it documents known quirks and solutions.
