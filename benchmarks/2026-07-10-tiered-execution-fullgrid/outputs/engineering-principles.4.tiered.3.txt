# Config Subsystem Completion Plan

## Status: Core Requirement — DONE

The validated requirement is met: **read `config.json` at startup, validate three fields (`api_url`, `team_id`, `log_level`).** The JSON adapter is finished and handles this completely. No further work is required to satisfy the current user need.

---

## ⚠️ Decision Point: Unrequested Scope

The following was built ahead of confirmed demand:
- YAML, TOML, XML adapter scaffolding
- Hot-reload file watcher (partially wired)
- Schema migration framework (interfaces only)

**No other team has asked for these formats, dynamic reload, or versioning.** Before completing them, pick one:

### Option A (Recommended): Stop Here
Delete or archive unused adapter stubs, half-wired watcher, and migration interfaces. Ship JSON-only. 

**Rationale:** Sunk cost (two weeks already spent) doesn't justify further spend on unrequested features. Unused pluggable architecture is a maintenance burden—three more adapters to security-patch and test forever with zero users.

**Effort to finish:** ~2 days cleanup work (marked in task list below).

### Option B: Finish Only When Requested
Keep the adapter interface documented as an extension point. Don't build YAML/TOML/XML now. Remove watcher and migration code.

**Rationale:** Hedge on the architecture; pay cleanup cost now, but unblock future formats if a second consumer appears.

**Effort to finish:** ~1 day cleanup work.

### Option C: Complete All Originally Scoped
Finish YAML, TOML, XML adapters; wire the watcher fully; implement the migration framework.

**Only pick this if** a concrete near-term reason exists (e.g., a second team is already onboarding and needs YAML). If so, state that reason when committing to this path.

**Effort to finish:** ~8-10 days (detailed breakdown below). **Incompatible with "finish this sprint" unless sprint is longer than one week and you have full capacity.**

## If You Pick C: Completion Plan

**Prerequisite:** State the concrete reason (which team needs which format) before starting.

### Phase 1: YAML Adapter (2–3 days)
- [ ] Add YAML parser dependency (e.g., PyYAML or equivalent; finalize choice based on existing stack)
- [ ] Implement `YamlAdapter.parse()` to read config, map keys to the shared `{api_url, team_id, log_level}` schema
- [ ] Add validation logic (reuse JSON adapter's field validators)
- [ ] Test: valid config, missing required field, type mismatch, file not found
- [ ] Update JSON adapter test suite; run in parallel to ensure parity

### Phase 2: TOML Adapter (2–3 days)
- [ ] Add TOML parser (e.g., `toml` library)
- [ ] Implement `TomlAdapter.parse()` with same schema/validation as YAML
- [ ] Test parity with JSON/YAML
- [ ] Document field mapping (especially handling of nested tables if config grows)

### Phase 3: XML Adapter (2–3 days)
- [ ] Add XML parser (e.g., ElementTree or lxml)
- [ ] Implement `XmlAdapter.parse()` with schema mapping
- [ ] Test: nested elements, attributes vs. text content; validate parity
- [ ] Document XSD or expected structure

### Phase 4: File Watcher (1–2 days)
- [ ] Wire file listener fully; detect changes to active config file
- [ ] Reload config on change; validate new config before applying
- [ ] **Decide:** on reload failure, keep last-good config or crash? (Recommend: keep last-good; log error; emit warning to user.)
- [ ] Add debounce (e.g., 500ms) to avoid thrashing on rapid file writes
- [ ] Test: valid reload, invalid reload (recovery), delete/recreate file

### Phase 5: Migration Framework (1–2 days)
- [ ] **Define versioning:** how does a 3-field config version? (e.g., top-level `version: 1` field)
- [ ] Define what "migration" means before implementing—e.g., adding a 4th field in future versions
- [ ] Implement `Migrator.apply(old_config, old_version, new_version)` with at least one example migration
- [ ] Test: version upgrade, no-op (already current), rollback scenario
- [ ] **Note:** This may not be needed if config never changes. Consider deferring until a second version exists.

### Sequencing & Effort
Run Phases 1–3 in parallel (adapters are independent). Phase 4 after adapters are stable. Phase 5 last (lowest priority).

**Total: 8–10 days.** Not compatible with "finish this sprint" if sprint < 2 weeks and you have concurrent commitments.
