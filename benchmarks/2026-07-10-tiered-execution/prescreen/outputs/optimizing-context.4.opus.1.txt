## `layout.md`

```markdown
# atlas — Claude Code context layout

Principle: root holds only what's true everywhere; every quirk lives next to the
code it constrains, so Claude loads it exactly when it's editing that subtree and
the root file stays short and high-signal.

## Files & mechanisms

| Location | Mechanism | What it holds | Rationale (one line) |
|---|---|---|---|
| `CLAUDE.md` | Root memory (always loaded) | Org conventions + repo map + pointers to skill/MCP | The only three things true in every service; everything else is delegated. |
| `api/CLAUDE.md` | Nested memory (loaded when editing `api/**`) | gRPC/Go notes + the `api/gen/` generated-code warning | Quirk loads only when relevant; keeps the overwrite trap in front of Claude at edit time. |
| `web/CLAUDE.md` | Nested memory (loaded when editing `web/**`) | `test:unit` vs `test` (Playwright) distinction | Stops Claude from spawning real browsers when it just wants unit tests. |
| `pipeline/CLAUDE.md` | Nested memory (loaded when editing `pipeline/**`) | `AIRFLOW_HOME` requirement before any DAG import/test | The one fact without which nothing in pipeline runs, right where DAGs are edited. |
| `.claude/skills/incident-response/SKILL.md` | Skill (model-invoked, progressive disclosure) | The 20-step runbook | Same for every team, triggered by natural language ("we have an incident"), walked step-by-step — exactly a skill, not root context. |
| `.mcp.json` | MCP server (committed, shared) | `warehouse` connector for live SQL | Analysts' live-query need is a tool, not prose; committing it gives every analyst the same read-only access. |
| `.claude/settings.json` | Shared settings (committed) | PreToolUse hook blocking writes to `api/gen/**`; read-only warehouse permission | Turns "hand-edits get overwritten" from a hope into an enforced guard; scopes SQL to read-only. |

## Why each quirk goes where it does (not root)

- **`api/gen/` overwrite** → nested CLAUDE.md *plus* a hard hook. Documentation alone
  is advisory; the hook makes an accidental edit impossible, which matters because the
  overwrite is silent.
- **web Playwright** → nested, not root: only relevant inside `web/`, and putting it in
  root would train Claude to reason about browsers while editing Go.
- **pipeline `AIRFLOW_HOME`** → nested: it's a precondition for a command Claude only
  runs inside `pipeline/`.
- **Runbook** → skill, not CLAUDE.md: it's a *procedure to execute on demand*, used
  company-wide and identically. Loading 20 steps into every session's context is waste;
  a skill loads them only when an incident is actually in play.
- **Warehouse SQL** → MCP, not CLAUDE.md: a capability, not a convention. Prose can't run
  a query; a connector can, and read-only permission keeps it safe.
- **Conventional commits / `make check` / trunk-based** → root: the only rules with no
  service scope. They belong in exactly one place, loaded always.

## Supporting file contents (proposed)

### `api/CLAUDE.md`
```
# api — Go, gRPC

- Everything under `api/gen/` is generated by `make proto`. **Never hand-edit it —
  edits are silently overwritten.** Change the `.proto` sources and run `make proto`.
- Regenerate with `make proto` after any proto change before running `make check`.
```

### `web/CLAUDE.md`
```
# web — React / TypeScript

- `npm run test:unit` runs unit tests. Use this by default.
- `npm test` launches **real browsers via Playwright** (slow, requires a display).
  Only run it when the user explicitly wants end-to-end/browser tests.
```

### `pipeline/CLAUDE.md`
```
# pipeline — Python, Airflow

- Before importing, testing, or listing any DAG you MUST set:
  `export AIRFLOW_HOME="$(pwd)/.airflow"` (from `pipeline/`).
  DAGs will not import otherwise. Do this first, every time.
```

### `.claude/skills/incident-response/SKILL.md` (front-matter)
```
---
name: incident-response
description: >
  Company-wide 20-step incident-response runbook. Use when the user reports or is
  handling a production incident, outage, or page. Walk them through one step at a
  time, confirming completion before advancing.
---
(20 steps follow, one per section, so each is surfaced progressively.)
```

### `.mcp.json` (shape)
```
{ "mcpServers": { "warehouse": { "command": "...", "args": ["--readonly"] } } }
```
Pair with a read-only permission entry in `.claude/settings.json`.
```

## `CLAUDE.md`

```markdown
# atlas

Monorepo with three services. Each has its own `CLAUDE.md` with rules that only
apply inside it — read the one for the service you're editing.

- `api/` — Go, gRPC → `api/CLAUDE.md`
- `web/` — React / TypeScript → `web/CLAUDE.md`
- `pipeline/` — Python, Airflow → `pipeline/CLAUDE.md`

## Conventions (every service, no exceptions)

- **Commits:** Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, …), one
  logical change per commit.
- **Before any PR:** `make check` must pass. Do not open a PR, or tell the user the
  work is ready to merge, until it does.
- **Branching:** Trunk-based. Branch off `main`, keep it short-lived, merge back
  quickly. No long-lived feature branches.

## Capabilities available to you

- **Incidents:** When the user is dealing with a production incident or outage, use
  the `incident-response` skill and walk them through the runbook one step at a time.
- **Product metrics:** For questions about product metrics, run live SQL against the
  analytics warehouse via the connected `warehouse` tool (read-only). Query it rather
  than guessing or estimating.
```

---

Two design calls worth flagging: the runbook is a **skill** rather than a slash command because engineers want to be *walked through* it conversationally (a command dumps text; a skill drives a step-by-step interaction), and the `api/gen/` protection is backed by a **PreToolUse hook**, not just documentation — since the overwrite is silent, an advisory note isn't enough. If you'd prefer the runbook as a `/incident` command or want to skip the hook, say so and I'll adjust the layout.
