# Code Review: `process_refund` Handler

## Defects (ordered by severity)

### 1. Order state mutated before API call succeeds
**Location:** Lines 7–8
**Severity:** Critical
**Why it matters:** If the payments API call (line 11) fails with an exception or returns a non-ok status, the order is already marked as refunded with refunded_amount set. A subsequent inspection of the order or database will show the refund as applied even though the payment processor rejected it. This corrupts order state and breaks accounting reconciliation.
**Fix:** Move the state mutations (lines 7–8) to occur only after a successful API call and status check:
```python
try:
    result = payments_api.refund(order.payment_ref, amount)
except Exception as e:
    log.error("refund error for order %s: %s", order_id, str(e), exc_info=True)
    return False
if result.get("status") != "ok":
    log.warning("refund not ok for order %s", order_id)
    return False
order.status = "refunded"
order.refunded_amount = amount
order.fee = amount * FEE_RATE
return True
```

### 2. result.get() called on potentially None or non-dict object outside try block
**Location:** Line 17
**Severity:** Critical
**Why it matters:** If `payments_api.refund()` returns `None` (or any non-dict object) instead of raising an exception, the bare `.get()` call crashes with an uncaught AttributeError, terminating the calling process without cleanup. This converts an API contract violation into a production outage.
**Fix:** Move the result status check inside the try block:
```python
try:
    result = payments_api.refund(order.payment_ref, amount)
    if result.get("status") != "ok":
        log.warning("refund not ok for order %s", order_id)
        return False
except Exception as e:
    log.error("refund error for order %s: %s", order_id, str(e), exc_info=True)
    return False
```

### 3. Uncaught KeyError on missing order_id
**Location:** Line 7
**Severity:** Critical
**Why it matters:** `orders[order_id]` raises an uncaught KeyError if the order does not exist. This crashes the function without logging which order was requested.
**Fix:** Validate the order exists before attempting to refund:
```python
if order_id not in orders:
    log.error("order not found: %s", order_id)
    return False
order = orders[order_id]
```

### 4. Non-ok refund result treated as success
**Location:** Lines 16–18
**Severity:** High
**Why it matters:** When the API returns `status != "ok"`, the code logs a warning but continues to set the fee and returns `True`, signaling success. This hides a refund failure and leads to silent billing errors.
**Fix:** Return `False` if the result is not ok:
```python
if result.get("status") != "ok":
    log.warning("refund not ok for order %s", order_id)
    return False
```

### 5. Bare except without exception context or rollback
**Location:** Lines 12–14
**Severity:** High
**Why it matters:** The `except Exception` block logs only a static string and swallows the actual error, traceback, and exception type, making it impossible to diagnose API failures in production.
**Fix:** Log the full exception and include the order_id:
```python
except Exception as e:
    log.error("refund error for order %s: %s", order_id, str(e), exc_info=True)
    return False
```

### 6. No validation on refund amount parameter
**Location:** Line 6 (parameter), Line 18 (usage)
**Severity:** High
**Why it matters:** Negative amounts, zero, or amounts exceeding the original order total can all be passed to the API, corrupting order accounting. For example, `amount = -100` would record a negative refund; `amount = 100000` could exceed the original charge and create a negative balance.
**Fix:** Validate amount before processing:
```python
if not isinstance(amount, (int, float)) or amount <= 0 or amount > order.total:
    log.error("invalid refund amount %s for order %s (order total: %s)", amount, order_id, order.total)
    return False
```

### 7. Float arithmetic on FEE_RATE and amount risks precision loss
**Location:** Lines 4, 18
**Severity:** Medium
**Why it matters:** `amount * FEE_RATE` with floating-point values produces rounding artifacts (e.g., `0.1 + 0.2 == 0.30000000000000004`). Over thousands of refunds, cumulative rounding error will cause fee totals to diverge from audited amounts.
**Fix:** Use `Decimal` for monetary calculations:
```python
from decimal import Decimal

FEE_RATE = Decimal("0.02")

# ...
order.fee = Decimal(str(amount)) * FEE_RATE
```

### 8. Logging lacks order_id and API response context
**Location:** Lines 13, 16
**Severity:** Medium
**Why it matters:** The log lines `"refund error"` and `"refund not ok"` do not include the order_id or response details. In high-volume production, these logs are noise and you cannot identify which orders failed.
**Fix:** Include contextual fields in all log statements:
```python
log.error("refund error for order %s: %s", order_id, str(e), exc_info=True)
log.warning("refund not ok for order %s: status=%s", order_id, result.get("status"))
```

### 9. No idempotency guard against duplicate refunds
**Location:** Line 6 (entry point)
**Severity:** Medium
**Why it matters:** Calling `process_refund` twice on the same order will attempt to refund twice, potentially charging the customer twice or leaving refunded_amount and fee doubled.
**Fix:** Check if the order is already refunded:
```python
if order.status == "refunded":
    log.warning("order already refunded: %s", order_id)
    return False
```
