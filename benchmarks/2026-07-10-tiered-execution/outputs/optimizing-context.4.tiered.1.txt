# api/: Go & gRPC Context

This service houses gRPC APIs and Go microservices.

---

## Critical Constraint: Auto-Generated Code

**`api/gen/` is auto-generated by `make proto`.**

- **Do not hand-edit files in `api/gen/`** — changes will be silently overwritten on the next build.
- All gRPC service definitions and message types must be maintained in `.proto` files, not in generated stubs.
- If you encounter merge conflicts in `api/gen/`, regenerate the entire directory with `make proto` instead of resolving conflicts manually.

---

## Workflow: Updating Proto Definitions

1. Edit `.proto` files in `api/`
2. Run `make proto` to regenerate stubs in `api/gen/`
3. Implement changes against the new generated interfaces
4. Commit both proto source and `api/gen/` changes in a single commit

**Example commit message**:  
```
feat(api): add GetUserMetrics rpc to User service

- Add GetUserMetrics rpc definition to api/proto/user.proto
- Regenerate stubs with `make proto`
```

---

## Before Opening a PR

- Run `make check` (required by atlas org-wide convention).
- Verify all `.proto` changes have been regenerated via `make proto`.
- Do not commit hand-edited `api/gen/` files.
```

## Critical Distinction: Test Commands

**`npm test` launches real browsers via Playwright — expensive and slow.**  
**`npm run test:unit` runs Jest in-process — fast, safe for iteration.**

### Test Workflow

1. **Write/edit code**
2. **Run `npm run test:unit`** for rapid feedback (jest, no browsers)
3. **Before committing or opening a PR**, run `npm test` if time allows, or rely on CI to verify browser tests
4. **If browser tests fail in CI**, reproduce locally with `npm test` to debug

### Performance Note

A single `npm test` run can take 2–5 minutes depending on test suite size.  
Use `npm run test:unit` for tight iteration loops; save `npm test` for pre-commit validation or CI.

---

## Before Opening a PR

- Run `make check` (required by atlas org-wide convention).
- Run `npm run test:unit` locally to validate.
- Run `npm test` before committing if time allows; otherwise rely on CI.

# pipeline/: Python & Airflow Context

This service houses Apache Airflow data pipelines.

---

## Critical Requirement: AIRFLOW_HOME

**DAGs will not import unless `AIRFLOW_HOME=pipeline/.airflow` is set.**

This is a silent-failure trap. If DAGs appear missing or imports fail, this environment variable is likely unset.

### Setup

Before any Airflow CLI operations or DAG testing, always run:

```bash
export AIRFLOW_HOME=pipeline/.airflow
```

Verify it's set:
```bash
echo $AIRFLOW_HOME  # should print: pipeline/.airflow
```

### Common Workflow

1. Set the environment variable (add to your shell profile or run before each session)
2. List DAGs: `airflow dags list`
3. Test a DAG: `airflow dags test <dag_id> <date>`
4. Dry-run validation: `python -m airflow dags validate` (checks for import and parse errors)

---

## Troubleshooting

- **"No DAGs found"** → Check if `AIRFLOW_HOME` is set correctly.
- **"ModuleNotFoundError"** → First thing to check: `echo $AIRFLOW_HOME`.
- **Merge conflicts in DAGs** → Usually caused by stale imports; re-run `export AIRFLOW_HOME=pipeline/.airflow` and test again.

---

## Before Opening a PR

- Run `make check` (required by atlas org-wide convention).
- Verify `AIRFLOW_HOME=pipeline/.airflow` is set.
- Run `python -m airflow dags validate` to ensure no import errors.
```
