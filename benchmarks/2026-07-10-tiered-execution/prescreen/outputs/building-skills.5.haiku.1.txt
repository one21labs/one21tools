# Release Notes Generator

Automatically generate formatted release notes from git commits using conventional commit conventions.

## Overview

This skill generates release notes by analyzing git commits since the last release tag and organizing them into categorized sections with proper formatting. It handles conventional commit types, semantic versioning, and creates a professional markdown output.

## Background

Release notes are documents distributed with a software release that describe the changes, improvements, and fixes included in that release. They help users and stakeholders understand what changed. Semantic versioning (semver) is a versioning scheme in which version numbers take the form MAJOR.MINOR.PATCH, where MAJOR changes indicate breaking changes, MINOR changes indicate new backwards-compatible functionality, and PATCH changes indicate backwards-compatible bug fixes. Git is a distributed version control system that records changes to files over time; a git tag marks a specific point in history, typically a release.

## History of This Process

We used to write release notes by hand in Confluence, which took hours every sprint. In Q3 2025 we moved to a shell script that dumped raw git log output, but nobody could read it. In January 2026 Priya rewrote it around conventional commits, and in March we retired the old script after the mobile team migrated. This skill replaces all of that.

## How To Generate Release Notes

When someone wants release notes, follow these steps:

1. Look at the commits since the last release tag by running:
   ```
   git describe --tags --abbrev=0
   ```

2. Collect the commits in that range with:
   ```
   git log <tag>..HEAD --oneline
   ```

3. The commits are then grouped by their conventional-commit type prefix, and each group becomes a section.

## Commit Type Mapping

The following commit types are recognized:

| Type | Section | Notes |
|------|---------|-------|
| `feat` | Features | New backwards-compatible functionality |
| `fix` | Bug Fixes | Backwards-compatible bug fixes |
| `fix(security)` or security scope | Security | Always listed before Bug Fixes |
| `perf` | Performance | Performance improvements |
| `docs` | Omit | Documentation only changes |
| `chore` | Omit | Build process, dependencies, etc. |
| `ci` | Omit | CI configuration changes |
| `build` | Omit | Build system changes |
| `refactor` | Omit | Code refactoring (unless user-facing) |
| Any type with `!` (e.g., `feat!`) | Breaking Changes | Always the FIRST section |
| `revert` | Reverts | Remove the reverted entry if in the same release |

## Output Rules

The release notes are written in markdown to `RELEASE_NOTES.md` at the repo root with the following rules:

- **Format**: Markdown with each entry as one line
- **PR Links**: Every entry links its PR number as `[#123]`
- **Style**: Imperative mood, no trailing period
- **Section Order** (omit empty sections):
  1. Breaking Changes (always first if present)
  2. Security
  3. Features
  4. Performance
  5. Bug Fixes
  6. Reverts

## Usage

Call this skill when you need to generate release notes for a new version. It will:
- Automatically detect the last release tag
- Parse all commits since that tag
- Categorize them by conventional commit type
- Generate a professionally formatted markdown file
- Handle special cases like breaking changes, security fixes, and reverts
