## layout.md

```markdown
# Claude Code context layout — atlas

| Location | Mechanism | Contents | Rationale |
|---|---|---|---|
| `CLAUDE.md` (root) | Auto-loaded memory | Org-wide rules (conventional commits, `make check`, trunk-based dev), repo map, pointers to service docs, runbook/warehouse pointers | Loaded on every session — keep it small and stable so it doesn't crowd context with things only one service needs |
| `api/CLAUDE.md` | Auto-loaded nested memory | Go/gRPC conventions, build/test commands, warning that `api/gen/` is generated | Only pulled in when Claude is actually working under `api/`; keeps Go specifics out of root |
| `api/gen/CLAUDE.md` | Auto-loaded nested memory (tiny, ~2 lines) | "Generated by `make proto` — hand-edits are silently overwritten, run `make proto` instead" | If context gets summarized or Claude cd's straight into `gen/`, this repeats the warning right where the damage would happen |
| `web/CLAUDE.md` | Auto-loaded nested memory | React/TS conventions, explicit note that `npm test` launches real Playwright browsers vs `npm run test:unit` for fast checks | Stops Claude from defaulting to `npm test` for a quick sanity check and popping browser windows |
| `pipeline/CLAUDE.md` | Auto-loaded nested memory | Python/Airflow conventions, requirement to `export AIRFLOW_HOME=pipeline/.airflow` before anything DAG-related | DAG import failures are otherwise a confusing red herring; scoped to where it matters |
| `.claude/commands/incident-response.md` | Slash command | Full 20-step incident runbook | It's one fixed procedure used identically company-wide — invoke on demand (`/incident-response`) instead of loading it into every session's context |
| `.claude/settings.json` | MCP server config + permissions | Read-only `warehouse` MCP connector (scoped credentials), plus an allow-list entry for `make check` | Gives Claude a real SQL tool for live queries instead of encoding query logic in markdown; keeps credentials out of version-controlled docs |
| `docs/analytics-warehouse.md` | Plain doc, referenced (not auto-loaded) from root `CLAUDE.md` | Table/schema notes, which tables are canonical per product metric, query conventions | Detailed enough to matter for analyst questions, but not universal enough to burn context on every engineer's session |

General principle: nested `CLAUDE.md` files carry anything a Claude session needs *only* while working in that subtree (so it loads automatically and scoped); the root file stays a thin index of rules + pointers; anything invoked occasionally (runbook, warehouse) becomes a command or tool rather than always-loaded text.
```

## CLAUDE.md

```markdown
# atlas

Monorepo for the Atlas platform.

- `api/` — Go, gRPC services. See `api/CLAUDE.md`.
- `web/` — React/TypeScript frontend. See `web/CLAUDE.md`.
- `pipeline/` — Python, Airflow DAGs. See `pipeline/CLAUDE.md`.

Claude Code loads each service's `CLAUDE.md` automatically when working in that directory — don't duplicate service-specific detail here.

## Rules that apply everywhere
- Use conventional commits for every commit message.
- `make check` must pass before opening or updating a PR.
- Trunk-based development: no long-lived branches. Branch, land, delete quickly.

## Incident response
Run `/incident-response` to walk through the standard 20-step runbook. It is identical across every team in the company — follow it as written, don't improvise a different order.

## Analytics warehouse
A read-only `warehouse` MCP connector is configured for this repo. When asked about product metrics, query it directly instead of guessing from code or stale docs. See `docs/analytics-warehouse.md` for schema notes and which tables are canonical. Never attempt writes/DDL through this connector — it's read-only by design.
```

Two follow-ups worth doing when you have file access: write the actual 20-step runbook into `.claude/commands/incident-response.md`, and set up the `warehouse` MCP connector's read-only credentials in `.claude/settings.json`.
