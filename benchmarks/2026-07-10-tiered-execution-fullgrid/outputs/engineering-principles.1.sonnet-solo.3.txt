# Post-Incident Analysis: Nightly Customer-Export Job Delivering Empty Files

**Status:** Draft for team review
**Incident date:** 2026-07-09 (this occurrence); two prior occurrences in the preceding ~2 months
**Severity:** High — client-visible data delivery failure, third recurrence
**Author:** Data Platform Team (draft)

## Summary

The nightly customer-export job produced an empty output file that was delivered to a client without any internal alert firing. This is the third occurrence of this exact failure mode in two months. Each occurrence had a different immediate trigger, but all three shared the same underlying failure: the job completed with exit code 0 despite writing zero rows, and monitoring only checks exit code, so nothing detected the empty file before the client did.

This time, the immediate trigger was a query change made by Priya (joined ~6 weeks ago) that was not run through the documented manual verification step ("after changing the query, run it once by hand and eyeball the row count").

## Impact

- A client received an empty export file instead of their expected data (third time in two months).
- Client trust/confidence in data delivery reliability is degraded by a repeat failure.
- Engineering time spent on incident response and remediation, again.
- No data was corrupted or lost — the failure is one of non-delivery/empty delivery, not incorrect data.

## Timeline of This Occurrence

*(Fill in exact timestamps from logs/paging system.)*

1. Priya modifies the export SQL query as part of a change.
2. The documented manual verification step (run once by hand, eyeball row count) is not performed.
3. Nightly job runs on schedule, executes the modified query, writes zero rows to the output file.
4. Job exits 0 (success) — no rows written, but the process itself completed without error.
5. Monitoring checks exit code only; exit code 0 is treated as a healthy run. No alert fires.
6. Empty file is delivered to the client via the normal delivery pipeline.
7. Client notices the file is empty and reports it — client is the first to detect the failure, again.

## Root Cause Analysis (Five Whys)

1. **Why was an empty file delivered to the client?**
   The export job wrote zero rows but still completed successfully and delivered the output.

2. **Why did zero rows get written?**
   The SQL query was changed and the new version returned no rows under production conditions (this time). In prior incidents, the zero-row condition was caused by a schema rename and by a swallowed expired-credential error, respectively.

3. **Why wasn't the zero-row output caught before delivery?**
   No automated check validates row count or output size; monitoring only inspects the job's process exit code, and the job's exit code does not reflect whether meaningful output was produced.

4. **Why didn't the manual verification step catch it this time?**
   The step is documented only in the team wiki and depends on an individual engineer remembering and choosing to perform it. It is not enforced by tooling, code review, CI, or the deployment process. A newer team member unfamiliar with the tribal-knowledge process missed it.

5. **Why does the team rely on a manual, undocumented-in-code step for a recurring, high-consequence check?**
   There is no technical control that makes "verify the export actually produces rows" a mandatory, automated gate. The team has treated this as a process/training problem rather than a systems problem, despite two prior incidents that should have surfaced the same underlying gap.

**Root cause:** The system has no automated safeguard that ties job success to *meaningful output* (non-empty, plausible row count). Success is currently defined purely as "process exited 0." This makes the pipeline structurally dependent on every upstream cause of zero-row output never happening and/or on a human remembering a manual, wiki-documented step every time. Three different upstream triggers (schema rename, swallowed expired credential, unreviewed query change) have all exploited the same structural gap.

## Contributing Factors

**Immediate/proximate trigger (specific to this occurrence):**
- Query change made without running the documented manual verification step.
- No enforcement mechanism (code review checklist, CI check, pairing requirement) ensures the manual step actually happens, especially for newer team members who may not know the wiki page exists.

**Systemic factors (present in all three occurrences):**
- Monitoring only checks process exit code, not job output/semantic success.
- The job's success/failure signal is decoupled from whether it produced correct, non-empty output.
- At least one prior root cause (expired credential) was actively swallowed by the job rather than surfaced as an error — indicating the job's own error handling suppresses failures instead of propagating them.
- No automated row-count or output-size floor/threshold check exists anywhere in the pipeline.
- No alerting exists for "export delivered zero rows," despite this being the third time it has happened.
- Client is consistently the first to detect the failure — there is no internal detection layer at all for this failure mode.
- Reliance on a wiki-documented manual step as the *sole* safeguard against a recurring, high-impact failure mode, with no fallback control.

## Why This Keeps Happening

The three incidents had unrelated proximate causes (schema rename, expired credential, unreviewed query change), which is why each individual "cause" was treated as a one-off and fixed narrowly (or not fixed at all) rather than recognized as symptoms of the same missing control. As long as job success is defined as "exited 0," any new and previously-unseen upstream failure mode can and will produce the same outcome: empty file, exit 0, no alert, client discovers it. Fixing only the proximate cause of this incident (e.g., reminding Priya, or adding a review step) would not prevent a fourth, differently-triggered occurrence.

## Corrective Actions

### Must do (addresses the systemic gap directly)

1. **Add an automated output-validation gate to the export job itself.**
   The job should fail (non-zero exit and/or explicit alert) if it produces zero rows, or if row count falls below a sane floor/expected range. This directly breaks the pattern shared by all three incidents. Owner: TBD. Target: next sprint.

2. **Decouple "success" from "process exited without exception."**
   The job should explicitly check that it produced non-empty, plausible output before reporting success and before triggering delivery to the client. Delivery should be gated on this check, not just on the query having run.

3. **Fix the swallowed-credential error handling.**
   Any place in the job that catches and suppresses errors (as happened with the expired credential in a prior incident) should be audited; failures must propagate as failures, not be silently absorbed into a "successful" empty run.

4. **Add monitoring/alerting on row count / output size, not just exit code.**
   Monitoring should treat "zero rows written" (or anomalously low row count) as an alertable condition independent of exit code.

### Should do (reduces likelihood of the proximate trigger recurring)

5. **Turn the manual verification step into an enforced control, not a wiki suggestion.**
   Options: require it as a checklist item in PR review for any change to export queries, add a pre-merge CI check that runs the query against a representative dataset and asserts non-zero/expected row count, or require pairing/second-approval on export query changes. Prefer automation (CI check) over process/memory-based controls, since this incident shows memory-based controls fail, especially for newer team members.

6. **Onboarding gap:** Ensure new engineers (e.g., Priya) are walked through critical operational runbooks for the systems they'll touch, not just pointed at a wiki. Consider a lightweight "you're now touching X — read these Y docs" checklist for onboarding, though this should not be relied upon as the primary fix (see #5).

### Should investigate

7. **Audit other jobs for the same anti-pattern.** Determine whether other data-delivery jobs on the team have the same "exit 0 even on empty/degenerate output" property and apply the same fix proactively rather than waiting for a fourth incident elsewhere.

8. **Add a client-facing safety net if feasible** (e.g., delivery pipeline itself refuses to deliver a zero-byte/empty file and alerts instead), as defense-in-depth in case the job-level check is ever bypassed or has its own bug.

## Lessons Learned

- A manual, documented-only process step is not a control — it's a hope. Three incidents with three different triggers converged on the same failure because the actual safety net (automated output validation) never existed.
- "Exit code 0" is not a sufficient definition of job success for a data-delivery pipeline. Success should be defined in terms of the job's actual purpose (deliver correct, non-empty data), not merely "the process didn't crash."
- Repeat incidents with different proximate causes but identical outcomes are a strong signal to fix the shared structural gap rather than the latest trigger in isolation.
- The client should never be the first to detect a delivery failure; that indicates a missing detection layer, not just a missing fix for this specific bug.
