# Generating Release Notes

Generate release notes for a release by reading the git commits since the last
release tag, grouping them by their Conventional Commit type, and writing a
grouped markdown document to `RELEASE_NOTES.md` at the repo root.

## Workflow

1. **Find the last release tag:**
   ```bash
   git describe --tags --abbrev=0
   ```

2. **Collect the commits in range** (from that tag to `HEAD`):
   ```bash
   git log <tag>..HEAD --oneline
   ```

3. **Parse each commit's Conventional Commit prefix** (the `type(scope): subject`
   at the start of the message) and map it to a section using the table below.

4. **Group commits into sections** and drop any commit whose type maps to *omit*.

5. **Write the notes as markdown** to `RELEASE_NOTES.md` at the repo root,
   following the Output Rules.

## Commit type mapping

| Commit prefix                       | Section                                              |
| ----------------------------------- | ---------------------------------------------------- |
| `feat`                              | Features                                             |
| `fix`                               | Bug Fixes                                            |
| `perf`                              | Performance                                          |
| `revert`                            | Reverts (also remove the reverted entry if it is in the same release) |
| any type with `!` (e.g. `feat!`)    | Breaking Changes — **always the first section**      |
| `fix(security)` / any security scope| Security — **always listed before Bug Fixes**        |
| `refactor`                          | omit *unless user-facing*                            |
| `docs`, `chore`, `ci`, `build`      | omit                                                 |

## Output rules

- Write markdown to **`RELEASE_NOTES.md`** at the repo root.
- Every entry links its PR number as `[#123]`.
- Entries are **one line, imperative mood, no trailing period**.
- **Section order** (omit any empty section):
  1. Breaking Changes
  2. Security
  3. Features
  4. Performance
  5. Bug Fixes
  6. Reverts

### Background

Release notes are documents distributed with a software release that describe
the changes, improvements, and fixes included in that release. They help users
and stakeholders understand what changed. Semantic versioning (semver) is a
versioning scheme in which version numbers take the form MAJOR.MINOR.PATCH,
where MAJOR changes indicate breaking changes, MINOR changes indicate new
backwards-compatible functionality, and PATCH changes indicate
backwards-compatible bug fixes. Git is a distributed version control system that
records changes to files over time; a git tag marks a specific point in history,
typically a release.

### History of This Process

We used to write release notes by hand in Confluence, which took hours every
sprint. In Q3 2025 we moved to a shell script that dumped raw git log output,
but nobody could read it. In January 2026 Priya rewrote it around conventional
commits, and in March we retired the old script after the mobile team migrated.
This skill replaces all of that.

### How To Generate

When someone wants release notes, what you would generally want to do is first
look at the commits since the last release tag, which you would typically find
by running `git describe --tags --abbrev=0`, and then you would want to collect
the commits in that range with `git log <tag>..HEAD --oneline`. The commits
should then be grouped by their conventional-commit type prefix, and each group
becomes a section.

### Commit Type Mapping

```
feat -> Features
fix -> Bug Fixes
perf -> Performance
docs -> omit
chore -> omit
ci -> omit
build -> omit
refactor -> omit unless user-facing
revert -> Reverts; also remove the reverted entry if it is in the same release
any type with ! (e.g. feat!) -> Breaking Changes, always the FIRST section
fix(security) or any security scope -> Security, always listed before Bug Fixes
```
