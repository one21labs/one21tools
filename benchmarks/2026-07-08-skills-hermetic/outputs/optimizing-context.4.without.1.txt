## Design rationale (brief)

Claude Code has several context mechanisms, and each quirk in your list maps to a different one:
- **Nested CLAUDE.md** (auto-loaded based on which directory you're working in) for the three services' local quirks.
- **A custom slash command** for the incident runbook, since it's a fixed, identical procedure — not something to re-explain in prose every time.
- **A dedicated subagent + MCP server** for warehouse SQL, so "run live SQL" is scoped to read-only access instead of living in the main loop's general permissions.
- **`.claude/settings.json`** to make the `api/gen/` "don't hand-edit" rule an enforced permission, not just a comment Claude might ignore under pressure.

```markdown
# Atlas — Claude Code Context Layout

| File / Mechanism | Location | Rationale |
|---|---|---|
| Root `CLAUDE.md` | `/CLAUDE.md` | Org-wide rules (commits, `make check`, trunk-based dev) that apply no matter which service you're touching; always loaded. |
| `api/CLAUDE.md` | `/api/CLAUDE.md` | Go/gRPC conventions + the `api/gen/` generated-code warning; auto-loaded only when working under `api/`. |
| `web/CLAUDE.md` | `/web/CLAUDE.md` | React/TS conventions + the Playwright-vs-unit-test distinction; auto-loaded only under `web/`. |
| `pipeline/CLAUDE.md` | `/pipeline/CLAUDE.md` | Python/Airflow conventions + the `AIRFLOW_HOME` gotcha; auto-loaded only under `pipeline/`. |
| `.claude/settings.json` (committed) | `/.claude/settings.json` | Shared, enforced permissions — e.g. deny Edit/Write on `api/gen/**` — so the "don't hand-edit generated code" rule is a hard stop, not just advice Claude can be talked out of. |
| `.claude/settings.local.json` (gitignored) | per engineer | Personal overrides (extra allowed commands, etc.) that shouldn't be forced on the whole team. |
| `.claude/commands/incident-response.md` | `/.claude/commands/incident-response.md` | One canonical, version-controlled copy of the 20-step runbook, invoked as `/incident-response`; guarantees every team gets the identical procedure instead of a paraphrase. |
| `.claude/agents/analytics-sql.md` | `/.claude/agents/analytics-sql.md` | A subagent scoped to the warehouse MCP tool with read-only instructions, so "run SQL against the warehouse" requests don't hand the main session unrestricted DB access. |
| `.mcp.json` | `/.mcp.json` | Registers the analytics-warehouse MCP server (read-only credentials) so the SQL tool exists for the `analytics-sql` subagent to call. |
| Optional `PreToolUse` hook on `git commit` | inside `/.claude/settings.json` | Local nicety that runs `make check` before Claude attempts a commit, catching failures before they even reach CI — not a substitute for CI enforcement. |

Notes:
- Nested `CLAUDE.md` files are additive to the root one, not a replacement — keep root generic, keep service files terse and quirk-specific.
- Generated code under `api/gen/` should remain *readable* (Claude often needs to see it), just not *writable* — hence a permission deny rule, not an ignore/exclude rule.
- The runbook command lives in the repo (not a personal `~/.claude` command) specifically because it must be identical for every team, and repo-tracked means everyone who clones `atlas` gets the same version.
```

### `CLAUDE.md` (root)

```markdown
# Atlas Monorepo

Atlas contains three services:
- `api/` — Go, gRPC
- `web/` — React, TypeScript
- `pipeline/` — Python, Airflow

Each has its own `CLAUDE.md` with local quirks; Claude Code loads it automatically when you're working in that directory. This file covers rules that apply everywhere.

## Conventions (all services)
- Commits follow Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`, ...).
- `make check` must pass before any PR is opened. Run it from the repo root — it covers lint + tests for all three services.
- Trunk-based development: no long-lived branches. Branch, commit in small increments, open the PR quickly, merge, delete the branch.

## Generated code
`api/gen/` is generated by `make proto` — never hand-edit it, edits are silently overwritten on the next generation. See `api/CLAUDE.md` for details.

## Incident response
Run `/incident-response` to walk through the company's standard 20-step incident runbook. It's identical across every team — use it as-is rather than improvising a variant.

## Analytics / product metrics
For questions that need live SQL against the analytics warehouse, delegate to the `analytics-sql` subagent (read-only access). Don't run warehouse queries or writes directly from this session.
```
