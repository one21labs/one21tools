# docs-plan.md — Documentation Restructuring Decisions

## 1. Local Setup Steps

**Fact:** Setup instructions are duplicated in README.md and CONTRIBUTING.md, with substantive conflicts:
- README: `clone, npm install, copy .env.example, npm run dev`
- CONTRIBUTING: `clone, npm ci, create .env by hand, npm start`
- Conflicts on: package manager command (npm install vs npm ci), env file handling (copy vs create), run command (npm run dev vs npm start)

**Canonical location:** README.md

**Disposition of other copies:** CONTRIBUTING.md will be updated to remove duplicate setup steps and replace with a cross-reference to README.md setup section.

**Conflict flagged for human:** The two versions conflict on three operative choices:
1. Package manager: `npm install` vs `npm ci` — these have different cache/lock behaviors; confirm which is intended for local development.
2. Environment file: `copy .env.example` vs `create .env by hand` — confirm whether `.env.example` exists and should be copied, or whether `.env` must be created manually.
3. Run command: `npm run dev` vs `npm start` — confirm which is the actual development server entry point.

**Recommendation for resolver:** Verify the actual state of the repo, confirm the correct commands, and update README.md with the verified steps before redirecting CONTRIBUTING.md.

---

## 2. API Timeout Value

**Fact:** API timeout is stated in prose in two places with conflicting values:
- README.md: "API requests time out after 30 seconds"
- CONTRIBUTING.md: "API requests time out after 45 seconds"
- Actual source of truth: constant defined in `src/http/client.js`

**Canonical location:** `src/http/client.js` (the code constant is the single source of truth)

**Disposition of other copies:**
- README.md: Replace the hardcoded "30 seconds" statement with a reference to the source: "See `src/http/client.js` for the current API request timeout value."
- CONTRIBUTING.md: Remove the hardcoded "45 seconds" statement entirely or replace with the same reference to `src/http/client.js`.

**Rationale:** Numeric values in prose drift over time when the code changes. Referencing the code location prevents future contradictions.

---

## 3. Support Email

**Fact:** A support email is currently mentioned in README.md.

**Canonical location:** README.md (reasonable primary location for user-facing support contact)

**Disposition of other copies:** None identified as duplicate. No change required if this is the only location; keep it.

**Note:** If support contact information should also appear elsewhere (e.g., a SUPPORT.md or CODE_OF_CONDUCT.md), those secondary locations should reference or cross-link to README.md rather than duplicate the email.

---

## 4. `recalcTotals()` Billing Rule

**Fact:** A rule in draft CLAUDE.md states: "in src/billing.js, always call recalcTotals() after any mutation to line items" (this rule is scoped to src/billing.js only).

**Canonical location:** A comment in `src/billing.js` itself, placed near the line-item mutation code or in a module docstring

**Disposition of other copies:** Remove this rule from the draft CLAUDE.md. CLAUDE.md should contain repo-wide operative instructions; rules narrowly scoped to a single file belong closer to the code they govern (as a code comment or docstring) for maintainability.

**Rationale:** Developers working on billing.js will see the rule where they need it. Storing file-specific rules in a distant CLAUDE.md creates maintenance risk and decreases discoverability.

---

## 5. "Lessons Learned" Narrative Section

**Fact:** Draft CLAUDE.md contains a "Lessons learned" section with historical narratives:
- "2025-11: we originally used npm start, switched to npm run dev after the port clash"
- "2026-01: renamed the export module..."

**Canonical location:** None — this content should not be retained in CLAUDE.md

**Disposition of other copies:** Delete from draft CLAUDE.md entirely.

**Rationale:** 
- CLAUDE.md is for operative rules and current instructions, not historical narrative.
- The history of why a decision was made (port clash, module rename) is already captured in git commit messages and pull request descriptions, which are the appropriate authoritative source for historical context.
- The substantive outcome of the npm start → npm run dev decision is addressed in item 1 (setup steps resolution).
- Keeping outdated narratives in active documentation creates confusion about what is current vs. historical.
