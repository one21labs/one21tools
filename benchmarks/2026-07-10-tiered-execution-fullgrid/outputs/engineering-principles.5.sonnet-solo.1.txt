**User Data Export — Implementation Tasks**

1. **Auth/authorization guard** — endpoint must verify the requester is exporting their *own* data (session/token check), not an arbitrary user ID.
2. **Data inventory** — enumerate every table/store containing user data (profile, activity, uploads, linked accounts, etc.) so the export is actually complete.
3. **Export format** — pick JSON (fastest to implement, no extra library overhead) as the "whatever is easiest" format.
4. **Query/aggregation layer** — write the data-gathering logic that pulls from each source identified in (2).
5. **Serialization** — assemble gathered data into the JSON response payload.
6. **Endpoint wiring** — route, controller/handler, request validation.
7. **Rate limiting** — throttle export requests per user to prevent abuse/load spikes (data export queries are typically expensive).
8. **Audit logging** — log who exported data and when (standard for personal-data-access endpoints).
9. **Error handling** — partial-failure behavior if one data source is unavailable.
10. **Tests** — unit tests per data source, integration test for full export, auth-bypass test (user A cannot export user B's data).
11. **Manual QA pass** — verify export completeness against a seeded test account.
12. **Ship** — deploy behind existing release process, confirm target date.

Note: items 1, 7, 8, and 10's auth-bypass case aren't optional add-ons — they're baseline requirements for any endpoint that returns a full dump of personal data, so I've folded them into the breakdown rather than treating them as separate scope questions.
