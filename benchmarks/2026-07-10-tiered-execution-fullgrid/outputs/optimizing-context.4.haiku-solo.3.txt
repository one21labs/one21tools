```markdown
Each encapsulates quirks that would break workflows in other services; prevents cross-service interference.

- **api/CLAUDE.md** — gRPC workflow, `make proto` guard (warn on api/gen/ edits), Go toolchain notes
- **web/CLAUDE.md** — React/TS patterns, Playwright vs `npm run test:unit` distinction, browser test tradeoffs
- **pipeline/CLAUDE.md** — Airflow DAG patterns, AIRFLOW_HOME=pipeline/.airflow requirement, DAG import mechanics

## Shared Reference Files (`.claude/`)
- **`runbooks/incident-response.md`** — 20-step org runbook; linked from root CLAUDE.md so Claude proactively offers to walk through it
- **`analytics/warehouse-query-template.md`** — SQL patterns and connection notes for analysts; linked from root CLAUDE.md

## Rationale
- **Root = org conventions only** — narrow and stable; prevents root CLAUDE.md from becoming a grab-bag that overshadows service-specific context
- **Service CLAUDE.md = generated-code warnings + workflow quirks** — ensures Claude warns before editing api/gen/, knows to run `npm run test:unit` not `npm test`, etc.
- **Shared runbooks/analytics** — centralized, org-wide, easy to link from root; not duplicated per service
```

## CLAUDE.md

```markdown
# Atlas Monorepo — Claude Code Context

## What This Codebase Is
A three-service monorepo: **api/** (Go gRPC server), **web/** (React/TypeScript frontend), **pipeline/** (Python Airflow DAGs).

All services follow the org's core conventions; each has service-specific quirks documented in its own CLAUDE.md.

## Core Conventions (All Services)

### Commit Style
Conventional commits only (`feat:`, `fix:`, `refactor:`, etc.). No "wip" or "todo" commits to trunk.

### Pre-PR Gate
`make check` must pass locally before pushing any PR. This runs linting, type checks, and quick unit tests across affected services.

### Branching Model
Trunk-based development. Feature branches exist only for active work; merge to main within 2–3 days max. No long-lived branches.

## Service-Specific Workflows

**Read the service's own CLAUDE.md before working in that directory:**
- **api/** → `api/CLAUDE.md` — gRPC and generated code rules
- **web/** → `web/CLAUDE.md` — React/TS and Playwright test distinction
- **pipeline/** → `pipeline/CLAUDE.md` — Airflow DAG setup and import quirks

## Org Tools & References

### Incident Response
If asked to help debug a production incident, Claude can walk engineers through the org's standard **20-step incident-response runbook** in `.claude/runbooks/incident-response.md`. This runbook is used identically across all teams.

### Analytics & Live SQL
Analysts may ask Claude to run live SQL queries against the analytics warehouse (e.g., "How many users signed up last week?"). See `.claude/analytics/warehouse-query-template.md` for connection details and query patterns. Queries are safe and read-only.

## When to Edit Generated Code
Code under `api/gen/` is regenerated by `make proto` and will silently overwrite hand-edits. Do not edit it directly; change the `.proto` files instead.

## Key Assumptions
- Developers run `make` commands from the repo root; all paths are repo-relative.
- Local environment has Go, Node.js, and Python installed per the root Makefile.
- If working in `pipeline/`, AIRFLOW_HOME is set to `pipeline/.airflow` by shell or IDE integration.
```

**Why this split:**

- **layout.md** is your design artifact—shows the reasoning behind the context hierarchy, helps future maintainers understand why info is where it is.
- **CLAUDE.md** is the live file. It's narrow (org conventions + service pointers + runbook/analytics callouts) so Claude stays focused; service-specific quirks live in service CLAUDE.md files so they don't clutter the root or create cross-service confusion (e.g., Claude won't think it should open a browser in the Go service just because web/ does Playwright).

The incident runbook and analytics setup are centralized (not duplicated per service) but discoverable from the root, so Claude proactively offers them when relevant.
