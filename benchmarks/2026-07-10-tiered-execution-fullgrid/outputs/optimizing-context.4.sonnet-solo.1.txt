# Atlas — Claude Code Context Layout

| Path / Mechanism | Purpose | Rationale |
|---|---|---|
| `/CLAUDE.md` | Org-wide conventions + repo map | Root file Claude always loads; holds only what applies to *every* service so it stays short and doesn't drift out of sync with service-level details. |

| `/web/CLAUDE.md` | React/TS conventions, `npm test` vs `npm run test:unit` split | Same auto-load mechanism; prevents Claude from ever running the Playwright-launching `npm test` when it meant a fast unit check. |

| `.claude/hooks/` — `PreToolUse` hook on edits under `api/gen/**` | Blocks/warns on hand-edits to generated code | Turns "hand-edits get silently overwritten" from a prose warning into an enforced guardrail — Claude (or a human) can't accidentally skip past it. |
| `.claude/hooks/` — `PreToolUse` hook on `git push` / PR creation | Reminds or gates on `make check` | Backstops the "must pass before any PR" rule mechanically rather than trusting every session to remember the prose rule. |
| `.claude/skills/incident-response/SKILL.md` (+ supporting files: contact list, templates) | The 20-step IR runbook | Identical across every team → lives once at the org level, not duplicated per service; Skill format supports progressive disclosure and bundling reference material (escalation contacts, message templates) beyond a single markdown file. |
| `.claude/settings.json` MCP entry for the analytics warehouse (read-only credentials) | Registers the live-SQL tool | Live query access is a *capability* (a tool), not context — belongs in MCP config with scoped/read-only creds, not as instructions in a CLAUDE.md. |
| Root `CLAUDE.md` — "Analytics / warehouse SQL" section | Tells Claude when/how to use the warehouse tool and to confirm scope first | Keeps usage guardrails (confirm before running, prefer limited/read-only queries) discoverable without bloating every service-level file. |

### `CLAUDE.md` (repo root)

```markdown
# Atlas

Monorepo for the Atlas platform: `api/` (Go, gRPC), `web/` (React/TypeScript), `pipeline/` (Python/Airflow).

Each service directory has its own `CLAUDE.md` with service-specific conventions and quirks — Claude Code loads it automatically whenever work touches that directory. This file only covers conventions that apply everywhere.

## Org-wide conventions (apply to every service)

- **Commits**: Conventional Commits format (`feat:`, `fix:`, `chore:`, etc.) for every commit message.
- **Before any PR**: `make check` must pass at the repo root. Do not open or update a PR with a failing `make check`.
- **Branching**: trunk-based development. No long-lived branches — branch from `main`, keep branches short-lived, merge back promptly.

## Service map

- `api/` — Go / gRPC service. See `api/CLAUDE.md`. Never hand-edit anything under `api/gen/` — it's regenerated by `make proto` and hand-edits get silently overwritten.
- `web/` — React/TypeScript frontend. See `web/CLAUDE.md`. `npm test` launches real browsers via Playwright — use `npm run test:unit` for fast unit runs.
- `pipeline/` — Python/Airflow DAGs. See `pipeline/CLAUDE.md`. DAGs will not import unless `AIRFLOW_HOME=pipeline/.airflow` is set first.

## Incident response

For any live incident, use the `/incident-response` skill (`.claude/skills/incident-response/`) to walk through the standard 20-step runbook. It's identical across every team — don't improvise a different process.

## Analytics / warehouse SQL

Analysts can ask Claude to query the analytics warehouse directly via the configured warehouse MCP tool (read-only credentials). Confirm the query and its intended scope with the analyst before running it, and default to limited/read-only queries unless asked otherwise.
```

One design call worth flagging: I put the incident-response runbook in a **Skill** rather than a slash command, since it's identical org-wide and benefits from bundling supporting material (contacts, templates) — a slash command would be simpler if you'd rather keep it to a single file with no extras.
