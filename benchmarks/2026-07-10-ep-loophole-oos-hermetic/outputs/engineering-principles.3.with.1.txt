Building a generic plugin framework for one exporter is premature abstraction — you'd be designing an interface against a single implementation, which usually means guessing wrong about what actually needs to vary. I'd only add the plugin boundary once a second exporter is real (or at minimum concretely specified), and let that second implementation drive the interface.

If you have a second exporter format already planned/specced, that changes things — tell me what it is and I can factor the interface off both. Otherwise, my recommendation is: keep the current exporter as a plain module now, and extract the plugin seam when #2 shows up.
