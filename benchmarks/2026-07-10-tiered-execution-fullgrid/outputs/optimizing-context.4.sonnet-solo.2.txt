## layout.md

```markdown
# Claude Code context layout — atlas monorepo

| File / mechanism | Location | Rationale |
|---|---|---|
| `CLAUDE.md` | repo root | Org-wide rules (commits, `make check`, trunk-based dev) that must apply no matter which service you're in; loaded on every session. |
| `CLAUDE.md` | `api/` | Go/gRPC conventions + the `api/gen/` codegen warning, auto-loaded only when work touches `api/`; keeps root file short. |
| `CLAUDE.md` | `web/` | React/TS conventions + the `npm test` (Playwright, opens browsers) vs `npm run test:unit` distinction — a footgun worth surfacing right where it'll be hit. |
| `CLAUDE.md` | `pipeline/` | Python/Airflow conventions + the `AIRFLOW_HOME=pipeline/.airflow` requirement, without which DAG imports silently fail. |
| `CLAUDE.md` (optional, one line) | `api/gen/` | A trip-wire directly inside the generated tree ("generated by `make proto`, do not edit") for the moment someone `cd`s in or opens a file there — belt-and-suspenders on top of the note in `api/CLAUDE.md`. |
| `.claude/skills/incident-response/SKILL.md` | repo root | The 20-step runbook is identical across teams and invoked on demand, not needed in every context window — a Skill loads its full detail only when triggered, keeping it out of the always-on CLAUDE.md budget while staying discoverable/company-wide. |
| `.mcp.json` | repo root | Declares the analytics-warehouse MCP server so Claude has a live SQL tool for analyst questions about product metrics, without embedding credentials or query patterns in prose context. |
| `.claude/settings.json` | repo root | Enforces guardrails procedurally: allow `make check` / `npm run test:unit` / warehouse MCP tool, deny writes under `api/gen/**` and deny/gate `npm test` (Playwright) — rules Claude can't accidentally ignore the way it might ignore prose. |
| `CLAUDE.local.md` (gitignored) | repo root (per engineer) | Personal scratch notes/overrides that shouldn't pollute the shared, checked-in context files. |

**Why nested over one giant root file:** Claude Code auto-loads the nearest CLAUDE.md for the directory you're working in, so service-specific quirks (codegen, test commands, env vars) only enter context when relevant — the root file stays a stable, rarely-changing contract.

**Why a Skill over stuffing the runbook into CLAUDE.md:** the runbook is large, procedural, and only needed during an actual incident; a Skill is invoked by name/intent rather than always resident, and every team gets the same copy from one source of truth in the repo.

**Why MCP over a CLAUDE.md instruction for SQL:** "run live SQL" is a capability (a tool), not a convention — it needs an actual connection/tool definition (`.mcp.json`) plus a permission grant (`settings.json`), not just prose telling Claude it's allowed to.
```

## CLAUDE.md

```markdown
# atlas — root context

Monorepo for the Atlas platform:

- `api/` — Go, gRPC
- `web/` — React, TypeScript
- `pipeline/` — Python, Airflow

Each service has its own `CLAUDE.md` with service-specific conventions, loaded automatically when you're working in that directory:
- `api/CLAUDE.md`
- `web/CLAUDE.md`
- `pipeline/CLAUDE.md`

## Org-wide conventions (apply everywhere, all services)

- **Commit messages**: Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`, etc.) — no exceptions.
- **Before any PR**: `make check` must pass at the repo root. Never open or push a PR update if it fails.
- **Branching**: trunk-based development — no long-lived branches. Small, frequent merges to `main`.

## Cross-cutting capabilities

- **Incident response**: a 20-step runbook, identical across every team, is available as a Skill (`.claude/skills/incident-response/`). If asked to run or walk through an incident, use it rather than improvising steps.
- **Product metrics**: analysts can ask about product metrics and Claude can run live, read-only SQL against the analytics warehouse via the MCP tool defined in `.mcp.json`. Prefer scoped, read-only queries; confirm before anything resembling a write or schema change.

## Do not touch

- `api/gen/**` is generated by `make proto`. Hand-edits are silently overwritten — regenerate instead of editing (details in `api/CLAUDE.md`).
```

One thing worth deciding before you commit this: whether the incident-response Skill and the warehouse MCP config should live in this repo at all, or in a separate org-wide tooling repo that `atlas` references — since you said the runbook is identical "across every team," duplicating it per-repo would mean N copies to keep in sync.
