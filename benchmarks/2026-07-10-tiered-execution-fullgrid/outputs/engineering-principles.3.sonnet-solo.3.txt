# Release Process Review

## Current State
- PR review: ~2 day average wait
- Merged changes queue on `main` until a biweekly release train
- Each train batches ~40 changes
- Release manager manually transcribes changelog entries from PR descriptions
- Three sequential sign-offs required per release: QA lead → eng manager → product manager
- Hotfixes go through the identical process — no expedited path
- Team sentiment: busy but slow, releases feel stressful

## Root Causes of the Slowness/Stress

1. **Batch size drives risk, and risk drives ceremony.** Batching 40 changes means any single release can contain a bad change, so the org has (reasonably) added three approvals to compensate. But the approvals don't reduce risk from batching itself — they just add latency. The batch size is the actual root cause; the sign-offs are a symptom/compensation.

2. **No hotfix lane.** Because urgent fixes share the same pipeline as routine changes, "urgent" gets the same 2+ week worst-case latency as everything else. This is the most acute pain point and easiest to fix independently of everything else.

3. **Changelog generation is manual and duplicative.** PR descriptions already contain the information; hand-copying is pure toil and a error-prone bottleneck that only the release manager can do, creating a single point of contention right before every release.

4. **Sequential sign-offs compound latency.** Three approvals done in sequence (rather than in parallel) means the release manager is waiting on three separate people's calendars serially. Each sign-off may only take 30 minutes, but if each participant takes a day to get to it, that's 3+ days added to every release regardless of the size or risk of the change.

## Recommendations

### 1. Decouple hotfixes from the release train (highest priority, lowest effort)
Create a separate expedited path for urgent fixes: smaller diff, single reviewer, single approver (e.g., eng manager only, or on-call lead), deploy independently of the train. This alone removes the worst-case "urgent fix waits 2 weeks" failure mode without touching the rest of the process.

### 2. Increase release frequency / shrink batch size
Moving from biweekly to weekly (or daily/continuous, if the org can get there) directly reduces the blast radius of any single release. Smaller batches are easier to reason about, easier to roll back, and lower-stakes enough that heavy sign-off ceremony becomes less necessary. This is the highest-leverage change but requires the most process and possibly tooling investment (e.g., feature flags to decouple deploy from release-to-users).

### 3. Automate changelog generation
Since the information already lives in PR descriptions, generate the changelog automatically from merged PR metadata (e.g., a bot that compiles PR titles/descriptions tagged with a `changelog:` convention, or a script run at release-cut time). This removes a manual, error-prone, single-person bottleneck.

### 4. Parallelize sign-offs, and scale them to risk
- Run QA/eng/product sign-off concurrently instead of sequentially — cuts multi-day serial waiting to the length of the slowest single approval.
- Consider whether every release needs all three sign-offs, or whether low-risk releases (e.g., no schema changes, no customer-facing behavior changes) can go out with a lighter-weight single approval, reserving the full three-person sign-off for higher-risk releases.

### 5. Address the 2-day PR review wait
Not explicitly part of the release process, but it's the first domino: reducing this (e.g., review SLAs, smaller PRs, more reviewers) shrinks the total lead time from "code written" to "code shipped," compounding with the release-frequency improvements above.

## Suggested Sequencing
Given effort vs. impact, tackle in this order:
1. Hotfix lane (days to implement, immediate relief)
2. Automated changelog (low effort, removes a chronic bottleneck)
3. Parallel sign-offs (process change only, no tooling needed)
4. Increase release cadence / shrink batch size (highest impact, most investment — do once 1–3 reduce the pain enough to make smaller/more-frequent releases feel low-stakes rather than high-overhead)
