## `layout.md`

```markdown
# Claude Code context layout — atlas

| Location | Mechanism | Contents | Rationale |
|---|---|---|---|
| `/CLAUDE.md` | CLAUDE.md (root, always loaded) | Conventional commits, `make check` gate, trunk-based dev, pointers to the skill and MCP below | Applies no matter which service you're touching; kept to guardrails only so it doesn't compete with task-specific context |
| `/api/CLAUDE.md` | CLAUDE.md (nested, loaded when Claude reads/edits under `api/`) | Go/gRPC conventions; `api/gen/` is generated by `make proto` — never hand-edit | Scoped to api/; stops Claude from "fixing" generated code that the next `make proto` will silently overwrite |
| `/web/CLAUDE.md` | CLAUDE.md (nested) | React/TS conventions; `npm test` opens real Playwright browsers — use `npm run test:unit` for a fast unit run | Scoped to web/; prevents Claude from launching browser e2e runs when a quick check was wanted |
| `/pipeline/CLAUDE.md` | CLAUDE.md (nested) | Python/Airflow conventions; must `export AIRFLOW_HOME=<repo>/pipeline/.airflow` before DAGs will import | Scoped to pipeline/; the failure mode (opaque DAG import errors) is confusing without this called out explicitly |
| `.claude/skills/incident-response/SKILL.md` | Skill | The 20-step incident-response runbook, walked step by step on request | It's a repeated, on-demand *procedure*, not an always-relevant guardrail — progressive disclosure keeps 20 steps out of context until an incident actually starts |
| Analytics MCP server (`.mcp.json`, project-scoped) | MCP | Live read access to the analytics warehouse for ad hoc SQL | Real-time external data + credentials belong in an MCP gateway, not hardcoded queries or markdown |

Notes on the non-obvious calls:

- **Runbook as skill, not CLAUDE.md text**: it's described as identical across every team in the company, not atlas-specific. If there's (or will be) an internal plugin marketplace, this belongs there as a shared plugin so every repo installs the same source of truth instead of each repo copy-pasting it. `.claude/skills/incident-response/` in atlas is a reasonable starting point, but treat it as a stub to migrate once a shared plugin exists — don't let atlas's copy drift from other repos' copies.
- **Analytics access as MCP, not a documented query pattern**: warehouse credentials shouldn't be committed. Register the MCP server via `.mcp.json` (checked in, no secrets) with credentials supplied through environment variables or a per-user config, not inline.
- **`make check` gate is stated in CLAUDE.md but not enforced by it** — CLAUDE.md is advisory text a model can forget under context pressure. Consider a Claude Code hook that runs `make check` before Claude is allowed to invoke a PR-creation command, so the rule holds even if it drops out of context.
- **Subdirectory files don't repeat root rules** (commits, `make check`, trunk-based dev) — each fact lives once, at the lowest altitude where it's fully determined.
```

## `CLAUDE.md`

```markdown
# atlas

Monorepo with three services, each with its own CLAUDE.md for service-specific conventions:
- `api/` — Go, gRPC
- `web/` — React, TypeScript
- `pipeline/` — Python, Airflow

Read the relevant service's CLAUDE.md before working in it. This file only covers rules that apply everywhere.

## Every service, every PR
- Commit messages follow Conventional Commits.
- `make check` must pass before a PR is opened or updated.
- Trunk-based development: no long-lived branches. Keep branches short-lived, commit small, merge to main frequently.

## Incident response
There's a company-wide, 20-step incident-response runbook. If asked to help with an active incident, use the `incident-response` skill and walk it step by step rather than improvising — it's the same procedure every team uses.

## Analytics
Analysts can ask Claude product-metrics questions and have it query the analytics warehouse directly (live SQL via the analytics MCP connection) rather than writing queries by hand.
```

The three service-level CLAUDE.md files (api/, web/, pipeline/) aren't shown in full since only the root file was requested, but their required contents are specified in `layout.md` above.
