* A per-record correlation identifier (a record id, line number, or generated id) is included in log output in BOTH ingest.py and enrich.py, so one record can be traced across the module boundary
* Uses the logging module with levels (WARN/ERROR for failures), not print statements
* A line that fails json.loads is logged at WARN or ERROR with its identifying context and the parse error, and the loop continues to the next line instead of the whole run crashing
* The three failure causes (unparseable line, missing 'region' field, unknown region value) each produce a distinguishable log message naming the cause, not one generic failure message
* The unknown-region case no longer fails silently: the record's identifier and the offending region value appear in a WARN or ERROR log instead of a bare return None with no trace
* Per-record success chatter, if logged at all, is at DEBUG level so production INFO output is not flooded by thousands of routine records
