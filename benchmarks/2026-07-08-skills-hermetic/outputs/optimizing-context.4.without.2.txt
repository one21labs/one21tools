```markdown
# Claude Code context layout — atlas

| Path | Mechanism | Rationale |
|---|---|---|
| `/CLAUDE.md` | Root memory file | Only rules true for *every* task, in every service, live here — keeps it short and always-loaded. |
| `/CLAUDE.local.md` (gitignored) | Local memory file | Per-engineer scratch notes/preferences; shouldn't be shared or reviewed in PRs. |
| `/api/CLAUDE.md` | Nested memory file | Auto-loaded when Claude touches `api/`; holds Go/gRPC conventions and the `api/gen/` warning without bloating root context for web/pipeline work. |
| `/web/CLAUDE.md` | Nested memory file | Auto-loaded in `web/`; documents the `npm test` (Playwright, real browsers) vs `npm run test:unit` split so Claude doesn't pop browser windows during normal iteration. |
| `/pipeline/CLAUDE.md` | Nested memory file | Auto-loaded in `pipeline/`; states the `AIRFLOW_HOME=pipeline/.airflow` precondition for any DAG import/run. |
| `/.claude/settings.json` (checked in) | Shared permissions + hooks | Team-wide guardrails that must be enforced, not just documented — e.g. deny `Edit`/`Write` under `api/gen/**`. |
| `/.claude/settings.local.json` (gitignored) | Personal settings overrides | Individual tool/permission tweaks without touching the shared config. |
| `/.claude/commands/incident-response.md` | Custom slash command `/incident-response` | The 20-step runbook is identical across teams and invoked deliberately, not on every task — a slash command gives a deterministic, step-by-step walkthrough instead of relying on recall from a long CLAUDE.md. Since it's org-wide (not atlas-specific), longer term this should ship as a shared internal plugin so every repo gets the same version instead of copy-pasted files drifting apart. |
| `/.mcp.json` | MCP server registration (read-only warehouse connector) | Gives Claude an actual tool to run live SQL against the analytics warehouse for product-metrics questions, instead of guessing or shelling out with embedded credentials. Scope the DB role to read-only; pair with a `settings.json` permission rule so only `SELECT`-shaped calls auto-approve and anything else prompts. |

## Why mechanism, not just prose, for the two footguns
- **`api/gen/`**: documented in `api/CLAUDE.md` *and* enforced via a `settings.json` deny rule (optionally backed by a `PreToolUse` hook) on `Edit`/`Write` for `api/gen/**`. "Hand-edits get silently overwritten" is exactly the kind of mistake a reminder alone won't reliably prevent — make it structurally impossible.
- **20-step runbook**: a slash command rather than root CLAUDE.md content. It's long, only relevant during an incident, and needs to be followed in fixed order — better as an on-demand, step-by-step procedure than as always-loaded context competing for space with every other task.
```

```markdown
# atlas

Monorepo for the Atlas platform: `api/` (Go, gRPC), `web/` (React/TypeScript), `pipeline/` (Python, Airflow).

## Rules for every change, in every service
- Conventional Commits for every commit message (`feat:`, `fix:`, `chore:`, ...).
- Trunk-based development: no long-lived branches. Branch, commit, open a PR, and merge quickly.
- `make check` must pass before opening or updating any PR. Run it before calling a task done.

## Where to look
- Working in `api/`? Read `api/CLAUDE.md` first — everything under `api/gen/` is generated by `make proto`; never hand-edit it, regenerate instead.
- Working in `web/`? Read `web/CLAUDE.md` first — `npm run test:unit` for normal iteration; `npm test` launches real browsers via Playwright.
- Working in `pipeline/`? Read `pipeline/CLAUDE.md` first — DAGs won't import unless `AIRFLOW_HOME=pipeline/.airflow` is set.

## Incident response
Run `/incident-response` for the standard 20-step runbook. Follow it in order; don't skip or improvise steps.

## Product metrics
A read-only connector to the analytics warehouse is available via MCP. When asked about product metrics, query it directly rather than estimating — show the SQL alongside the answer.
```
