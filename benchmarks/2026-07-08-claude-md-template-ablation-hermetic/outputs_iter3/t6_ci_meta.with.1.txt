**Which CI system are you on?** The syntax differs a lot:
- GitHub Actions: no native "retry failed job" — usually done via `nick-fields/retry` action wrapping the test step, or `strategy` isn't for retries (that's matrix). 
- GitLab CI: has a built-in `retry:` keyword per-job (`retry: 2` or scoped to `when: script_failure`).
- CircleCI: no built-in job retry; typically done with a wrapper script or the "Rerun failed" API triggered externally.
- Jenkins: `retry(N) { ... }` in a pipeline stage.

**Before blindly adding retries, worth deciding deliberately rather than just flipping it on:**
- *Scope* — retry all test jobs, or only ones with known flakiness? Blanket retries can mask real, newly-introduced bugs, not just infra flakiness.
- *Retry count* — 1 retry is usually enough to smooth over infra blips; more than that starts hiding genuine intermittent failures.
- *Visibility* — you'll want retried-but-passed runs flagged somewhere (annotation/report) so flaky tests still get tracked and fixed, not just silently swept under the rug.
