# Release Process Review

## Current State Summary

Our release process follows this flow:

1. **PR Review Phase**: Developers open PRs and wait an average of 2 days for review before merge.
2. **Accumulation Phase**: Approved changes sit on main until the biweekly release train.
3. **Batch Size**: Each release typically ships approximately 40 changes at once.
4. **Changelog Compilation**: The release manager manually copies a changelog entry from each PR description into the release document for distribution.
5. **Sign-off Gates**: Every release requires sequential approval from three parties in this order:
   - QA lead
   - Engineering manager
   - Product manager
6. **Hotfix Path**: Urgent fixes follow the identical process as standard releases, including the same biweekly cadence.

## Problems Identified

### 1. Large Batch Size Increases Risk and Release Stress
A biweekly release with ~40 changes creates a large blast radius and increases the complexity of testing, deployment, and troubleshooting. When issues arise in a massive batch, isolating the root cause becomes harder. This contributes to the team feeling that "each release is stressful."

### 2. Sequential Sign-offs Multiply Wait Time
Three sequential approval gates mean each release passes through QA, then EM, then PM approval in strict order. This creates unnecessary delays—any one gate can block all three, and process time stacks end-to-end rather than overlapping. This is a bottleneck independent of the actual review content.

### 3. Manual Changelog Compilation Is Toil and Error-Prone
The release manager hand-copies changelog entries from PR descriptions into a release document. This is repetitive manual transcription work that is:
- Time-consuming (multiplied by ~40 changes per release)
- Prone to copy errors or inconsistent formatting
- A process that offers no value beyond data entry

### 4. No Expedited Path for Hotfixes
A one-line critical bug fix waits for the next biweekly train (potentially up to 2 weeks). Urgent changes are routed through the same full process as routine work, defeating the purpose of rapid response to production issues. This friction discourages timely hotfixes and increases risk exposure.

### 5. PR Review Latency Adds to Overall Lead Time
A 2-day average review wait delays the entry of changes into the main branch. While not the only factor, this contributes to slow perceived shipping velocity from a contributor's perspective and lengthens the total cycle time from "code ready" to "in customer hands."

## Recommendations

### 1. Increase Release Frequency and Reduce Batch Size
**Action**: Move toward more frequent releases (weekly, or even continuous deployment). Aim to cut batch sizes in half or smaller.

**Expected benefit**: 
- Smaller batches are faster to test and review, reducing pre-release validation burden.
- Faster feedback loop—bugs caught sooner, features reach users sooner.
- Reduced blast radius per release lowers stress and recovery time if issues occur.
- Aligns with modern delivery practices that reduce risk through smaller, frequent changes.

### 2. Parallelize Sign-offs Instead of Sequential Approval
**Action**: Send release artifacts to QA, EM, and PM in parallel. All three review concurrently and sign off; the release proceeds once all three approve (or define a minimum consensus threshold).

**Expected benefit**:
- Dramatically reduces sign-off wait time (concurrent gates instead of stacked serial delays).
- No single gate can hold up the other two.
- Keeps release manager and developers unblocked during the approval window.

### 3. Automate Changelog Generation
**Action**: Extract changelog data automatically from PR titles, descriptions, or labels into the release document. Configure tooling (e.g., a GitHub Action or similar) to compile the changelog on release.

**Expected benefit**:
- Eliminates manual transcription toil.
- Reduces errors and formatting inconsistencies.
- Frees the release manager for higher-value tasks.
- Changelog is consistent and complete.

### 4. Create an Expedited Hotfix Path
**Action**: Define a lightweight hotfix process that bypasses the biweekly train:
- One or two pre-approved reviewers (e.g., a rotating on-call engineer).
- Parallel or reduced sign-off (e.g., QA lead + EM only, or QA lead single sign-off for critical bugs).
- Separate hotfix deployment pipeline that ships within hours, not weeks.
- Limited scope: only critical production bugs, not features or routine changes.

**Expected benefit**:
- Urgent fixes ship fast, reducing customer impact.
- Encourages timely response to production issues.
- Keeps the standard release train uninterrupted.

### 5. (Optional) Reduce PR Review Latency
**Action**: Consider implementing a PR review SLA (e.g., all open PRs reviewed within 24 hours) or a reviewer rotation to ensure timely feedback.

**Expected benefit**:
- Faster feedback loop for developers, encouraging rapid iteration.
- Reduces the perception of shipping slowness.
- May require reviewer workload balancing but improves overall flow.

## Suggested Prioritization

### Phase 1: Quick Wins (Low Effort, High Impact)
1. **Parallelize sign-offs** — Requires only a process change, no tooling. Eliminates sequential delays immediately.
2. **Automate changelog generation** — One-time configuration of existing tools (GitHub Actions, release tooling). Eliminates recurring toil and errors.

### Phase 2: Structural Changes (Medium Effort, High Impact)
3. **Create expedited hotfix path** — Requires defining criteria, approval workflow, and optionally a separate deployment pipeline. High payoff for critical issues.
4. **Increase release frequency** — Requires coordinating with QA, EM, PM on cadence and testing scope. Consider starting weekly before moving to continuous.

### Phase 3: Ongoing Optimization (Ongoing)
5. **Reduce PR review latency** — Implement SLA and monitor; adjust reviewer assignments as needed. Complements higher-frequency releases.

---

**Next Steps**: Start with Phase 1 (parallelize sign-offs, automate changelog) to reduce friction immediately, then tackle hotfix path and release frequency in Phase 2 as the team adjusts.
