# Release Process Review

## Current State
- PR review: ~2 day average wait
- Post-merge: changes queue on `main` until a biweekly release train
- Batch size: ~40 changes per release
- Changelog: manually assembled by release manager, copying from PR descriptions
- Sign-off: sequential — QA lead → Engineering Manager → Product Manager
- Hotfixes: no separate path; ride the same train as everything else

## Key Problems

**1. Batch size is the root cause of the stress.** Bundling ~40 changes means when something breaks, you're debugging across 40 candidates instead of 1-2. This is why releases feel stressful even though the team is busy — risk compounds with batch size, not with effort.

**2. Hotfixes inherit the batch's latency for no reason.** A one-line urgent fix waiting up to 2 weeks defeats the purpose of calling it urgent. This is the most concrete, fixable gap.

**3. Changelog generation is manual toil that's also a bottleneck.** Hand-copying from PR descriptions in front of a release is (a) release-manager-hours spent right before a stressful event, (b) error-prone, and (c) entirely automatable if PR descriptions have a predictable format.

**4. Sequential sign-offs serialize latency that doesn't need to be serial.** QA, EM, and PM are very likely evaluating different concerns (correctness, engineering risk, business impact) that don't depend on each other's output. Chaining them adds N-1 rounds of wait-for-availability on top of each review's own time.

**5. Two-day PR review wait is likely a symptom, not the disease.** Once releases are smaller and more frequent, review latency matters more (it's a bigger fraction of time-to-ship), so it's worth fixing, but it's not the top lever here.

## Recommendations, in priority order

**1. Decouple deploy from the train — ship on merge (or close to it).**
Move toward continuous delivery: each merged PR deploys independently (or in small groups), gated by automated tests + a fast-follow smoke check, rather than waiting for a fixed calendar date. This is the single highest-leverage change: it shrinks the batch size from ~40 to ~1, which shrinks blast radius, shrinks rollback scope, and removes the "big scary release" feeling entirely. If full continuous deploy is too big a jump, start by moving from biweekly to daily or twice-weekly trains — even that alone cuts batch size 5-10x.

**2. Give hotfixes an actual fast path.** Define a hotfix criteria (e.g., sev-1/sev-2 production issue) and let those changes deploy immediately with an abbreviated sign-off (e.g., one approver, or EM+QA in parallel) rather than the full three-stage sequential process. This is a near-zero-cost fix and removes the most obvious absurdity in the current process.

**3. Automate the changelog.** Require a structured field in the PR template (or a commit convention like Conventional Commits) and generate the changelog automatically from merged PRs at release time. This removes manual toil and a pre-release bottleneck, and scales naturally as release frequency increases (it has to — you can't hand-copy 40 entries every day).

**4. Parallelize sign-offs.** Have QA, EM, and PM review concurrently instead of in sequence, with a defined process for resolving conflicting feedback. If there's a genuine dependency (e.g., PM sign-off requires QA to have already confirmed no regressions), keep only that dependency and parallelize the rest.

**5. Consider whether all three sign-offs are needed for every release.** As batch size shrinks (per #1), full three-party sign-off on every single small release becomes disproportionate. Consider risk-tiering: low-risk changes (docs, config, small fixes) get lighter sign-off; larger/riskier changes keep the full process.

**6. Revisit PR review SLAs once release cadence improves.** With smaller, more frequent releases, a 2-day review wait becomes a larger relative share of total lead time. Consider a review SLA (e.g., first response within 4 business hours) or a rotation to ensure PRs don't stall waiting on a specific reviewer.

## Suggested Sequencing
Do #2 (hotfix path) and #3 (changelog automation) first — both are low-risk, low-effort, and immediately reduce toil and stress. Then tackle #1 (smaller/more frequent releases) as the main structural change, since #4 and #5 (parallel, risk-tiered sign-off) become easier and more natural once batches are smaller.
