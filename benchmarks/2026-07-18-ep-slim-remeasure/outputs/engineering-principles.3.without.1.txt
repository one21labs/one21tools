# Release Process Review

## Current State
- PR → ~2 day review wait → merge → sits on `main` until biweekly train
- Release train ships ~40 changes in one batch
- Release manager hand-copies changelog entries from PR descriptions
- Three sequential sign-offs required (QA lead → EM → PM) for **every** release, including hotfixes
- Hotfixes use the same path as normal releases

## Diagnosis
The core problem is **batch size**, and almost everything else follows from it. Bundling 40 unrelated changes into one release means: any single regression forces a full-batch investigation, the three-sign-off gate becomes justifiably cautious (nobody wants to approve 40 changes lightly), and a one-line urgent fix is stuck waiting behind the same 40-change queue. Batching didn't cause the stress — it caused the process built around it to be heavy, and now the heavy process is what makes small releases feel expensive too.

## Recommendations, ranked by impact

**1. Give hotfixes a separate, lightweight path.**
This is the most urgent fix. A one-line urgent change should never wait for a 40-change train. Define what qualifies as a hotfix (severity threshold), and let it skip the batch entirely: merge → deploy on its own, with a reduced approval bar (e.g., one engineering sign-off + post-hoc QA notification, not all three gates). Every day this stays broken, an "urgent" fix is by definition not urgent.

**2. Shrink batch size / increase release frequency.**
Move from biweekly to weekly, or ideally toward continuous/on-demand deployment for low-risk changes. Smaller batches mean smaller blast radius per release, easier root-causing when something breaks, and less pressure riding on any single release event — directly addressing "each release is stressful because so much changes at once."

**3. Automate the changelog.**
Hand-copying PR descriptions into a release doc is pure toil and an error source. Generate changelogs from PR metadata automatically (e.g., conventional commit messages, PR labels, or tooling like Changesets/release-please) at merge time or release-build time. This removes a manual step the release manager currently owns and eliminates a class of "the changelog didn't match what shipped" bugs.

**4. Make sign-offs proportional to risk, not uniform per release.**
Three sequential human sign-offs for every release, regardless of size or risk, is a fixed cost that scales badly as release frequency increases. Options:
   - Tier changes by risk (schema/payments/auth changes need full sign-off; copy tweaks and internal tooling don't).
   - Parallelize the three sign-offs instead of running them sequentially — sequential gates multiply wait time for no added safety if the reviewers aren't dependent on each other's findings.
   - For low-risk releases, consider replacing manual PM/EM sign-off with automated checks (test coverage, canary metrics) and reserve human sign-off for exceptions.

**5. Address the PR review latency itself.**
Two days average review time is a separate bottleneck upstream of the release train. Consider: a review SLA (e.g., first response within 4 business hours), smaller PRs (large PRs take longer to review and are a likely contributor), or a rotation ensuring a designated reviewer is always available.

**6. Consider decoupling deploy from release via feature flags.**
If not already in use, feature-flagging in-progress work lets code merge to `main` and deploy continuously while still controlling *user-facing* release timing. This breaks the "changes must wait for the train" dynamic without requiring every change to be release-ready the moment it merges.

## Suggested sequencing
Start with #1 (hotfix path) since it's low-risk and high-relief. Then #3 (automate changelog) as a quick win. #2 and #4 are the structural changes — do them together, since reducing batch size makes proportional/parallel sign-offs easier to justify. #5 and #6 are longer-term investments once the release cadence itself is healthier.
