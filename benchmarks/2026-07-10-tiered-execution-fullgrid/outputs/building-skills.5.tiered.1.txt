```markdown
---
name: generating-release-notes
description: Generates RELEASE_NOTES.md from git history since the last tag, grouped by conventional-commit type. Use when the user asks for release notes, changelog, or 'what changed since the last release'.
---

## Procedure

1. Find the last release tag: `git describe --tags --abbrev=0`
2. Get commits since that tag: `git log <tag>..HEAD --oneline`
3. Parse conventional-commit prefixes from each commit message

## Commit Type → Section Mapping

| Conventional Commit Type | Output Section | Action |
|---|---|---|
| `feat` | Features | Include |
| `fix` | Bug Fixes | Include |
| `perf` | Performance | Include |
| `docs` | — | Omit |
| `chore` | — | Omit |
| `ci` | — | Omit |
| `build` | — | Omit |
| `refactor` | — | Omit (unless user-facing) |
| `revert` | Reverts | Include; remove reverted entry if in same release |
| `<type>!` | Breaking Changes | Include (always first section) |
| `fix(security)` or any `security` scope | Security | Include (before Bug Fixes) |

## Section Ordering

1. Breaking Changes (if present)
2. Security (if present)
3. Features (if present)
4. Performance (if present)
5. Bug Fixes (if present)
6. Reverts (if present)

Omit empty sections.

## Entry Formatting

- One line per entry
- Imperative mood (e.g., "Add support for...", "Fix crash in...")
- No trailing period
- PR number linked as `[#123]` (e.g., "Add API authentication support [#456]")

## Output

Write to `RELEASE_NOTES.md` at the repository root.

## Revert Handling

If a commit is reverted and both the original and revert are in the same release, remove the original entry from the notes (keep only the revert).
```

```markdown
# Background and History

## Background

Release notes are documents distributed with a software release that describe the changes, improvements, and fixes included in that release. They help users and stakeholders understand what changed. Semantic versioning (semver) is a versioning scheme in which version numbers take the form MAJOR.MINOR.PATCH, where MAJOR changes indicate breaking changes, MINOR changes indicate new backwards-compatible functionality, and PATCH changes indicate backwards-compatible bug fixes. Git is a distributed version control system that records changes to files over time; a git tag marks a specific point in history, typically a release.

## History of This Process

We used to write release notes by hand in Confluence, which took hours every sprint. In Q3 2025 we moved to a shell script that dumped raw git log output, but nobody could read it. In January 2026 Priya rewrote it around conventional commits, and in March we retired the old script after the mobile team migrated. This skill replaces all of that.
```
