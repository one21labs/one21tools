# Post-Incident Analysis: Silent Data Export Failure

**Date:** 2026-07-09  
**Incident:** Empty customer-export file delivered to client  
**Severity:** High  
**Total Incidents in 60 days:** 3

---

## Executive Summary

The nightly customer-export job produced and silently delivered an empty output file to a client for the third time in two months. The immediate trigger was an undocumented query change by a recently-hired engineer that bypassed the manual verification step. However, this incident is the symptom of a systematic monitoring and validation gap: the job exits with code 0 regardless of whether it actually exports data, and no alerting exists for invalid output. The previous two incidents had entirely different root causes (schema rename, expired credential) but identical outcomes, revealing a pattern of architectural vulnerability.

---

## Incident Timeline

**Most Recent Incident (2026-07-09):**
- Priya (6 weeks tenure) modified the export SQL query without running manual verification
- Job executed and produced zero-row output
- Job exited with code 0
- Empty file was delivered to client
- Client discovered the issue before internal team

**Previous Incident #2:**
- Credential used by job expired
- Job swallowed the authentication error and wrote empty output
- Job exited code 0
- Silent delivery; client notified team

**Previous Incident #1:**
- Upstream schema was renamed; export query referenced old schema name
- Query returned zero rows
- Job exited code 0
- Silent delivery; client notified team

---

## Root Cause Analysis

### Immediate Causes (This Incident)

1. **Missing verification step execution**
   - Documentation exists (team wiki: "after changing the query, run it once by hand and eyeball the row count")
   - Priya did not run or was not aware of this requirement
   - No blocking gate enforces the verification step
   - No peer review required before query deployments

2. **Insufficient onboarding for data pipeline ownership**
   - Priya joined 6 weeks ago; critical procedures may not be top-of-mind
   - No checklist or automated enforcement of pre-deployment steps
   - Knowledge transfer may have been verbal/informal

### Systemic Root Causes (All Three Incidents)

1. **Monitoring only observes exit codes, not data validity**
   - Job success metric is binary: process completed or crashed
   - No validation of output semantics (row count, file size, data freshness)
   - Exit code 0 is achievable even with zero data, authentication errors, or schema mismatches

2. **No circuit breaker or data quality gate before delivery**
   - Output is delivered to client regardless of validity checks
   - Empty files reach clients without friction
   - No guardrail prevents invalid data from propagating

3. **Errors are silently swallowed**
   - Expired credential error did not bubble up to alerting
   - Schema errors did not trigger diagnostics
   - Job design allows critical failures to produce exit 0

4. **No peer review or staged deployment for query changes**
   - Individual engineers can modify export SQL without review
   - No requirement for second pair of eyes on data-affecting changes

---

## Why Previous Incidents Didn't Prevent This One

- **Incident #2 and #1 had different triggers**, creating the false impression each was a one-off
- No systemic improvements were made after either; fixes were likely query-specific
- No metrics were added to detect the *pattern* of zero-row exports
- Team did not converge on root cause (validation gap) because each incident was treated independently

---

## Impact

- **Customer trust:** Third silent failure in 8 weeks; client now questions data pipeline reliability
- **SLA/contractual:** Empty export violates implicit data delivery SLA
- **Detection latency:** Team discovered issue only after client complaint (24–48 hour lag likely)
- **Operational:** Each incident required manual investigation and re-export

## Recommended Changes

### Immediate (Deploy This Week)

1. **Add output validation to the export job**
   - Check row count is > 0 before marking job as successful
   - Check file size is not zero or suspiciously small
   - If validation fails: write to a quarantine location, exit with code 1, and alert

2. **Add alerting on validation failures**
   - Alert on: exit code 1, zero-row export, file size validation failure
   - Route to data platform on-call immediately
   - Prevent delivery to client if validation fails

3. **Document the change requirement in code comments**
   - Add comments at the top of the export script listing required manual checks before any query modification

### Short Term (This Sprint)

4. **Implement peer review gate for export query changes**
   - Require second engineer approval before deploying query modifications
   - Use branch protection or CI gate if possible

5. **Automate the manual verification step**
   - Query dry-run or shadow export before production delivery
   - Validate row count matches expected range (or has > threshold)
   - Report summary (row count, file size, schema columns) in logs for on-call visibility

6. **Onboarding update for data pipeline roles**
   - Add mandatory section: "Before modifying export queries, follow [verification checklist]"
   - Pair new engineers with export job owner for first query change
   - Checklist: run query locally, verify row count, review schema changes, get approval

### Medium Term (Next Sprint)

7. **Implement data quality monitoring dashboard**
   - Track export row count and file size over time by customer/export type
   - Alert on anomalies (sudden drop to zero, unexpected file size change)
   - Surface this dashboard to the team daily

8. **Circuit breaker for invalid exports**
   - If validation fails N times in a row, temporarily disable delivery and alert on-call
   - Require manual review and approval to re-enable

9. **Post-incident review process**
   - After Incidents #2 and #1, write a summary and identify shared patterns
   - Use post-incident to drive systemic fixes, not just incident-specific patches

### Long Term

10. **Audit all critical data pipelines for silent-failure patterns**
    - Any job that delivers data without semantic validation is at risk
    - Apply the validation + alerting pattern across all exports, data syncs, etc.

## Lessons Learned

1. **Exit code is not data validity.** Process success ≠ semantic correctness. Validate the output.
2. **Repeat incidents with different triggers indicate a systemic gap, not isolated bugs.** The pattern is the root cause.
3. **Documentation without enforcement is hope, not process.** Either automate the verification or add a blocking gate.
4. **New engineers need scaffolding, not just docs.** Pair them with a mentor for the first risky operation.
