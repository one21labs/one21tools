## `layout.md`

````markdown
# atlas — Claude Code context layout

**Guiding principle:** the root `CLAUDE.md` carries only what is true for *every* file in the
repo. Anything service-specific lives in a nested `CLAUDE.md` that loads only when Claude touches
that service. Procedures and external systems use *lazy* mechanisms (commands / skills / MCP) so
they cost zero tokens until invoked. Rules that must never be violated are enforced by
settings/hooks, not prose — prose can be ignored, a `deny` rule cannot.

## Placement map

| Path | Mechanism | What goes here | One-line rationale |
|------|-----------|----------------|--------------------|
| `/CLAUDE.md` | Root memory (always loaded) | Org conventions (conventional commits, `make check` before PR, trunk-based), repo map, pointers to procedures/data | It's in context for every turn, so it must stay short and contain only what applies everywhere. |
| `/api/CLAUDE.md` | Nested memory (loads under `api/`) | gRPC/Go notes; `api/gen/` is generated by `make proto`, never hand-edit | Service quirks belong next to the service so they load exactly when relevant, not globally. |
| `/.claude/settings.json` → `permissions.deny` | Permission rule | `Edit`/`Write` on `api/gen/**` blocked | A silent-overwrite footgun deserves a hard block, not just a warning Claude might overlook. |
| `/web/CLAUDE.md` | Nested memory (loads under `web/`) | Default to `npm run test:unit`; `npm test` launches real Playwright browsers | Stops Claude from spawning real browsers by reflexively running `npm test`. |
| `/pipeline/CLAUDE.md` | Nested memory (loads under `pipeline/`) | `export AIRFLOW_HOME="$PWD/.airflow"` before importing/running any DAG | The import-failure cause is non-obvious; documenting it where DAG work happens prevents dead-ends. |
| `/.claude/commands/open-pr.md` | Slash command | Run `make check`, then open the PR only if it passes | Turns the "`make check` before PR" rule into an action Claude can't skip. |
| Org plugin → `skills/incident-response/SKILL.md` | Skill (distributed via org marketplace) | The 20-step runbook | Company-wide + identical everywhere → one source of truth outside atlas, lazy-loaded, auto-invoked on incident context. |
| `/.claude/commands/incident-response.md` | Slash command (fallback) | Same runbook, invokable as `/incident-response` if the plugin isn't installed | Guarantees the runbook is reachable in this repo even before the org plugin rolls out. |
| `/.mcp.json` | MCP server (project-scoped, checked in) | `warehouse` server = read-only connection to the analytics warehouse | Live SQL needs a real connection, not text; project scope gives every analyst the same tool automatically. |
| `/.claude/settings.local.json` | Personal settings (gitignored) | Individual warehouse credentials / overrides | Secrets and per-user prefs must never be committed. |

## Why the runbook is a Skill, not root memory
"Used identically by every team" means it should live **once**, outside atlas, and be installed
org-wide as a plugin — otherwise it drifts across repos. A Skill is loaded on demand and can be
auto-invoked when Claude detects an incident, so it costs nothing until needed. The in-repo
`/incident-response` command is a stopgap until the plugin is standard.

## Why the warehouse is MCP, not a doc
Analysts want Claude to *execute* SQL, not read about the schema. That requires a live tool, which
is exactly what an MCP server provides. Scope the credentials **read-only** and point the server at
the warehouse; the root `CLAUDE.md` just tells Claude the capability exists.

## Appendix — proposed content for the nested files

### `api/CLAUDE.md`
```markdown
# api/ — Go, gRPC

- Everything under `api/gen/` is generated by `make proto`. **Never hand-edit it** — edits are
  silently overwritten on the next generate. To change generated code, edit the `.proto` source
  and re-run `make proto`.
- Run the service's own build/test targets from the Makefile rather than guessing `go` commands.
```

### `web/CLAUDE.md`
```markdown
# web/ — React / TypeScript

- Use `npm run test:unit` for unit tests. **Do not run `npm test`** unless explicitly asked — it
  launches real browsers via Playwright and is slow/interactive.
```

### `pipeline/CLAUDE.md`
```markdown
# pipeline/ — Python, Airflow

- DAGs will not import unless `AIRFLOW_HOME` points at this directory's `.airflow`. Before importing
  or running any DAG, run:  `export AIRFLOW_HOME="$PWD/.airflow"`  (from `pipeline/`).
```

### `.claude/settings.json` (checked in)
```json
{
  "permissions": {
    "deny": ["Edit(./api/gen/**)", "Write(./api/gen/**)"]
  }
}
```

### `.claude/commands/open-pr.md`
```markdown
Run `make check`. If and only if it passes, open a PR against `main` with a Conventional Commits
title summarizing the change. If it fails, stop and show me the failures — do not open the PR.
```

### `.mcp.json` (checked in — no secrets)
```json
{
  "mcpServers": {
    "warehouse": {
      "command": "<analytics-warehouse-mcp-server>",
      "args": ["--readonly"],
      "env": { "WAREHOUSE_URL": "${WAREHOUSE_URL}" }
    }
  }
}
```
(Provide `WAREHOUSE_URL` and read-only credentials per-user via `.claude/settings.local.json` or
the environment; keep them out of the committed file.)
````

## `CLAUDE.md`

````markdown
# atlas

Monorepo with three services, each with its own `CLAUDE.md` that loads when you work in it — read
the service note before changing anything there:

- `api/` — Go, gRPC
- `web/` — React / TypeScript
- `pipeline/` — Python, Airflow

## Engineering standards (apply everywhere)
- **Commits:** Conventional Commits (`feat:`, `fix:`, `chore:`, …), one logical change per commit.
- **Before any PR:** `make check` must pass. Do not open a PR — or tell me one is ready — until it
  does. (`/open-pr` runs this check and opens the PR for you.)
- **Branching:** Trunk-based. Branch off `main`, keep branches short-lived, merge back fast. No
  long-lived feature branches.

## Working here
- **Generated code is never hand-edited.** `api/gen/` is produced by `make proto`; see `api/CLAUDE.md`.
- Prefer each service's documented Makefile / npm / Airflow commands over guessing.

## Procedures & data
- **Incident response:** walk me through the 20-step runbook with `/incident-response` (or just ask
  me to handle an incident).
- **Product metrics:** I can run live, read-only SQL against the analytics warehouse via the
  connected `warehouse` MCP tools — just ask.
````
