## Executive Summary

Our nightly customer-export job delivered an empty output file to a client for the third time in two months. This recurring incident reveals a critical gap: we lack automated validation of job outputs. While each occurrence had a different trigger (schema rename, expired credential, SQL query change), all three bypassed detection because:

1. The job exits with status code 0 regardless of output validity
2. No monitoring checks whether the export contains data
3. The manual verification step that could have caught this was skipped
4. The client is our primary detection mechanism

This analysis identifies both immediate and systemic causes, and recommends changes that will eliminate this failure mode entirely.

---

## Timeline

**Occurrence 1 (2+ months ago):** Schema rename in source system  
**Occurrence 2 (1+ month ago):** Expired credential; error was silently swallowed  
**Occurrence 3 (2026-07-09):** SQL query modified by Priya (6 weeks tenure) without running documented manual verification step

**Discovery:** Client identified empty file during their own data validation

---

## Immediate Cause

Priya modified the customer-export SQL query and did not execute the documented manual verification step (run the query manually and eyeball row count before deployment). This is a process compliance issue, but the process itself is the problem—a single missing step cascaded into customer impact.

---

## Root Causes

### 1. **No Automated Output Validation (Critical)**
The job has no mechanism to detect or alert on zero-row outputs. It exits with status 0 in all cases—success, partial failure, or total failure. No monitoring rule exists to verify the export contains expected data.

**Why this matters:** Each of the three triggers (schema change, credential expiry, SQL modification) should have failed loudly. Instead, they all produced the same silent failure.

### 2. **Single Point of Failure: Manual Verification**
Output validation depends entirely on a human running a query and "eyeballing the row count." This step is:
- Not automated
- Not enforced (Priya didn't run it; nobody caught this before deployment)
- Not version-controlled or audit-logged
- Undiscoverable until something breaks

**Why this matters:** The documented step exists but has no teeth. A new team member (6 weeks tenure) reasonably may not prioritize a wiki note or may forget it under deployment pressure.

### 3. **Monitoring Only Watches Exit Code**
Monitoring checks the job's exit code but does not validate the *result*. An empty file is indistinguishable from a successful run.

**Why this matters:** This is a fundamental gap. Exit code is a proxy for success; the real signal we care about is "does the customer have data they expect?"

### 4. **No Clear Responsibility for Verification**
It's unclear who owns validating the export before it reaches the client. Is it:
- The engineer making the change?
- A code reviewer?
- Automation?
- QA?

With no clear owner, the task falls through the cracks.

---

## Contributing Factors

- **Knowledge distribution:** The manual verification step is documented in the wiki but not enforced in the deployment process. Onboarding may not have surfaced this critical step for Priya.
- **Process vs. automation confusion:** A documented manual step is not the same as a process guarantee. Teams often assume documentation is sufficient; it rarely is.
- **Recency bias:** Because occurrence 1 and 2 happened weeks ago, there was no recent incident review to refresh the team's memory or drive automation fixes.
- **No pre-deployment catch:** The query change was not validated in staging or pre-production. A staging environment running the modified query would have surfaced the empty result before production deployment.

---

## Impact

- **Customer trust:** Client discovered the problem, indicating our monitoring failed completely. This erodes confidence in data reliability.
- **Data loss/latency:** Customer missed one full export cycle of critical business data.
- **Operational friction:** Incident response, customer communication, and re-export effort were needed.
- **Pattern:** Three incidents in two months signals a systemic issue, not a one-off mistake.

---

## Corrective Actions (Required)

### Immediate (implement within 1 week)
1. **Add output validation to the job:** Before marking the job as successful, assert that the output file contains data (row count > 0, or a business-meaningful minimum). Fail the job and alert if this check fails.
   
2. **Add monitoring rule:** Alert if the job exits but the output file is empty or smaller than a historical baseline.

3. **Manual rerun and audit:** Rerun the export for the affected period with a verified query. Deliver to client and confirm receipt.

### Short-term (implement within 2–3 weeks)
4. **Code review enforcement:** Require code review for any change to the export SQL query. The reviewer must confirm that:
   - The query was executed in a staging environment
   - Row count and data quality match expectations
   - This becomes part of the PR template or checklist

5. **Pre-production validation:** Add a pre-prod step to the deployment pipeline that runs the query and validates the output before production deployment.

6. **Onboarding checklist:** Add the manual verification step to the data platform team's onboarding checklist and link to incident postmortems so new team members understand why this matters.

### Medium-term (implement within 4–6 weeks)
7. **Eliminate the manual step:** Replace the documented "eyeball the row count" step with an automated assertion in the code:
   ```python
   # Example: fail the job if row count is 0 or suspiciously low
   row_count = query_result.shape[0]
   assert row_count > 0, f"Export produced {row_count} rows; expected > 0"
   assert row_count > historical_average * 0.8, f"Row count {row_count} is 20%+ below average {historical_average}"
   ```

8. **Add data quality metrics:** Instrument the export job to log:
   - Row count
   - File size
   - Sample of first/last rows
   - Query execution time
   
   These become part of the monitoring dashboard and can surface anomalies even before a client complains.

9. **Test the failure paths:** Add unit and integration tests that verify the job *fails loudly* when:
   - The source query returns 0 rows
   - The output file is not created
   - A credential is expired
   - The schema changes unexpectedly

- **Within 2 weeks:** Review the other recurring export jobs in the platform for the same gap (exit 0 even on failure, no output validation). Apply the same fixes.
- **Within 1 month:** Schedule a team discussion on "failure modes we haven't thought of yet" for other data pipelines. Use this incident as a case study.
- **Within 2 months:** Review whether Priya's recent changes (and other recent changes) need similar validation. Backfill missing checks.

---

## Lessons Learned

1. **Automation > documentation.** A wiki note is not a control. If something must happen, automate it or fail loudly if it's skipped.

2. **Exit code ≠ success.** A job exiting 0 says nothing about whether its output is correct. Always validate the actual result.

3. **The customer shouldn't be your QA.** If a customer discovers a data issue, your monitoring failed. Catching the empty file ourselves is table stakes.

4. **Knowledge isn't authority.** New team members can't be expected to know about critical manual steps from a wiki. The process must force correctness.

5. **Recurring incidents demand systemic fixes.** Three similar incidents with different triggers = the root cause is systemic, not human error. Don't blame Priya; fix the system so the next engineer can't make the same mistake.
