# Post-Incident Analysis: Empty Customer Export File Delivered to Client (Recurrence #3)

**Status:** Draft for review
**Severity:** High (client-facing data delivery failure, third recurrence)
**Date of incident:** 2026-07-09 (report date)

## Summary

The nightly customer-export job produced a zero-row output file, which was delivered to a client without any internal alert firing. This is the **third occurrence in two months** of the same failure mode: empty file produced → job exits 0 → monitoring reports healthy → file delivered → client discovers the problem before we do.

The immediate trigger this time was a query change made by Priya (an engineer, 6 weeks tenure) who did not perform the documented manual verification step ("run the changed query by hand and eyeball the row count"). The two prior occurrences had different immediate triggers (a schema rename; an expired credential that the job swallowed as a non-fatal error) but converged on the identical outcome. The recurrence with three distinct triggers and one identical outcome is itself the primary finding: **the system has no failure mode for "zero rows" other than "success."**

## Timeline (this occurrence)

- Priya modifies the export SQL query as part of unrelated work.
- Documented manual verification step ("run once by hand, eyeball row count") is not performed.
- Nightly job runs, executes modified query, writes 0 rows to output file, exits 0.
- Monitoring evaluates exit code only → reports job healthy, no alert.
- Empty file is delivered to client on schedule.
- Client contacts us reporting an empty/missing export.
- Internal investigation begins (this analysis).

## Impact

- Client received an empty deliverable for at least one delivery cycle; trust/SLA impact with the client (third time — likely compounding reputational damage).
- No internal detection; time-to-discovery was entirely dependent on the client noticing.
- Engineering time spent on reactive investigation instead of proactive prevention.

## Root Cause Analysis

### Surface-level (proximate) triggers — different each time
1. **Occurrence 1:** Schema rename broke a query dependency in a way that didn't raise a SQL error (e.g., a join silently matched nothing).
2. **Occurrence 2:** Expired credential caused an upstream read/auth failure that the job caught and swallowed, continuing to the write step with no data.
3. **Occurrence 3 (this one):** Query edit introduced a logic error (e.g., an overly restrictive filter or broken join) that returned zero rows; no manual check caught it before the scheduled run.

### Systemic root cause — the same every time
The job's definition of "success" is **"the process exited 0,"** not **"the process produced correct, expected output."** Nothing in the pipeline validates the *semantic* correctness of the output before it is delivered. This means:
- Any failure mode upstream of the write step — bad query logic, silently-broken joins from schema drift, swallowed auth errors, or (unlisted, but equally plausible) an empty source table, a timezone/date-filter bug, or a partial upstream load — degrades gracefully into a zero-row file rather than a loud failure.
- The only safety net for correctness is a **manual, memory-dependent human step** documented in a wiki page. This step has no enforcement, no reminder, no gate — it depends on every engineer, every time, remembering that a wiki page exists and following it exactly. It has now failed at least once (this incident) and is generally unlikely to be discovered by new team members without explicit onboarding coverage.
- There is no **circuit breaker** between "job produced output" and "output reached the client." Delivery is unconditional on successful process exit, not on output passing any sanity check.

### Why the manual step failed specifically this time
- Priya joined 6 weeks ago. There's no indication the verification step is covered in onboarding, checklists, PR templates, or tooling — it exists only as tacit knowledge in a wiki page. A new engineer has no structural cue that this step exists or is mandatory.
- Even for tenured engineers, a step that lives only in a wiki and is "run by hand" is inherently fragile: it's easy to forget under time pressure, easy to skip for a change that "looks small," and impossible to audit after the fact (no record of whether it was done).

### Five Whys
1. Why was an empty file sent to the client? → The job wrote 0 rows and monitoring didn't catch it.
2. Why didn't monitoring catch it? → Monitoring only checks exit code, not output content.
3. Why does the job exit 0 on zero rows? → There's no output validation step in the job itself; writing zero rows is not treated as an error condition.
4. Why did the query change produce zero rows undetected? → The only safeguard against a bad query change is a manual, undocumented-in-tooling verification step that wasn't performed.
5. Why wasn't the manual step performed? → It's wiki-documented tribal knowledge with no enforcement mechanism, and no onboarding process ensures new engineers (like Priya) know it exists or is mandatory.

**Root cause:** The pipeline has no automated, enforced check that validates export output before delivery; correctness is entirely delegated to an unenforced manual process and to exit-code-only monitoring, neither of which can catch content-level failures. Three different upstream triggers all reached the client because the last line of defense was a human memory step, not a system control.

## Contributing Factors

- **Monitoring gap:** Exit-code-only health checks are structurally incapable of catching "succeeded but wrong" failures — a well-known class of silent failure.
- **No delivery gate:** File delivery is not conditioned on any output validation (row count, schema shape, non-zero size relative to historical baseline, etc.).
- **Process reliance on memory, not tooling:** The verification step is documented but not embedded in the workflow (no PR checklist item, no CI gate, no required sign-off, no automated diff/row-count comparison run alongside a query change).
- **Onboarding gap:** No evidence that new engineers are walked through job-specific "silent failure" risks or the location/importance of operational wiki docs.
- **No repeat-incident tracking:** This is the third occurrence of an identical outcome, but the fix after each prior occurrence apparently addressed only the specific trigger (schema rename, credential handling) rather than the shared systemic gap — indicating prior post-incident actions were too narrowly scoped.
- **Error swallowing:** In occurrence 2, an exception (expired credential) was caught and allowed execution to continue rather than aborting the job — a general anti-pattern that likely exists elsewhere in the same codebase.

## What We're Changing

### Must do (blocks recurrence of this exact failure mode)
1. **Add an output validation gate to the job itself**, run before any delivery step: fail (non-zero exit, alert) if row count is 0, or if row count deviates beyond a threshold (e.g., >50%) from a trailing baseline (e.g., 7-day median). This is the single highest-leverage fix — it addresses all three prior triggers at once, and any future trigger of the same shape.
2. **Change monitoring to check output content, not just exit code.** At minimum: alert on zero-row output; ideally, alert on row-count anomalies vs. historical baseline.
3. **Gate delivery on validation passing.** The client should never receive a file that failed the row-count/sanity check — delivery must be blocked, not just logged, until a human confirms and overrides.
4. **Audit for swallowed exceptions** in this job's code (the occurrence-2 credential bug is a strong signal this pattern may recur elsewhere). Any caught exception during data extraction should abort the job, not allow it to proceed to a degraded write.

### Should do (reduces reliance on human memory)
5. **Convert the manual verification step into a required, tooled step** — e.g., a pre-merge CI check that runs any changed export query against a recent snapshot and posts the row count/diff on the PR, or a required checklist item in the PR template linking to the wiki doc. Tribal knowledge in a wiki does not scale past the person who wrote it.
6. **Add job-specific silent-failure risks to onboarding** for new data platform engineers, including a walkthrough of this wiki doc and why it exists (with this incident as the example).
7. **Retroactively review prior two incidents' remediation items** to confirm whether they were actually completed, and note explicitly that they did not address the systemic gap — this incident report should not be closed the same way.

### Nice to have
8. **Add a lightweight "last N runs" dashboard** (row counts, file sizes) for all client-facing export jobs, so trends are visible without waiting for an alert.
9. **Consider a client-facing acknowledgment/checksum step** (e.g., client-side validation that a delivered file is non-empty) as a defense-in-depth backstop, discussed with the client relationship owner.

## Lessons Learned

- A failure mode that recurs with **different triggers but identical impact** is a signal that prior fixes patched symptoms, not the underlying gap — treat "same outcome, new cause" as its own incident category worth escalating.
- **Exit code 0 is not a correctness signal** for data pipelines; any job that can succeed while producing wrong/empty output needs a content-level check, not just a process-level one.
- Manual, wiki-documented verification steps are a weak control for anything client-facing and irreversible (once delivered, it's delivered) — they should be treated as a stopgap, not a durable safeguard, especially on teams onboarding new members.
- Delivery to an external client should always be conditional on a passing sanity check, never on process exit status alone.
