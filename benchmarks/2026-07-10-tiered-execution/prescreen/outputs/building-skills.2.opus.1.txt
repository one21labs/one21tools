  Validate a JSON configuration file for syntax errors and, when a JSON Schema
  is available, check the config against that schema. Use when the user asks to
  validate, lint, or check a JSON config file, when a config change should be
  verified before deploy, or when JSON parse or schema-validation errors need

# JSON Config Check

Validate a JSON configuration file and report any problems clearly. Validation
has two layers: (1) syntax/well-formedness, always performed; (2) schema
conformance, performed only when a schema is found.

## When to use

- The user asks to validate / lint / check a JSON config file.
- A config file was just edited and should be verified before it is committed or
  deployed.
- A tool or service is rejecting a JSON config and the cause needs to be found.

## Inputs

- **Config file** — the path the user provides. If none is given, ask for it, or
  look for common config filenames (`config.json`, `*.config.json`,
  `settings.json`) in the working directory and confirm the target with the user.
- **Schema (optional)** — a JSON Schema describing the config's expected shape.

## Procedure

1. **Read the config file.** If it does not exist or cannot be read, stop and
   report the path and the error.

2. **Check syntax.** Parse the file as JSON. If parsing fails, report the parse
   error with the line and column and stop — schema validation is meaningless on
   unparseable input.

3. **Locate a schema.** Look, in order:
   - a schema path the user supplied;
   - a `$schema` field inside the config that points to a local file;
   - a sibling file named `<config-basename>.schema.json` or `schema.json` in the
     same directory.
   If none is found, skip to step 5 and report a syntax-only pass.

4. **Validate against the schema.** Validate the parsed config against the
   schema, collecting *all* violations (not just the first). For each violation
   capture: the JSON path to the offending value, the failing rule (e.g.
   `required`, `type`, `enum`), and a human-readable message.

5. **Report results** (see format below).

## Implementation

Use Python's standard `json` module for parsing and the `jsonschema` package for
schema validation. Collect every error rather than failing fast:

```python
import json
from jsonschema import Draft7Validator

with open(config_path) as f:
    config = json.load(f)          # step 2 — JSONDecodeError => syntax error

with open(schema_path) as f:       # step 4, only if a schema was found
    schema = json.load(f)

errors = sorted(
    Draft7Validator(schema).iter_errors(config),
    key=lambda e: e.path,
)
for e in errors:
    path = "/".join(str(p) for p in e.absolute_path) or "(root)"
    print(f"{path}: {e.message}")
```

If `jsonschema` is not installed, install it (`pip install jsonschema`) or, if
installation is not possible, report that schema validation was skipped for that
reason rather than silently doing only a syntax check.

## Reporting format

State plainly what was checked and the outcome. Do not claim success for a step
that was skipped.

- **Valid, schema checked:**
  `✅ config.json — valid JSON and conforms to schema.json.`
- **Valid, no schema found:**
  `✅ config.json — valid JSON. No schema found; schema conformance not checked.`
- **Syntax error:** report the message with line/column, e.g.
  `❌ config.json — invalid JSON: Expecting ',' delimiter at line 12, column 5.`
- **Schema violations:** list each as `path: message`, e.g.
  ```
  ❌ config.json — 2 schema violations:
    server/port: 'eighty' is not of type 'integer'
    (root): 'database' is a required property
  ```

## Example

> Validate `deploy/app.config.json`.

1. Read `deploy/app.config.json` → parses cleanly (syntax OK).
2. Find sibling `deploy/app.config.schema.json` → use it.
3. Validate → two violations found.
4. Report:
   ```
   ❌ deploy/app.config.json — 2 schema violations:
     retries: 5 is greater than the maximum of 3
     logging/level: 'verbose' is not one of ['debug', 'info', 'warn', 'error']
   ```
```
