## **CLAUDE.md** — Root Configuration

```markdown
# atlas Monorepo

A three-service monorepo: **api** (Go/gRPC), **web** (React/TypeScript), **pipeline** (Python/Airflow).

## Shared Principles

### Commit & Deployment
- **Conventional Commits** required: types are `feat`, `fix`, `docs`, `refactor`, `test`, `chore`. No commits without a type.
- **Trunk-based development** only: no long-lived branches. PR → `make check` must pass → merge to main → deploy.
- **make check must pass** before any code merges. This is non-negotiable. If check fails locally, the PR will be blocked.

### Code Ownership & Assumptions
- **Do not assume the user is familiar with the entire monorepo.** Each service may be worked on in isolation; provide context.
- **Do not refactor across service boundaries** without explicit user request. Each service has independent conventions.
- **Verify that `make check` runs successfully** after any change before considering the task complete.

---

## Per-Service Context

### api/ — Go & gRPC Service
- **Quirk:** Everything under `api/gen/` is auto-generated by `make proto`. Hand-edits will be silently overwritten.
  - Never suggest edits to proto-generated code; regenerate protos instead.
- See `api/.claude/context/README.md` for full service details (imports, proto-to-service workflow, typical dev loop).

### web/ — React & TypeScript
- **Quirk:** `npm test` runs end-to-end tests in real browsers (Playwright). For unit tests only, use `npm run test:unit`.
  - Do not assume test failures are unit test failures; clarify which test suite was run.
- See `web/.claude/context/README.md` for full service details (dev server, build, e2e vs unit, CI).

### pipeline/ — Python & Airflow
- **Quirk:** DAGs require `AIRFLOW_HOME=pipeline/.airflow` to import; without it they silently fail at runtime.
  - Set the environment variable before running Airflow CLI or spawning worker processes.
- See `pipeline/.claude/context/README.md` for full service details (DAG structure, testing, deployment).

---

## Shared Runbooks & Processes

### Incident Response
All teams use a single 20-step incident response runbook stored at `.claude/runbooks/incident-response.md`.

**When to invoke:** A user asks Claude to "walk me through the incident response" or similar.  
**Claude's role:** Read the runbook step-by-step, ask for status/decision at each step, adapt if user skips steps or provides context.  
**Important:** The runbook is identical org-wide; do not improvise or skip steps.

---

## Analytics & SQL Queries

### Live Analytics Warehouse Queries
Analysts may ask Claude to query the analytics warehouse for product metrics.

**Authorization:** Analysts are authorized to run SELECT queries against the analytics warehouse.  
**Process:**
1. See `.claude/analytics/warehouse-guide.md` for schema, table definitions, and example queries.
2. Read the warehouse guide to understand safe query patterns and common metrics.
3. If unsure about a query's correctness, explain the SQL to the user for approval before running.
4. Never make assumptions about sensitive columns or PII; confirm with the user first.

---

## Allowed Autonomy & Confirmation Gates

### Auto-allowed (no confirmation needed)
- Editing code files (api/**, web/**, pipeline/**, excluding api/gen/**)
- Running tests locally (`make check`, `npm test`, `pytest`)
- Creating branches with conventional commit names
- Editing documentation
- Merging PRs (if user explicitly says "merge this")

### Requires confirmation
- Committing to main (ask first; assume trunk-based means frequent, intentional commits)
- Running live SQL against analytics warehouse (show the query first)
- Pushing to any remote branch (confirm scope and target)
- Major refactors that span services or affect multiple teams
- Deletion of files or branches

### Explicitly disallowed
- Editing anything under `api/gen/` (regenerate protos instead)
- Running long-lived branches or feature branches (violates trunk-based policy)
- Applying backwards-compatibility shims when simpler refactors are possible

---

## Context Load Strategy

When Claude is invoked in a service-specific directory:
1. Load root CLAUDE.md first for org context.
2. Load the service's `.claude/context/README.md` for language/tool-specific details.
3. Load `.claude/runbooks/` if the task involves an incident or org-wide process.
4. Load `.claude/analytics/warehouse-guide.md` if queries are planned.

When in doubt, ask the user which context is relevant.
```
