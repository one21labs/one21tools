# Generating Release Notes

## Background

Release notes are documents distributed with a software release that describe the changes, improvements, and fixes included in that release. They help users and stakeholders understand what changed.

Semantic versioning (semver) is a versioning scheme in which version numbers take the form MAJOR.MINOR.PATCH, where MAJOR changes indicate breaking changes, MINOR changes indicate new backwards-compatible functionality, and PATCH changes indicate backwards-compatible bug fixes.

Git is a distributed version control system that records changes to files over time; a git tag marks a specific point in history, typically a release.

## Procedure

1. Find the last release tag:
   ```
   git describe --tags --abbrev=0
   ```
2. Collect the commits since that tag:
   ```
   git log <tag>..HEAD --oneline
   ```
3. Group the commits by their conventional-commit type prefix (see mapping below).
4. Build one section per non-empty group, in the fixed order below.
5. Write the result as markdown to `RELEASE_NOTES.md` at the repo root.

## Commit type mapping

| Commit prefix | Section | Notes |
|---|---|---|
| `feat` | Features | |
| `fix` | Bug Fixes | |
| `perf` | Performance | |
| `docs` | *(omit)* | not included in output |
| `chore` | *(omit)* | not included in output |
| `ci` | *(omit)* | not included in output |
| `build` | *(omit)* | not included in output |
| `refactor` | *(omit)* | include only if user-facing |
| `revert` | Reverts | also remove the original reverted entry if it's in the same release |
| any type with `!` (e.g. `feat!`) | Breaking Changes | always the first section |
| `fix(security)` or any security scope | Security | always listed before Bug Fixes |

## Section order

Breaking Changes → Security → Features → Performance → Bug Fixes → Reverts. Omit any section that has no entries.

## Entry formatting rules

- Every entry links its PR number as `[#123]`.
- Entries are one line, written in imperative mood, with no trailing period.
- Output file: `RELEASE_NOTES.md` at the repo root, written in markdown.
