One thing before that, though: "approved" and "complete" aren't the same thing, and this plan is thin enough that a task breakdown alone would just push the ambiguity into implementation. A user-data-export endpoint is also one of the higher-risk features to under-specify — it's a PII bulk-access surface — so I've split the list into blocking decisions vs. build tasks rather than skip straight to build tasks.

```markdown
# User Data Export — Implementation Tasks

## Blocking decisions (resolve before coding starts)
- [ ] Define "all of a user's data": enumerate every source that holds it — profile, content, activity/logs, billing, data about them stored in *other* users' records (messages, comments, shares)
- [ ] Authorization: confirm the endpoint can only ever return the authenticated caller's own data; decide if bulk PII export needs a re-auth/identity-confirmation step
- [ ] Export format: pick one explicitly (JSON is the safe default once you're merging multiple sources — "whatever's easiest" stops being well-defined there)
- [ ] "Fast" → a number: p95 latency target, or is this async (job + notification) for larger accounts?
- [ ] Delivery mechanism: inline response body vs. background job + signed, expiring download link
- [ ] Rate limiting: cap requests per user to prevent scraping/abuse of a full-account-data endpoint
- [ ] Audit logging: record who exported what, when — bulk export is a common post-account-takeover step, so this is a detection signal, not a nice-to-have
- [ ] Compliance scope: does GDPR/CCPA right-to-access apply (response deadlines, required fields)?

## Build tasks (once the above are answered)
1. Inventory and list all data sources per the decision above
2. Design the export schema/contract
3. Build the data-aggregation logic (service or job, per delivery decision)
4. Implement the endpoint with authz + identity-confirmation checks
5. Implement delivery mechanism (sync payload or async job + link)
6. Add rate limiting
7. Add audit logging
8. Tests: unit, integration, and a security test that confirms user A cannot export user B's data
9. Perf/load test against the agreed target
10. Docs for the endpoint and any user-facing export flow
```
