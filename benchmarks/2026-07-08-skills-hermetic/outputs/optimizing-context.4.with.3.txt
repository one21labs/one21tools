```markdown
# Atlas — Claude Code context layout

| Location | Mechanism | Contains | Rationale |
|---|---|---|---|
| `atlas/CLAUDE.md` | Root CLAUDE.md (always loaded) | Conventional commits, `make check` gate, trunk-based dev, pointers to service CLAUDE.mds, skill/MCP pointers | Org-wide guardrails apply to every session regardless of which service is being touched; keep it short since it's loaded on every turn. |
| `atlas/api/CLAUDE.md` | Nested CLAUDE.md (loaded when working under `api/`) | `api/gen/` is generated by `make proto`, never hand-edit | Scoped to the one service where it's relevant — avoids polluting web/pipeline sessions with Go-specific rules. |
| `atlas/web/CLAUDE.md` | Nested CLAUDE.md (loaded when working under `web/`) | `npm test` opens real Playwright browsers; use `npm run test:unit` for fast iteration | Prevents Claude from defaulting to `npm test` as a quick sanity check and accidentally spawning browser sessions. |
| `atlas/pipeline/CLAUDE.md` | Nested CLAUDE.md (loaded when working under `pipeline/`) | `AIRFLOW_HOME=pipeline/.airflow` must be set before DAGs import | A hard local-environment quirk that would otherwise cause confusing false-negative import errors. |
| `.claude/skills/incident-response/SKILL.md` | Skill | The 20-step incident-response runbook | It's a repeated, identical procedure invoked on-demand by any team — a Skill loads it only when an incident is actually happening, instead of bloating every session's always-on context. Root CLAUDE.md just points to it. |
| `.mcp.json` (project-scoped) | MCP server | Warehouse connector for live SQL against the analytics DB | This is real-time external data access, not static instructions — MCP is the mechanism for tool/data connectivity, not CLAUDE.md prose. Root CLAUDE.md notes it exists plus usage guardrails (read-only, always scoped). |
| `.claude/settings.json` hook (optional, not one of the two requested files) | Hook | Pre-PR/pre-push check that runs `make check` and blocks on failure | `make check` is a hard gate today enforced only by convention in CLAUDE.md; a hook makes it deterministic instead of relying on Claude remembering to run it. Worth adding once the two core files are in place. |

## Why this split and not one big CLAUDE.md
- Claude Code auto-loads the root CLAUDE.md every session, then layers in nested CLAUDE.md files as you move into subdirectories — so service-specific quirks (`api/gen/`, Playwright, `AIRFLOW_HOME`) belong in each service's own file, not duplicated at root.
- The runbook is identical across teams but only relevant during an incident — that's exactly the "on-demand, progressive disclosure" case a Skill is for, not always-loaded context.
- Live SQL access is a tool/data-connectivity need, not a instruction-following need — that's what MCP is for, not a prompt telling Claude to "run SQL."
```

## `CLAUDE.md` (root)

```markdown
# Atlas Monorepo

Three services, each with its own CLAUDE.md — read it when you're working in that directory:
- `api/` — Go, gRPC
- `web/` — React, TypeScript
- `pipeline/` — Python, Airflow

## Non-negotiables (all services)
- Every commit follows Conventional Commits.
- `make check` must pass before opening or updating any PR — run it, don't assume.
- Trunk-based development: no long-lived branches. Land small changes directly against main.

## Incident response
If an engineer describes a live production incident, use the `incident-response` skill — it walks through the company's standard 20-step runbook. Trigger it based on the situation described, not only when someone explicitly asks for "the runbook."

## Analytics / warehouse access
For product-metrics questions, analysts can query the analytics warehouse live via the `warehouse` MCP connector (see `.mcp.json`). Treat it as read-only production access: always scope queries with a date range or `LIMIT`, and never run anything that mutates data.
```
